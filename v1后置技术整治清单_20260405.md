# v1 后置技术整治清单

**文档日期**：2026-04-05  
**文档定位**：`v1` 发布后进入的技术整治主清单  
**目标**：将当前已知的架构级问题、代码质量问题、历史兼容残留从业务发布主线中剥离，单独安排修复优先级。

---

## 一、为什么单独立项

这些问题不适合继续混在 `v1` 发布链路中处理，原因有三点：

1. 它们大多属于架构整治，不是局部 bug 修补
2. 它们会影响稳定性，但不是当前人工验收链路的即时阻断
3. 如果在发布窗口继续大改，反而容易破坏已稳定的业务链和展示层结果

因此，本清单的作用是：

- 固定问题范围
- 明确优先级
- 避免问题被遗忘
- 为后续技术整治排期提供依据

---

## 二、当前审查结论总览

当前需要后置整治的问题主要分为三类：

1. 数据访问与数据库层架构问题
2. 前端代码结构与遗留代码问题
3. 迁移体系与历史兼容清理问题

## 三、P0 必须先修

### P0-1 `executeSQL` 解析器系统性风险

**严重度**：高  
**涉及文件**：
- [dbService.ts](C:/Users/jjj64/WorkBuddy/20260318232610/server/src/services/dbService.ts)
- [tasks.ts](C:/Users/jjj64/WorkBuddy/20260318232610/server/src/routes/tasks.ts)

**当前问题**：

- `executeSQL` 是基于正则的 SQL 字符串解析器
- 注释明确说明不支持 `JOIN`
- 但业务代码中仍存在 `JOIN` 实际调用
- `UPDATE` 中对 `COALESCE` 等表达式存在静默跳过
- 稍复杂 SQL 会出现不可预测行为

**当前实测信息**：

- 当前仓库中 `executeSQL(` 共有约 `138` 处调用
- 涉及文件约 `26` 个

**风险**：

- 查询逻辑静默失真
- 更新逻辑部分字段丢失
- 复杂 SQL 表现不可信
- 难以继续在此基础上稳定扩展

**后置动作建议**：

1. 立即禁止新增复杂 `executeSQL` 调用
2. 识别所有 `JOIN / IN / COALESCE / 表达式 UPDATE` 场景
3. 将复杂查询逐步替换为明确的 SDK 查询或真实 SQL 执行层

### P0-2 数据访问模式统一规范

**严重度**：高  
**涉及文件**：
- [dbService.ts](C:/Users/jjj64/WorkBuddy/20260318232610/server/src/services/dbService.ts)
- [database.ts](C:/Users/jjj64/WorkBuddy/20260318232610/server/src/database.ts)

**当前问题**：

当前代码库存在三种数据库访问模式：

1. Supabase SDK 具名函数
2. `executeSQL` 正则解析器
3. `pg Pool / Client` 直连

**风险**：

- 连接管理分裂
- 错误处理分裂
- schema 变更影响面不可控
- 新开发者无法判断标准写法

**后置动作建议**：

1. 明确唯一推荐的数据访问标准
2. 将 `database.ts` 和 `dbService.ts` 的职责重新界定
3. 将 `pg` 直连限制在迁移/诊断/工具脚本，不再进入主业务路径
