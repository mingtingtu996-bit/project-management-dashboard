## 问题 1: 同步队列处理竞态条件

### 问题描述

`processSyncQueue()` 方法缺乏并发控制，当多个浏览器标签页同时尝试同步时，可能导致：
- 同一项目被多次同步
- 同步状态不一致
- 数据覆盖或丢失

### 修复方案

#### 1. 添加同步队列处理锁
```typescript
// 同步队列处理锁，防止并发执行
private isProcessingSyncQueue = false
private syncQueueAbortController: AbortController | null = null
```

#### 2. 实现取消机制
```typescript
// 如果正在处理，则取消当前操作并重新启动
if (this.isProcessingSyncQueue) {
  console.log('[StorageService] 同步队列正在处理中，取消当前操作')
  this.syncQueueAbortController?.abort()
  await new Promise(resolve => setTimeout(resolve, 100))
}
```

#### 3. 添加原子性检查
```typescript
// 检查项目是否仍存在于队列中（可能被其他标签页处理）
const currentItem = this.syncQueue.find(i => i.id === item.id)
if (!currentItem || currentItem.status !== 'pending') {
  continue
}
```

#### 4. 实现失败通知机制
```typescript

  // 触发全局事件，让UI可以显示同步失败提示

    detail: { item, error: errorMessage }
  }))
}
```

#### 5. 添加重试机制

## 问题 2: 乐观锁竞态条件

### 问题描述

原有的乐观锁实现采用"先读后写"模式，存在竞态条件：
```typescript
// 旧代码 - 有竞态条件
const existing = await getProject(id)
if (!existing || existing.version !== expectedVersion) {
  throw new Error('VERSION_MISMATCH')
}
// 这里可能被其他请求修改！
await supabase.from('projects').update({...}).eq('id', id)
```

### 修复方案

#### 1. 原子性更新 (UPDATE ... WHERE version = ?)
```typescript
// 新代码 - 原子性更新
const { error, count } = await supabase
  .from('projects')
  .update({ 
    ...fields, 
    updated_at: now(), 
    version: expectedVersion + 1 
  })
  .eq('id', id)
  .eq('version', expectedVersion)  // 原子版本检查

// 如果没有更新任何行，说明版本不匹配
if (count === 0) {
  throw new Error('VERSION_MISMATCH: 该项目已被他人修改，请刷新后重试')
}
```

#### 2. 修复的函数
- `updateProject()` - 项目更新
- `updateTask()` - 任务更新（已修复）
- `updateRisk()` - 风险更新
- `updateMilestone()` - 里程碑更新（通过 updateTask）
