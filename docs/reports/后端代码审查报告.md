### 2. 【严重】GET /:id 接口存在全表扫描性能问题

**位置**: 
- `server/src/routes/tasks.ts` (第 30-52 行)
- `server/src/routes/risks.ts` (第 30-52 行)
- `server/src/routes/milestones.ts` (第 30-52 行)
- `server/src/routes/members.ts` (第 30-52 行)
- `server/src/routes/invitations.ts` (第 41-63 行)

**问题描述**:
获取单个资源的接口先查询全表数据，然后在内存中过滤。这种方式在数据量大时会导致严重的性能问题和内存浪费。

**当前代码** (以 tasks.ts 为例):
```typescript
router.get('/:id', validateIdParam, asyncHandler(async (req, res) => {
  const { id } = req.params
  logger.info('Fetching task', { id })
  
  const tasks = await supabase.getTasks()  // 加载所有任务
  const task = tasks.find(t => t.id === id)  // 内存过滤
  
  if (!task) {
    // ... 404 响应
  }
  // ...
}))
```

**影响**:
- 当任务数量增加时，性能呈线性下降
- 浪费网络带宽和内存
- 数据库压力大

**改进建议**:
1. 在 SupabaseService 中添加 getById 方法
2. 直接通过 ID 查询，使用 `.eq('id', id).single()`
3. 参考 projects.ts 中的正确实现

**示例代码**:
```typescript
// 在 SupabaseService 中添加
async getTaskById(id: string): Promise<Task | null> {
  const { data, error } = await this.client
    .from(this.tableNames.tasks)
    .select('*')
    .eq('id', id)
    .single()
  
  if (error) {
    if (error.code === 'PGRST116') return null
    throw error
  }
  return data
}

// 在路由中使用
router.get('/:id', validateIdParam, asyncHandler(async (req, res) => {
  const { id } = req.params
  const task = await supabase.getTaskById(id)
  
  if (!task) {
    // ... 404 响应
  }
  // ...
}))
```

### 3. 【严重】乐观锁实现存在竞态条件

**位置**: `server/src/services/supabaseService.ts` (第 65-86, 121-141, 176-196, 231-251 行)

**问题描述**:
乐观锁的实现使用两次数据库查询（先查询 version，再更新），这两次操作之间不是原子的，存在竞态条件。在高并发场景下可能导致版本检查失效。

**当前代码**:
```typescript
async updateProject(id: string, updates: Partial<Project>, expectedVersion: number): Promise<Project | null> {
  // 乐观锁检查 - 第一次查询
  const { data: existing } = await this.client
    .from(this.tableNames.projects)
    .select('version')
    .eq('id', id)
    .single()
  
  if (!existing || existing.version !== expectedVersion) {
    throw new Error('VERSION_MISMATCH')
  }

  // 更新 - 第二次查询（非原子操作）
  const { data, error } = await this.client
    .from(this.tableNames.projects)
    .update({ ...updates, version: expectedVersion + 1 })
    .eq('id', id)
    .select()
    .single()
  
  if (error) throw error
  return data
}
```

**影响**:
- 两个并发请求可能同时通过版本检查
- 最终导致数据丢失或覆盖
- 乐观锁失效

**改进建议**:
使用数据库的原子更新操作，在 WHERE 条件中同时检查 version：

```typescript
async updateProject(id: string, updates: Partial<Project>, expectedVersion: number): Promise<Project | null> {
  // 原子更新：在 WHERE 中检查版本
  const { data, error } = await this.client
    .from(this.tableNames.projects)
    .update({ ...updates, version: expectedVersion + 1 })
    .eq('id', id)
    .eq('version', expectedVersion)  // 同时检查版本
    .select()
    .single()
  
  if (error) {
    if (error.code === 'PGRST116') return null
    throw error
  }
  
  if (!data) {
    throw new Error('VERSION_MISMATCH')
  }
  
  return data
}
```

### 6. 【警告】日志存储在内存中，重启后丢失

**位置**: `server/src/middleware/logger.ts` (第 10-11, 61-67 行)

**问题描述**:
日志仅存储在内存数组中，服务重启后所有历史日志丢失，不适合生产环境使用。

**当前代码**:
```typescript
class Logger {
  private logs: LogEntry[] = []
  private maxLogs = 1000

  getLogs(): LogEntry[] {
    return [...this.logs]
  }
}
```

**影响**:
- 无法追溯历史问题
- 重启后无法诊断之前的错误
- 缺少持久化审计日志

**改进建议**:
1. 集成专业的日志库（如 winston、pino）
2. 将日志写入文件或发送到日志服务（如 ELK、CloudWatch）
3. 保留内存日志用于快速访问，同时持久化到外部存储

**示例代码**:
```typescript
import winston from 'winston'

class Logger {
  private memoryLogs: LogEntry[] = []
  private maxMemoryLogs = 1000
  private logger: winston.Logger

  constructor() {
    this.logger = winston.createLogger({
      level: 'info',
      format: winston.format.json(),
      transports: [
        new winston.transports.File({ filename: 'logs/error.log', level: 'error' }),
        new winston.transports.File({ filename: 'logs/combined.log' }),
      ],
    })
  }

  private log(level: LogLevel, message: string, context?: Record<string, unknown>) {
    const entry: LogEntry = {
      level,
      message,
      timestamp: new Date().toISOString(),
      context,
    }
    
    // 内存日志（快速访问）
    this.memoryLogs.push(entry)
    if (this.memoryLogs.length > this.maxMemoryLogs) {
      this.memoryLogs.shift()
    }

    // 持久化日志
    this.logger.log(level, message, context)
  }
}
```
