## 一、前端性能问题

### 1.1 渲染性能问题

#### 严重问题

**问题1: Dashboard页面每次渲染都重新计算统计数据**

文件位置：`client/src/pages/Dashboard.tsx` 第29-36行

```typescript
// 问题代码
const totalTasks = tasks.length
const completedTasks = tasks.filter(t => t.status === 'completed').length
const inProgressTasks = tasks.filter(t => t.status === 'in_progress').length
const highRisks = risks.filter(r => r.level === 'high' || r.level === 'critical').length
const upcomingMilestones = milestones.filter(m => m.status === 'pending').length
const completedMilestones = milestones.filter(m => m.status === 'completed').length

const completionRate = totalTasks > 0 ? Math.round((completedTasks / totalTasks) * 100) : 0
```

问题描述：统计数据计算直接在组件主体内执行，每次渲染都会重新计算。当数据量较大时（如100+任务），`filter`操作会消耗显著的计算资源。

性能影响：中等。每当父组件触发重渲染时，这些计算都会重复执行。

优化建议：
```typescript
// 使用useMemo缓存计算结果
const stats = useMemo(() => {
  const totalTasks = tasks.length
  const completedTasks = tasks.filter(t => t.status === 'completed').length
  const inProgressTasks = tasks.filter(t => t.status === 'in_progress').length
  const highRisks = risks.filter(r => r.level === 'high' || r.level === 'critical').length
  const upcomingMilestones = milestones.filter(m => m.status === 'pending').length
  const completedMilestones = milestones.filter(m => m.status === 'completed').length
  
  return {
    totalTasks,
    completedTasks,
    inProgressTasks,
    highRisks,
    upcomingMilestones,
    completedMilestones,
    completionRate: totalTasks > 0 ? Math.round((completedTasks / totalTasks) * 100) : 0
  }
}, [tasks, risks, milestones])
```

**问题3: useEffect依赖不完整导致潜在的死循环或数据不同步**

文件位置：`client/src/pages/ProjectDetail.tsx` 第42-46行

```typescript
useEffect(() => {
  if (id) {
    loadProjectData()
  }
}, [id])  // 缺少loadProjectData依赖
```

问题描述：`loadProjectData`函数在useEffect外部定义，但没有添加到依赖数组中。这可能导致：
1. 闭包陷阱：useEffect使用旧的loadProjectData版本
2. 如果使用useCallback不当，可能导致无限循环

优化建议：
```typescript
// 选项1: 将loadProjectData移入useEffect
useEffect(() => {
  if (!id) return
  
  const loadProjectData = async () => {
    // 加载逻辑
  }
  loadProjectData()
}, [id])

// 选项2: 使用useCallback并添加正确依赖
const loadProjectData = useCallback(async () => {
  // 加载逻辑
}, [id])

useEffect(() => {
  if (id) loadProjectData()
}, [id, loadProjectData])
```

### 1.3 状态管理性能问题

#### 严重问题

**问题6: Zustand store缺少选择器优化**

文件位置：`client/src/hooks/useStore.ts`

当前使用方式（在Dashboard等组件中）：
```typescript
const { tasks, risks, milestones, setTasks, setRisks, setMilestones } = useStore()
```

问题描述：
1. 使用对象解构订阅整个store，任何状态变化都会触发组件重渲染
2. 当tasks、risks、milestones中任一个变化时，使用这些数据的组件都会重新渲染

优化建议：使用Zustand的选择器模式
```typescript
// 错误方式 - 订阅整个store
const { tasks } = useStore()

// 正确方式 - 只订阅需要的部分
const tasks = useStore(state => state.tasks)

// 更精确的选择器
const taskStats = useStore(state => ({
  total: state.tasks.length,
  completed: state.tasks.filter(t => t.status === 'completed').length
}))
```

## 二、后端性能问题

### 2.1 数据库性能问题

#### 严重问题

**问题8: N+1查询风险**

文件位置：`server/src/services/supabaseService.ts`

问题描述：在`updateProject`、`updateTask`、`updateRisk`等方法中，先执行一次查询检查版本，再执行更新操作：

```typescript
async updateTask(id: string, updates: Partial<Task>, expectedVersion: number): Promise<Task | null> {
  // 第一次查询 - 检查版本
  const { data: existing } = await this.client
    .from(this.tableNames.tasks)
    .select('version')
    .eq('id', id)
    .single()
  
  if (!existing || existing.version !== expectedVersion) {
    throw new Error('VERSION_MISMATCH')
  }

  // 第二次查询 - 执行更新
  const { data, error } = await this.client
    .from(this.tableNames.tasks)
    .update({ ...updates, version: expectedVersion + 1 })
    .eq('id', id)
    .select()
    .single()
  // ...
}
```

性能影响：高。每次更新操作需要两次数据库往返。

优化建议：
1. 使用数据库的RETURNING子句结合条件更新（可在Supabase中使用RPC函数）
2. 或者接受这个设计（乐观锁检查是必要的）

```sql
CREATE INDEX idx_milestones_status ON milestones(status);
```

## 三、内存使用问题

### 3.1 内存泄漏风险

#### 严重问题

**问题12: storageService事件监听器未清理**

文件位置：`client/src/lib/storageService.ts` 第138-139行

```typescript
constructor(config: Partial<StorageServiceConfig> = {}) {
  // ...
  if (typeof window !== 'undefined') {
    window.addEventListener('online', () => this.handleOnline())
    window.addEventListener('offline', () => this.handleOffline())
  }
  // 没有对应的removeEventListener
}
```

问题描述：构造函数中注册的事件监听器在组件卸载或服务销毁时没有清理，可能导致内存泄漏。

优化建议：
```typescript
class StorageServiceImpl implements StorageAdapter {
  private boundHandleOnline: () => void
  private boundHandleOffline: () => void
  
  constructor(config: Partial<StorageServiceConfig> = {}) {
    this.boundHandleOnline = () => this.handleOnline()
    this.boundHandleOffline = () => this.handleOffline()
    
    if (typeof window !== 'undefined') {
      window.addEventListener('online', this.boundHandleOnline)
      window.addEventListener('offline', this.boundHandleOffline)
    }
  }
  
  // 添加销毁方法
  destroy() {
    if (typeof window !== 'undefined') {
      window.removeEventListener('online', this.boundHandleOnline)
      window.removeEventListener('offline', this.boundHandleOffline)
    }
  }
}
```
