**问题描述**:  
`updateTask` 函数在 `mysqlService.ts` 第 410-445 行实现了乐观锁机制，但存在潜在的竞态条件：

```typescript
// 当前实现（第418-434行）
if (expectedVersion !== undefined) {
  const { error, count } = await supabase
    .from('tasks')
    .update({ ...fields, updated_at: now(), version: expectedVersion + 1 })
```

### P0-002: 同步队列处理竞态条件

**位置**: `storageService.ts` 第 420-431 行

**问题描述**:  
`processSyncQueue` 方法在处理同步队列时存在竞态条件：

```typescript
private async processSyncQueue(): Promise<void> {
  const pendingItems = this.syncQueue.filter(item => item.status === 'pending')
  
  for (const item of pendingItems) {
    item.status = 'syncing'  // 状态变更没有原子性保护
    // 这里可以调用云端API进行同步
    item.status = 'completed' // 简化处理，直接标记完成
  }
  
  this.persistSyncQueue() // 批量保存可能覆盖其他更新
}
```

**潜在风险**:  
1. **状态不一致**: 多个浏览器标签页同时处理队列时可能互相覆盖
2. **数据丢失**: `persistSyncQueue` 使用 localStorage，没有并发控制
3. **重复同步**: 没有机制防止同一项目被多次同步

**建议修复**:  
```typescript
// 添加锁机制
private syncLock = false

private async processSyncQueue(): Promise<void> {
  if (this.syncLock) return // 防止并发执行
  this.syncLock = true
  
  try {
    // ... 原有逻辑
    // 使用原子操作更新单个项目
    const index = this.syncQueue.findIndex(i => i.id === item.id)
    if (index !== -1) {
      this.syncQueue[index] = { ...item, status: 'completed' }
      this.persistSyncQueue() // 每次更新后立即保存
    }
  } finally {
    this.syncLock = false
  }
}
```

## 🟡 P1 警告问题

### P1-001: 缺乏实时同步机制

**位置**: 全局架构

**问题描述**:  
检查发现应用**没有实现 WebSocket 或 Server-Sent Events 实时同步机制**：

- `storageService.ts` 只有离线同步队列，没有实时推送
- `supabaseAdapter.ts` 没有使用 Supabase Realtime 订阅
- 后端没有 WebSocket 服务器实现

**当前行为**:  
用户A修改任务后，用户B需要手动刷新才能看到更新

**影响**:  
- 多人同时编辑同一项目时容易产生冲突
- 用户体验不佳，缺乏协作感
- 依赖乐观锁被动处理冲突

**建议方案**:  
1. **短期**: 添加定时轮询（每30秒刷新一次）
2. **中期**: 集成 Supabase Realtime 订阅
3. **长期**: 实现 WebSocket 服务器

### P1-002: 冲突解决机制过于简单

**位置**: `storageService.ts` 第 862-900 行

**问题描述**:  
冲突解决只提供了两个简单选项：

```typescript
// 强制使用本地版本
forceUpdate(entityId: string): void {
  const item = this.syncQueue.find(i => (i.data as any)?.id === entityId)
  if (item && item.data) {
    const d = item.data as { id: string; version?: number }
    d.version = (d.version || 1) + 1
    item.status = 'pending'
    item.retries = 0
    this.persistSyncQueue()
  }
}

// 应用服务器数据
applyServerData(entityId: string): void {
  this.syncQueue = this.syncQueue.filter(i => (i.data as any)?.id !== entityId)
  this.persistSyncQueue()
}
```

**问题**:  
- 没有字段级冲突检测
- 没有合并（merge）选项
- 用户无法看到冲突的具体内容

**建议**:  
实现一个冲突解决对话框，展示：
- 本地版本和服务器版本的差异
- 字段级对比
- 选择保留哪个版本或手动合并

### P1-003: 缺乏同步状态监控和重试机制

**位置**: `storageService.ts` 第 815-860 行

**问题描述**:  
`processSyncItem` 方法的重试机制过于简单：

```typescript
if (this.syncQueue[index].retries >= 3) {
  this.syncQueue[index].status = 'failed'
}
```

**问题**:  
- 失败后没有指数退避重试
- 没有失败通知机制
- 用户不知道哪些操作同步失败了
- 没有手动重试的UI

**建议**:  
1. 实现指数退避重试（1s, 2s, 4s, 8s...）
2. 添加同步失败通知Toast
3. 在UI上显示同步状态指示器
4. 提供手动重试按钮
