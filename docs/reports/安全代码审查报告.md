## 高危漏洞

### 1. 缺少认证授权机制

**严重程度**: 🔴 高危

**位置**:
- `server/src/index.ts` (第31-32行)
- 所有路由文件 (`routes/`)

**问题描述**:
Express应用未实现任何认证中间件。所有API端点(`/api/projects`, `/api/tasks`, `/api/risks`等)都公开可访问，任何人都可以创建、读取、更新、删除项目数据。

```typescript
// server/src/index.ts:31-32
app.use(cors())  // 无限制的CORS
app.use(express.json())
app.use(requestLogger)

// 所有路由都未受保护
app.use('/api/projects', projectsRouter)
app.use('/api/tasks', tasksRouter)
app.use('/api/risks', risksRouter)
```

**潜在风险**:
- 未授权用户可以访问、修改或删除任何项目数据
- 数据完全暴露给公开互联网
- 无法审计谁执行了什么操作

**修复建议**:
1. 实现JWT认证中间件：
```typescript
// server/src/middleware/auth.ts
import { Request, Response, NextFunction } from 'express'

export function authenticate(req: Request, res: Response, next: NextFunction) {
  const token = req.headers.authorization?.replace('Bearer ', '')

  if (!token) {
    return res.status(401).json({
      success: false,
      error: { code: 'UNAUTHORIZED', message: '未提供认证令牌' },
      timestamp: new Date().toISOString(),
    })
  }

  try {
    const decoded = verifyToken(token)
    req.userId = decoded.userId
    next()
  } catch (error) {
    return res.status(401).json({
      success: false,
      error: { code: 'INVALID_TOKEN', message: '无效的认证令牌' },
      timestamp: new Date().toISOString(),
    })
  }
}

export function authorize(roles: string[]) {
  return (req: Request, res: Response, next: NextFunction) => {
    const userRole = req.userRole
    if (!userRole || !roles.includes(userRole)) {
      return res.status(403).json({
        success: false,
        error: { code: 'FORBIDDEN', message: '权限不足' },
        timestamp: new Date().toISOString(),
      })
    }
    next()
  }
}
```

2. 在index.ts中添加认证中间件：
```typescript
app.use(authenticate)  // 保护所有API路由
app.use('/api/projects', projectsRouter)
app.use('/api/tasks', tasksRouter)
```

3. 实现登录/注册端点：
```typescript
// server/src/routes/auth.ts
router.post('/login', validate(loginSchema), asyncHandler(async (req, res) => {
  const { email, password } = req.body
  // 验证用户凭据
  const user = await authenticateUser(email, password)
  if (!user) {
    return res.status(401).json({ success: false, error: '无效的凭据' })
  }
  const token = generateToken(user)
  res.json({ success: true, data: { token, user } })
}))
```

### 2. Supabase密钥直接暴露在前端

**严重程度**: 🔴 高危

**位置**:
- `client/src/lib/supabase.ts` (第3-6行)
- `client/.env.example` (第16-17行)

**问题描述**:
Supabase的`ANON_KEY`直接暴露在前端环境变量中，该密钥会被打包到客户端JavaScript代码中。

```typescript
// client/src/lib/supabase.ts:3-6
const supabaseUrl = import.meta.env.VITE_SUPABASE_URL || ''
const supabaseAnonKey = import.meta.env.VITE_SUPABASE_ANON_KEY || ''

export const supabase = createClient(supabaseUrl, supabaseAnonKey)
```

```env
# client/.env.example:16-17
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-anon-key
```

**潜在风险**:
- 攻击者可以提取ANON_KEY并直接访问Supabase数据库
- 如果RLS(Row Level Security)配置不当，攻击者可以读取/修改所有数据
- 即使RLS配置正确，攻击者仍可以发起DoS攻击消耗配额

**修复建议**:
1. **方案A - 使用服务端代理（推荐）**:
   - 前端只调用后端API，不直接访问Supabase
   - 后端使用SERVICE_ROLE_KEY进行数据库操作
   - 前端完全不需要Supabase密钥

```typescript
// client/src/api.ts
// 前端不直接使用Supabase客户端
export async function getProjects() {
  const response = await fetch('http://localhost:3001/api/projects', {
    headers: {
      'Authorization': `Bearer ${localStorage.getItem('token')}`
    }
  })
  return response.json()
}
```

```typescript
// server/src/index.ts
// 后端使用SERVICE_ROLE_KEY（安全）
const url = process.env.SUPABASE_URL
const key = process.env.SUPABASE_SERVICE_KEY  // 服务端密钥，不在前端暴露
const supabase = createClient(url, key)
```

2. **方案B - 配置Row Level Security**:
   如果必须在前端使用Supabase，确保RLS策略严格配置：

```sql
-- Supabase RLS策略示例
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;

-- 仅允许项目成员访问
CREATE POLICY "Users can view projects they belong to"
ON projects
FOR SELECT
USING (
  id IN (
    SELECT project_id FROM project_members
    WHERE user_id = auth.uid()
  )
);

CREATE POLICY "Users can update projects they own"
ON projects
FOR UPDATE
USING (
  id IN (
    SELECT project_id FROM project_members
    WHERE user_id = auth.uid() AND permission_level = 'admin'
  )
);
```

### 3. CORS配置过于宽松

**严重程度**: 🔴 高危

**位置**:
- `server/src/index.ts` (第31行)
- `server/.env.example` (第23行)

**问题描述**:
CORS中间件使用默认配置，允许任何来源的跨域请求。

```typescript
// server/src/index.ts:31
app.use(cors())  // 相当于 cors({ origin: '*' })
```

```env
# server/.env.example:23
CORS_ORIGIN=http://localhost:5173
```

**潜在风险**:
- 任何网站都可以发起跨域请求到API
- 可能被利用进行CSRF攻击
- 敏感数据可能被恶意网站窃取

**修复建议**:
1. 限制允许的来源列表：
```typescript
// server/src/index.ts
const allowedOrigins = [
  process.env.CORS_ORIGIN || 'http://localhost:5173',
  'https://your-production-domain.com'
]

app.use(cors({
  origin: (origin, callback) => {
    if (!origin) return callback(null, true)  // 允许同源请求
    if (allowedOrigins.includes(origin)) {
      callback(null, true)
    } else {
      callback(new Error('不允许的CORS来源'))
    }
  },
  credentials: true,  // 如果需要发送cookies
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization']
}))
```

2. 在生产环境中使用环境变量配置多个允许的域名：
```env
# server/.env
CORS_ORIGINS=https://app.example.com,https://admin.example.com
```

### 4. 邀请码生成使用弱随机数

**严重程度**: 🔴 高危

**位置**:
- `server/src/routes/invitations.ts` (第16-23行)

**问题描述**:
邀请码生成使用`Math.random()`，这在密码学上不安全。

```typescript
// server/src/routes/invitations.ts:16-23
function generateInviteCode(): string {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'
  let code = ''
  for (let i = 0; i < 8; i++) {
    code += chars.charAt(Math.floor(Math.random() * chars.length))
  }
  return code
}
```

**潜在风险**:
- `Math.random()`不是密码学安全的随机数生成器
- 攻击者可能预测邀请码
- 暴力破解更容易成功

**修复建议**:
使用密码学安全的随机数生成器：

```typescript
import { randomBytes } from 'crypto'

function generateInviteCode(): string {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'
  const bytes = randomBytes(8)
  let code = ''
  for (let i = 0; i < 8; i++) {
    code += chars[bytes[i] % chars.length]
  }
  return code
}
```

### 5. 缺少速率限制

**严重程度**: 🔴 高危

**位置**:
- `server/src/index.ts` (所有路由)

**问题描述**:
API端点未实现速率限制，攻击者可以无限制地发起请求，导致DoS攻击或暴力破解。

**潜在风险**:
- 暴力破解邀请码、登录凭据
- DoS攻击耗尽服务器资源
- 恶意用户发送大量请求消耗Supabase配额

**修复建议**:
使用`express-rate-limit`中间件：

```bash
npm install express-rate-limit
```

```typescript
// server/src/index.ts
import rateLimit from 'express-rate-limit'

// 通用速率限制
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15分钟
  max: 100,  // 最多100个请求
  message: {
    success: false,
    error: { code: 'TOO_MANY_REQUESTS', message: '请求过于频繁，请稍后再试' },
    timestamp: new Date().toISOString(),
  }
})

// 邀请码验证严格限制
const invitationLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,  // 15分钟最多5次尝试
  skipSuccessfulRequests: true,
})

app.use('/api/', limiter)
app.use('/api/invitations/validate/:code', invitationLimiter)
```

### 12. 权限系统未实现

**严重程度**: 🟡 中危

**位置**:
- 所有CRUD操作端点

**问题描述**:
虽然数据库中有`permission_level`字段，但API端点未实现基于角色的访问控制(RBAC)。

**修复建议**:
实现权限中间件：

```typescript
// server/src/middleware/permission.ts
export function requirePermission(requiredPermission: string) {
  return async (req: Request, res: Response, next: NextFunction) => {
    const userId = req.userId
    const projectId = req.params.id || req.body.project_id

    // 查询用户在该项目中的权限
    const member = await supabase
      .from('project_members')
      .select('permission_level')
      .eq('user_id', userId)
      .eq('project_id', projectId)
      .single()

    if (!member) {
      return res.status(403).json({
        success: false,
        error: { code: 'FORBIDDEN', message: '您不是该项目的成员' },
      })
    }

    // 检查权限级别
    const permissionLevels = ['viewer', 'editor', 'admin', 'owner']
    const userLevel = permissionLevels.indexOf(member.permission_level)
    const requiredLevel = permissionLevels.indexOf(requiredPermission)

    if (userLevel < requiredLevel) {
      return res.status(403).json({
        success: false,
        error: { code: 'INSUFFICIENT_PERMISSIONS', message: '权限不足' },
      })
    }

    req.userRole = member.permission_level
    next()
  }
}

// 使用示例
router.delete('/:id', requirePermission('admin'), asyncHandler(async (req, res) => {
  // 只有admin或owner可以删除项目
}))
```

**位置**:
- `server/src/middleware/errorHandler.ts`

```typescript
// server/src/middleware/errorHandler.ts:67-76
const statusCode = err.statusCode || 500
const response: ApiResponse = {
  success: false,
  error: {
    code: err.code || 'INTERNAL_ERROR',
    message: statusCode === 500
      ? '服务器内部错误，请稍后再试'
      : err.message,
    // 仅在开发环境返回详细错误
    ...(process.env.NODE_ENV === 'development' && {
      details: err.message,
      stack: err.stack
    })
  },
  timestamp: new Date().toISOString(),
}
```
