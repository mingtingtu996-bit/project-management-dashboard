> 适用范围：`证照管理 / 施工图纸`
>
> 文档定位：用于统一施工图纸的产品定位、页面结构、字段口径与系统联动方式。

配套文档：

- `施工图纸字段模型+状态规则稿_20260409.md`

## 1. 文档目标

施工图纸板块不再被定义为“前期证照页里的一个 tab”，而是一个服务于施工全过程图纸管理的独立业务页。

本稿解决的核心问题：

- 施工图纸与前期证照混在一个页面里，语义不清。
- 图纸管理的重点是版本、审图、出图、变更与执行，而不是证照式办理节点。
- 当前字段已经能承接基础 CRUD，但还没有形成稳定的页面目标和产品边界。

本稿结论：

- 施工图纸应保留在 `证照管理` 父模块下。
- 施工图纸应成为与 `前期证照`、`验收时间轴` 平级的独立子页。
- 施工图纸主视图应围绕“图纸包状态”和“版本 / 审图 / 出图推进”展开。
- 图纸分类不宜只用一个字段硬切，而应采用 `专业` + `用途/属性` + `图纸包` 三层组织。
- 施工图纸不应默认以关系图为主视图。

## 2. 产品定位

施工图纸板块的定位不是“图纸台账附属页”，也不是“证照延伸页”，而是：

- `图纸准备度工作台`
- `图纸版本与审图管理台账`
- `施工执行前置控制页`

它主要回答 4 个问题：

- 现在项目主要图纸包出到什么程度
- 哪些专业图纸还在编制、审图或修改
- 哪些图纸已经可以支撑现场施工
- 哪些任务因图纸状态未满足而无法推进

补充两个关键问题：

- 图纸是否齐全，是否存在缺漏
- 当前有效版本到底是哪一版

## 3. 与前期证照、验收时间轴的关系

三者都在 `证照管理` 父模块下，但关注时段和业务重心不同：

- `前期证照`
  开工前关注，核心是四证办理
- `施工图纸`
  施工中关注，核心是图纸版本、审图、出图、变更
- `验收时间轴`
  竣工前后关注，核心是验收推进、前置满足、归档收口

三者的关系是“阶段串联”，不是“页面混合”：

- 导航层面可以串起来
- 业务数据不强行混在一个主页面
- 只在必要处做摘要跳转和状态联动

进一步说明：

- `前期证照` 与 `前期施工图`
  强相关，尤其在开工前阶段
- `施工任务` 与 `施工图 / 专项图 / 变更图`
  强相关
- `验收时间轴` 与 `竣工图`
  强相关

结论：

- 不能把“所有图纸”都用同一套关系逻辑处理
- 更不能简单地把“施工图纸关系图”理解成所有图纸统一挂任务或统一挂验收

## 4. 页面目标

施工图纸页要同时服务两类角色：

- 管理层
  看整体图纸准备情况和当前卡点
- 执行层
  看具体图纸版本、审图进度、出图与变更情况

因此页面建议采用三层结构：

1. `图纸准备度总览`
2. `图纸包主视图`
3. `图纸台账 + 详情处理`

## 5. 核心产品决策

### 5.1 不做关系图主视图

施工图纸不适合像验收那样把“前后依赖”做成页面主表达。

原因：

- 图纸管理本质是按专业、版本、审图状态推进
- 主要关注是“是否可用”“是否已通过”“是否变更”
- 复杂关系图会稀释版本和状态信息

因此：

- 不用 `ForceDirectedGraph` 作为施工图纸主视图
- 不把图纸主页设计成关系网或拓扑图

### 5.2 主表达是“图纸包视图 + 台账”

施工图纸的最佳主表达是：

- 上层看图纸准备度摘要
- 中层看专业图纸包状态
- 下层看明细台账和版本记录

这里的核心对象不是“每一张图纸”，而是：

1. `图纸包`
   页面主视图的管理单位
2. `图纸记录`
   台账中的单条图纸
3. `版本记录`
   图纸变更链

这样才能兼顾：

- 页面不被海量单图塞满
- 版本历史不丢
- 缺漏和齐套性可判断

### 5.3 关系只做辅助表达

图纸和任务、前期证照、验收项之间可以有状态联动，但不建议在图纸主页里直接铺复杂关系图。

更合适的做法是：

- 在 `图纸包` 层显示“主要影响对象摘要”
- 在详情中只对关键包或例外包显示具体影响对象
- 在任务页显示 `drawing` 类条件
- 在验收页把 `竣工图` 作为前置条件来源

补充判断：

- `施工执行类图纸`
  主要影响任务执行
- `竣工归档类图纸`
  主要影响验收与归档
- `前期报审相关图纸包`
  更适合与前期证照形成强关联

所以“统一把所有图纸去做同一类关系”是有问题的。

正确做法是：

- 先按 `专业 + 用途/属性 + 图纸包` 组织
- 再由系统规则判断该图纸包主要联动前期证照、任务还是验收

建议的展示规则：

- 主页面不要求每份图纸都显示“影响前期证照 / 影响任务 / 影响验收”
- 只在图纸包卡片上显示简短摘要，如 `影响任务`、`影响验收`
- 具体关联对象在详情抽屉中展开
- 如无明确联动，不强制显示影响标签

### 5.4 审图不是所有图纸的主节点

并非所有图纸都需要把“送审 / 审查 / 通过”作为主推进节点。

业务上更合理的分法是：

- `必须送审`
  重大图纸、法定审查图、专项送审图
- `可选送审`
  某些项目口径下需要专业审核，但不属于统一法定必审
- `无需送审`
  常规施工执行图、一般深化图、部分内部配合图

因此图纸模型里建议显式增加：

- `requires_review`
- `review_mode`
- `review_basis`

推荐 `review_mode`：

- `mandatory`
- `optional`
- `none`

这样页面上就不会把所有图纸都强行塞进同一条“审图节点链”。

### 5.5 审图参与度如何判定

审图参与度不建议逐张图纸人工判断，而应采用“规则判定 + 项目覆盖”的方式。

建议判定顺序：

1. 公司级规则模板
2. 项目级规则覆盖
3. 图纸包级人工修正

判定依据建议包含：

- 项目类型
- 地区规则
- 专业类型
- 图纸用途/属性
- 是否法定审查对象
- 是否专项报批资料包

也就是说：

- 常规施工执行图，默认 `review_mode = none`
- 法定施工图审查包、专项送审包，默认 `review_mode = mandatory`
- 部分项目内部要求的图纸包，可设为 `review_mode = optional`

页面上应能看到：

- `送审要求：必审 / 可选 / 不适用`
- `判定依据`

而不是要求用户逐条解释为什么要审。

### 5.6 审图规则判定是否能在系统里实现

可以实现，但采用简化规则，不做复杂规则引擎。

建议实现方式：

1. `图纸包模板默认规则`
   根据图纸包的专业和用途/属性给出默认送审要求
2. `项目级人工修正`
   对个别图纸包调整为 `必审 / 不适用 / 人工确认`

系统判定只依赖最小信息：

- `discipline_type`
  专业类型
- `document_purpose`
  用途/属性
- `package_code / package_name`
  图纸包归属
- `requires_review`
  是否需要送审
- `review_mode`
  `mandatory / none / manual_confirm`
- `review_basis`
  判定依据说明

结论：

- 系统可以实现简化后的自动判定
- 不需要补很多重字段
- 不能只靠当前单一 `drawing_type` 字段完成准确判断

### 5.7 本方案执行口径

本方案只保留这些关键能力：

- `图纸包`
- `应有项清单`
- `当前有效版`
- `历史版本链`
- `是否缺漏`
- `齐套度`
- `是否必审`
- `审图状态`
- `是否可施工`
- `竣工图是否可验收`
- `变更`
- `是否影响工期`

其中：

- `是否必审`
  采用简化后的图纸包级判定
- `变更`
  只保留一个变更字段或变更标记
- `是否影响工期`
  保留自动化判断

## 6. 页面结构

## 6.1 顶部总览区

建议展示：

- 图纸总数
- 已通过审图数
- 已出图数
- 审图中数
- 待修改 / 被驳回数
- 本月计划送审数

页面文案示例：

- `施工过程中的图纸版本、审图进度与出图状态总览`
- `当前卡点：结构专业施工图审查`

## 6.2 图纸包主视图

建议主视图采用：

- `按专业分组的图纸包看板`

并建议增加两个筛选维度，而不是一个混合类型切换：

- `专业`
- `用途/属性`

其中：

- `专业` 是主分组
- `用途/属性` 是辅助筛选

推荐 `专业`：

- 总图
- 建筑
- 结构
- 给排水
- 暖通
- 电气
- 智能化
- 消防
- 人防
- 幕墙
- 景观
- 精装
- 市政配套
- 其他

推荐 `用途/属性`：

- 施工执行
- 送审报批
- 变更修订
- 竣工归档

说明：

- 页面默认先按 `专业` 看
- `用途/属性` 只作为筛选或标签，不作为唯一主分类
- 更符合建筑工程实际图纸组织方式

每个专业图纸包卡片展示：

- 专业名称
- 图纸包名称 / 编号
- 当前最新版本
- 当前有效版本状态
- 送审要求
- 审图状态（如适用）
- 图纸齐套度
- 缺漏数
- 变更中数
- 当前有效版是否可用于施工
- 当前有效版是否可用于验收（仅竣工归档类）

说明：

- 主视图看“专业包准备度”
- 不是直接把每张图纸都平铺
- 页面首先展示“当前有效版本”，历史版本折叠到详情区
- 只有图纸包是主视图对象，单张图纸不直接上首页

## 6.4 详情处理区

点击图纸记录后，右侧抽屉或弹窗打开。

建议分 6 个区块：

1. 基本信息
2. 版本与审图信息
3. 出图与交付信息
4. 关联影响
5. 版本变更窗口
6. 历史记录

### 基本信息

- 图纸名称
- 专业类型
- 当前版本
- 图纸说明
- 设计单位
- 设计负责人

### 版本与审图信息

- 审图单位
- 审图状态
- 审图报告编号
- 审图日期
- 审图意见
- 当前版本是否为有效版
- 来源版本 / 上一版
- 本次修订原因
- 是否需要送审

### 出图与交付信息

- 图纸状态
- 图纸日期
- 计划送审 / 通过日期
- 实际送审 / 通过日期
- 当前是否可用于现场
- 当前是否可用于验收 / 归档

### 关联影响

- 默认显示图纸包级影响摘要
- 仅对关键联动包展开具体关联对象
- 施工执行类包显示任务影响
- 报批资料类包显示前期证照关联
- 竣工归档类包显示验收 / 归档关联

### 版本变更窗口

建议提供独立窗口或独立抽屉页签，用于专门展示版本与变更。

窗口建议展示：

- 版本时间轴
- 当前有效版
- 历史版本链
- 本次变更原因
- 影响范围
- 替代关系
- 是否已同步现场
- 是否已同步相关任务 / 验收

结论：

- 版本管理值得有独立窗口
- 但不一定要独立顶级页面
- 更适合做成图纸包详情中的独立重区块或弹窗

### 历史记录

- 版本变更
- 状态变化
- 审图意见变化
- 送审 / 通过时间记录

## 7. 建议状态体系

当前字段里已有：

- `DrawingStatus`
  `编制中 / 审图中 / 已通过 / 已驳回 / 已出图 / 已作废`
- `ReviewStatus`
  `未提交 / 审查中 / 已通过 / 已驳回 / 需修改`

继续保留“双状态”设计，但语义更清楚：

- `图纸状态`
  反映图纸自身生命周期
- `审图状态`
  反映审查流程结果

推荐页面主状态映射逻辑：

- `编制中`
  图纸未成型
- `待送审`
  图纸已形成但尚未送审
- `审图中`
  已送审，等待结果
- `需修改`
  审图反馈后需调整
- `已通过`
  审图通过
- `已出图`
  图纸正式出图可用
- `已作废`
  历史版本或废弃版本

实现上可以继续兼容现有 `status + review_status`，但页面展示建议收敛成上述主口径。

补充一点：

- 对 `requires_review = none` 的图纸
  页面不强制展示审图节点，只保留“审图不适用”
- 对 `mandatory / optional`
  才进入审图推进表达

## 8.1 图纸包下“应有项”如何实现

“应有项”不建议做成很重的 BIM 级对象体系，直接采用轻量模板清单。

建议实现方式：

1. `图纸包模板`
   先定义某个图纸包通常应包含哪些图纸项
2. `项目实例化`
   项目选定图纸包后，系统生成该包的应有项清单
3. `项目级可编辑`
   用户可以删除不适用项，或补充项目特有项

每条应有项最小字段建议：

- `id`
- `package_id`
- `item_code`
- `item_name`
- `discipline_type`
- `is_required`
- `current_drawing_id`
- `current_version`
- `status`
  `缺失 / 已上传 / 已通过 / 已出图 / 不适用`
- `notes`

系统自动判断逻辑：

- `缺漏`
  必需项没有 `current_drawing_id`
- `齐套`
  所有必需项都已有当前有效版
- `齐套度`
  已满足必需项数 / 应有必需项数

这套方式的好处：

- 不需要一开始就把所有图纸关系建得很重
- 能比较直接回答“图纸有没有缺”
- 能比较稳定地算出“齐套度”
- 项目又保留一定调整空间

建议的产品规则：

- 图纸包层是首页主对象
- 应有项层只在详情和台账深层展开
- 首页只展示 `应有 / 已有 / 缺漏 / 齐套度`

## 8.2 版本管理如何落地

版本管理是施工图纸板块的核心能力之一，不应该只停留在一个 `version` 文本字段。

产品上应体现为三层：

1. `当前有效版`
   页面主视图默认只看当前有效版
2. `历史版本链`
   详情中能看到 V1.0 -> V1.1 -> V2.0 的修订链
3. `版本影响说明`
   每次修订为什么变、影响哪些专业、是否替代旧版

建议页面体现方式：

- 图纸包卡片展示 `当前版本`
- 台账默认显示 `当前有效版本`
- 详情抽屉有单独 `版本变更窗口`
- 历史版本默认折叠，不和当前版混排
- 当图纸被新版本替代时，旧版自动弱化为“历史版 / 已作废 / 已替代”

建议记录的最小版本信息：

- `drawing_code`
- `version`
- `revision_no`
- `parent_drawing_id`
- `is_current_version`
- `issued_for`
- `change_reason`
- `effective_date`
- `superseded_at`

建议的产品规则：

- 一个图纸编码在同一时点只能有一个 `当前有效版`
- 施工现场默认读取“当前有效版”
- 验收 / 归档默认读取“竣工图当前有效版”
- 版本替换必须有明确的替代关系，不能只靠人工备注判断

## 8.3 变更与是否影响工期

按当前最小方案，变更只保留一个轻量能力，不展开成复杂影响树。

建议最小字段：

- `has_change`
- `change_reason`
- `schedule_impact_flag`

其中：

- `has_change`
  标记当前图纸包或图纸记录是否发生变更
- `change_reason`
  记录简短原因
- `schedule_impact_flag`
  是否影响工期

`是否影响工期` 建议做成自动化，但使用轻规则，不做过深智能判断。

建议自动判定规则：

- 当前有效版发生变更
- 且该图纸包已关联进行中的任务
- 且变更后图纸状态不是 `已出图可施工`
  或关键应有项变成 `缺失 / 待修改 / 待通过`

满足以上条件之一时：

- `schedule_impact_flag = true`

用户仍可人工覆写为：

- `确认影响`
- `确认不影响`

所以更准确地说：

- 系统先自动判定
- 用户可人工确认修正

## 8.4 如何简化展示但不遗失信息

建筑工程图纸非常庞杂，页面不能按“每张图纸一上来全展开”来做。

建议采用三级展示：

1. `图纸包`
   页面主视图层
2. `图纸记录`
   台账层
3. `版本变更`
   详情层

这样做的结果：

- 首页只看图纸包状态、齐套度、当前有效版
- 台账看单张图纸
- 详情看版本、审图意见、变更链、历史记录

不会丢失信息的关键在于：

- 不是删掉历史信息
- 而是通过层级折叠和逐级展开控制信息密度

## 8.5 图纸要填的信息是否过多

如果把所有字段都放在一个新增表单里，确实会过重。

建议改成：

- `核心字段`
  新建时必填
- `扩展字段`
  进入详情后补录

建议新建时核心字段只保留：

- 图纸包
- 图纸名称 / 编号
- 专业类型
- 用途/属性
- 当前版本
- 当前状态
- 送审要求
- 计划送审或出图日期

以下进入详情补录：

- 设计单位 / 负责人
- 审图单位 / 审图报告
- 审图意见
- 实际送审 / 通过日期
- 变更原因
- 备注

## 9. 与现有系统的联动

## 9.1 与任务管理

需要联动，而且这是施工图纸最重要的系统联动之一。

建议联动规则：

- `施工执行类`、`专项送审类`、`变更修订类` 图纸包可映射为任务条件满足
- 相关图纸包被驳回、需修改、作废时可解除相关条件满足
- 任务页应能看到 `drawing` 类条件

结论：

- `10.6` “图纸与任务条件联动”方向应保留

## 9.2 与前期证照

建议弱联动。

适合的联动方式：

- 某些报批资料类图纸包可关联前期证照项作为背景信息
- 在详情中显示“相关证照”

不建议：

- 把图纸主视图做成证照推进视图

## 9.3 与验收时间轴

建议弱联动。

适合的联动方式：

- `竣工图` 的完成、归档状态可作为验收前置条件来源
- 验收详情中可看到相关竣工图准备情况

不建议：

- 把所有施工图直接做成验收节点

## 9.4 与预警 / 风险 / 问题

建议联动，但不强耦合。

适合进入预警的场景：

- 审图临期
- 审图逾期
- 多轮驳回
- 长期未出图

适合进入问题 / 风险的场景：

- 关键专业图纸长期未通过，影响主体施工
- 图纸多轮退审，导致施工计划受阻
- 重大设计变更影响工期或成本

## 9.5 常见难题与方案覆盖

施工图纸管理常见难题至少包括：

- 版本管理混乱
- 现场不知道哪个是当前有效版
- 图纸是否缺漏、是否齐套不清楚
- 该送审的图纸不清楚
- 审图意见闭环不清楚
- 变更修订未同步到现场
- 竣工图收集和归档滞后
- 哪些图纸真正卡任务、卡验收不清楚

本方案对应解决方式：

- `版本管理混乱`
  通过 `当前有效版 + 历史版本链 + 版本变更窗口`
- `图纸是否缺漏`
  通过 `图纸包模板 + 应有/已有/缺漏统计`
- `该送审的图纸有哪些`
  通过 `review_mode + review_basis` 自动判定
- `图纸是否齐全`
  通过图纸包齐套度和缺漏清单
- `哪些图纸影响前期证照 / 任务 / 验收`
  通过图纸包级规则判断和摘要展示，不逐张强制标记
- `现场拿错版本`
  通过“当前有效版唯一”规则和历史版弱化
- `变更同步不到位`
  通过版本变更窗口记录影响范围与替代关系
- `竣工图拖延`
  通过竣工归档类包单独跟踪并接入验收前置

结论：

- 目前这套方案已经覆盖了大部分图纸管理常见难题
- 真正还需要后续专项细化的，是“图纸包模板库”和“版本变更规则”

## 9.6 页面是否能清晰反映这些问题

如果按本方案落地，页面可以较清晰地反映这些问题。

建议页面上明确提供 5 个高频视角：

- `缺漏视图`
  看哪些图纸包未齐套
- `送审视图`
  看哪些图纸包必须送审、当前送审到哪一步
- `版本变更视图`
  看当前有效版和历史替代链
- `任务影响视图`
  看哪些图纸包卡任务
- `验收影响视图`
  看哪些竣工图卡验收

这样用户不需要读很复杂的图，也能快速定位问题。

## 9.7 图纸问题判别是全自动还是半自动

我的判断是：

- `基础问题识别` 可以全自动
- `业务影响判断` 更适合半自动

### 可以全自动的部分

- 图纸包是否缺漏
- 图纸包是否齐套
- 哪些图纸包必须送审
- 是否逾期送审 / 逾期通过
- 是否存在多轮退审
- 是否没有当前有效版
- 哪些图纸包当前卡任务
- 哪些竣工图当前卡验收
- 哪些变更自动判定为可能影响工期

这些都可以依赖规则、模板、日期和联动关系自动识别。

### 更适合半自动的部分

- 某次图纸变更是否属于重大变更
- 变更是否已经实质影响工期、成本、施工方案
- 某个缺漏项是否确属缺漏，还是该项目不适用
- 某个图纸包是否已足够支撑现场施工
- 某个图纸问题是否应升级为风险

这类判断带强业务语境，不建议完全自动闭环。

### 推荐方案

图纸问题判别建议采用：

- `规则自动识别`
- `人工确认升级`

也就是：

1. 系统先自动标出异常
2. 用户再确认是否升级为问题 / 风险 / 阻塞

这比完全自动化更适合建筑工程图纸域。

## 10. 页面视觉建议

施工图纸页应保持当前系统视觉风格，不单独再造一套设计。

需要对齐：

- `PageHeader + Breadcrumb + shell-surface`
- 白底卡片
- `slate` 文本层级
- 蓝色主强调色
- `badge-base` 标签
- 与当前 `PreMilestones / Dashboard / Reports` 一致的圆角、边框、阴影层级

主视图建议的视觉语言：

- 专业图纸包卡片用白底浅边框
- 审图状态用统一状态徽标
- 当前关键卡点用橙 / 红标签提示
- 版本号使用小型标签
- 历史版本弱化展示，不与当前版本抢主视觉

补充要求：

- 首页摘要卡片继续沿用当前系统的白底轻卡片风格
- 版本变更窗口沿用现有抽屉 / 弹窗语言
- 问题标签通过 `badge` 和摘要数字表达，不用大面积警示色铺满页面
- 信息录入区和问题识别区要视觉分层清楚，避免所有信息堆在一屏

## 12. 结论

施工图纸后续最适合被做成“施工过程图纸准备度工作台”，而不是证照页里的附属 tab，也不是默认的关系图页。

这套形态最符合实际业务：

- 管理层能看专业图纸准备情况
- 执行层能维护图纸版本和审图状态
- 系统能把图纸状态联动到任务条件和后续验收前置
- 同时又保持和前期证照、验收时间轴的清晰边界
