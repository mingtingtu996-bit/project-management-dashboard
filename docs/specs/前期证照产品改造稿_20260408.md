# 前期证照产品改造稿_20260408

> 适用范围：`证照管理 / 前期证照`
>
> 文档定位：用于统一前期证照的产品定位、页面结构、字段口径与系统联动方式。

配套文档：

- `前期证照字段模型+状态流转稿_20260409.md`
- `前期证照页面原型稿_20260409.md`

## 1. 文档目标

前期证照不再被定义为“证照 + 图纸 + 关系图”的混合页，而是一个专门服务于项目开工前准备的业务板块。

本稿聚焦的问题只有一个：

- 如何让项目团队一眼看清“四证办理进度、当前卡点、预计具备开工条件时间”。

结合 `四证办理流程图.pdf`，前期证照板块聚焦四类核心证件：

- `土地证`
- `用地规划许可证`
- `工程规划许可证`
- `施工许可证`

说明：

- 本页只处理开工前证照
- 不承接施工过程中的图纸管理主视图
- 不承接竣工前后的验收流程主视图
- 三者都保留在 `证照管理` 父模块下，但作为平级业务页存在

## 2. 产品定位

前期证照板块的定位不是“流程图页”，也不是“资料仓库页”，而是：

- `四证推进看板`
- `办理台账`
- `开工准入判断入口`

它主要回答 4 个问题：

- 四证现在办到哪一步
- 哪张证卡住了项目开工
- 卡在什么环节、什么审批部门
- 预计什么时候具备开工条件

## 3. 页面目标

页面既要适合管理层看全局，也要适合执行层盯办理节点。

因此建议采用两层结构：

1. `开工准入总览`
2. `四证推进看板 + 办理台账`

其中：

- 总览负责看结果
- 看板负责看进度
- 台账负责看细项与维护数据

## 4. 页面结构

## 4.1 顶部总览区

建议展示：

- 四证完成数
- 当前卡点证件
- 预计具备开工条件日期
- 逾期事项数
- 待补正事项数
- 本周推进事项数

文案示例：

- `开工准入总览`
- `四证完成：2 / 4`
- `当前卡点：施工许可证`
- `预计具备开工条件：2026-06-18`

## 4.2 四证推进看板

建议固定四行，一证一行：

- 土地证
- 用地规划许可证
- 工程规划许可证
- 施工许可证

每行按办理阶段推进，不画复杂关系图。

建议的阶段列：

- `资料准备`
- `内部报审`
- `外部报批`
- `批复领证`

每个证件在本行展示：

- 当前所在阶段
- 当前状态
- 计划日期
- 审批部门
- 是否受阻
- 下一动作

## 4.3 办理台账

点击任一证件后，进入该证件的办理台账。

台账建议展示：

- 办理事项名称
- 所属证件
- 当前环节
- 当前状态
- 计划完成日期
- 实际完成日期
- 审批部门
- 经办说明
- 阻塞原因
- 操作

## 4.4 四证交叉事项与箭头依赖如何处理

四证流程并不是四条完全独立的线。你提到的流程图里很多节点有明确箭头，说明：

- 有些办理事项是多个证件共用的
- 有些办理事项完成后，会同时推动多个证件前进
- 有些证件之间存在明确前后依赖

因此系统里不能只做“四行独立证件”。

正确做法是：

- 页面主视图仍保持四证四行，方便理解
- 底层数据增加 `共享办理事项池 + 证照依赖关系`

建议拆成两层：

1. `证件层`
   土地证、用地规划许可证、工程规划许可证、施工许可证
2. `办理事项层`
   项目备案、地名确认、用地手续、规划报批、施工许可前置资料等

其中某些办理事项可以映射到多个证件。

例如：

- 一项共享资料准备完成
  可同时减少两个或多个证件的待办量
- 一项前置批复未完成
  会同时阻塞后续多个证件

结论：

- 页面不画复杂流程图
- 但系统底层必须承认“四证之间有共享事项和依赖箭头”

## 4.5 页面上如何体现交叉事项

为了既看清四证，又不把页面做成复杂拓扑图，建议这样表现：

- 四证推进看板仍按四行展示
- 首页在四证看板上方或中部增加 `共享事项条带`
- 共享事项条带按事项横向展示，并跨越它实际影响的证件行
- 受同一共享事项影响的证件卡片使用同色短条或同源标记高亮
- 点击任一证件或共享事项时，相关证件行和共享事项同步高亮，非相关内容弱化
- 对前置阻塞，在证件行上显示 `受 X 事项阻塞`
- 详情抽屉中增加 `轻量依赖矩阵`
- 轻量依赖矩阵按“证件 x 共享事项”展示勾连关系
- 详情抽屉中继续展示：
  - 当前证件依赖哪些共享事项
  - 当前共享事项还影响哪些其他证件

也就是说：

- 主页面看“结果 + 交叉影响”
- 详情里看“交叉逻辑 + 依赖矩阵”

这样既保留流程图里的箭头逻辑，又不会把首页做复杂

推荐的最终表达组合是：

- `四证推进看板`
- `共享事项条带`
- `同源标记高亮`
- `详情抽屉中的轻量依赖矩阵`

## 5. 业务状态建议

前期证照不适合直接复用验收状态机。

建议主状态：

- `待启动`
- `资料准备中`
- `报审中`
- `报批中`
- `待补正`
- `已批复`
- `已领证`
- `已失效`
- `已作废`

建议说明：

- `待补正` 是关键状态，必须明显可见
- `已批复` 与 `已领证` 分开，方便处理批复后正式取证的时间差
- `已失效` 需要单独保留，支撑有效期类证照管理

## 6. 字段口径

## 6.1 证件实例字段

建议核心字段：

- `id`
- `project_id`
- `certificate_type`
- `certificate_name`
- `status`
- `current_stage`
- `planned_finish_date`
- `actual_finish_date`
- `document_no`
- `issuing_authority`
- `responsible_authority`
- `next_action`
- `next_action_due_date`
- `is_blocked`
- `block_reason`
- `remarks`
- `created_at`
- `updated_at`

## 6.2 共享事项与依赖字段

为了表达流程图中的交叉箭头，建议增加两类对象：

1. `certificate_work_item`
   办理事项
2. `certificate_dependency`
   证件或事项之间的依赖

建议 `certificate_work_item` 最小字段：

- `id`
- `project_id`
- `item_name`
- `item_stage`
- `status`
- `planned_finish_date`
- `actual_finish_date`
- `approving_authority`
- `is_shared`
- `block_reason`
- `next_action`

建议 `certificate_dependency` 最小字段：

- `id`
- `project_id`
- `predecessor_type`
  `certificate / work_item`
- `predecessor_id`
- `successor_type`
  `certificate / work_item`
- `successor_id`
- `dependency_kind`
  `hard / soft`

说明：

- 页面主体仍按证件展示
- 共享事项和依赖主要用于驱动证件推进与阻塞判断
## 6.3 责任单位口径

本页中的 `责任单位`，按你的要求，不单独拆内部责任单位字段。

这里直接定义为：

- `责任单位 = 外部审批 / 主管部门`

也就是：

- 土地证对应的主管部门
- 用地规划许可证对应的主管部门
- 工程规划许可证对应的主管部门
- 施工许可证对应的主管部门

页面中建议统一显示为：

- `审批部门`
或
- `主管部门`

不建议继续使用歧义较大的“责任单位”字样作为主字段标题。

如果后续确实需要追内部谁负责推进，可以再补：

- `经办人`
- `跟进备注`

但不进入当前核心字段。

## 7. 是否需要流程图感

前期证照仍然需要“时间感”，但不需要做成关系图或流程轴主视图。

原因：

- 四证业务天然是分证件分事项推进
- 但底层又存在共享事项和依赖箭头
- 管理重点是卡点、批复、补正、领证时间
- 复杂关系图不会提升理解效率，反而会稀释证件办理本身

因此更适合的形式是：

- 上层：总览
- 中层：四证推进看板
- 下层：办理台账
- 详情层：共享事项与依赖逻辑

不是：

- 复杂流程图
- 图谱式关系页

## 8. 与当前系统的联动

## 8.1 与预警系统

需要联动。

建议自动进入预警的场景：

- 计划节点临期
- 办理事项逾期
- 同一事项多次补正
- 关键证件未取得且影响开工

结论：

- 证照页是事实层
- 预警系统是提醒层

## 8.2 与风险系统

建议联动，但不强耦合。

适合升级为风险的场景：

- 关键证件长期未推进
- 跨部门协调长期无结果
- 已明显影响项目开工面或总工期

联动方式建议：

- 证照项可一键转风险
- 风险回看时能反查对应证照事项

## 8.3 与问题系统

建议联动。

适合进入问题池的场景：

- 某个具体报批事项被退件、补正、卡批
- 某个环节存在明确待办障碍

联动方式建议：

- 从证照项一键创建问题
- 问题关闭后回写证照项处理结果摘要

## 8.4 联动原则

前期证照页不依赖风险 / 问题才能运行，但应该成为这些模块的重要上游来源。

因此采用：

- `弱耦合联动`

而不是：

- `强绑定共用状态`

## 9. 与施工图纸、验收时间轴的关系

三者都放在 `证照管理` 父模块下，但边界明确：

- `前期证照`
  开工前关注
- `施工图纸`
  施工中关注
- `验收时间轴`
  竣工前后关注

建议关系：

- 导航上串起来
- 数据上不强行混写
- 仅在必要处做摘要跳转或前置满足联动

## 10. 视觉建议

视觉上继续沿用当前系统风格：

- `PageHeader + Breadcrumb + shell-surface`
- 白底卡片
- 浅灰边框
- 蓝色强调色
- `badge-base` 标签

建议避免：

- 关系图式大画布
- 过重的流程箭头
- 图纸和证照混排在同一主内容区

## 11. 实施建议

前期证照后续改造建议按以下顺序：

1. 先明确页面边界
   让 `前期证照 / 施工图纸 / 验收时间轴` 三页彻底分开
2. 再做四证总览与推进看板
3. 再补办理台账与详情维护
4. 最后补预警 / 风险 / 问题联动

## 12. 结论

前期证照最适合被做成“开工前四证推进工作台”，而不是混合关系图页。

这套形态最能服务实际管理目标：

- 看四证是否齐套
- 看卡点在哪
- 看审批部门是谁
- 看什么时候能具备开工条件
