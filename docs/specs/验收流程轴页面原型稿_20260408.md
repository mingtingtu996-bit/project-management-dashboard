# 验收流程轴页面原型稿_20260408

> 适用范围：`证照管理 / 验收时间轴`
>
> 文档定位：承接 `验收流程轴产品改造稿_20260408.md` 与 `验收流程轴字段模型 + 状态机稿_20260408.md`，用于统一页面结构、交互方式与展示逻辑。
> 导航口径说明：侧边栏父级最终显示名统一为 `专项管理`；本文中保留的 `证照管理` 仅用于既有专项命名和技术语境。


## 1. 页面目标

验收页不是单纯的“时间轴页”，也不是复杂关系图页，而是一个面向项目经理与执行人员的验收推进工作台。

页面要同时回答 4 个问题：

- 现在项目整体走到哪个阶段
- 哪些验收项可以并行推进
- 哪些验收项被前置卡住
- 下一步最该处理什么

因此页面采用三层工作方式：

1. `阶段-时间双轴流程板`
   主理解入口
2. `台账视图`
   主编辑入口
3. `详情抽屉`
   主处理入口

## 2. 页面信息架构

## 2.1 页面分区

页面分为 4 个区域：

1. 顶部概览区
2. 筛选与视图工具区
3. 主内容区
4. 右侧详情抽屉

默认进入 `流程板视图`。

## 2.2 顶部概览区

顶部概览区展示项目级摘要，不进入复杂细节。

建议指标：

- 总验收项数
- 当前阶段主分布
- 已通过数
- 整改中数
- 受阻数
- 近 30 天临期项
- 预计完成关键节点日期

页面文案示例：

- `项目验收推进`
- `当前处于：专项验收阶段`
- `关键阻塞：2 项`
- `预计备案完成：2026-09-20`

## 2.3 筛选与工具区

建议控件：

- 项目 / 楼栋筛选
- 范围筛选：项目级 / 楼栋级 / 单位工程级 / 专项级
- 状态筛选
- 仅看阻塞
- 仅看临期
- 时间尺度切换：月 / 双周 / 周
- 视图切换：流程板 / 台账
- 新增验收项

交互原则：

- 筛选作用于流程板与台账
- 抽屉不受切换影响，保持当前选中项

## 3. 主视图：阶段-时间双轴流程板

## 3.1 双轴结构

流程板采用双轴：

- 横轴：计划完成时间
- 纵轴：阶段泳道

建议纵向阶段：

- `准备阶段`
- `专项验收`
- `单位工程验收`
- `备案归档`
- `交付收口`

说明：

- 阶段顺序固定
- 项目可隐藏不适用阶段
- 自定义项必须落入既有泳道

## 3.2 页面骨架示意

建议的骨架如下：

```text
┌──────────────────────────────────────────────────────────────┐
│ 项目验收推进   当前阶段：专项验收   受阻 2   临期 3          │
├──────────────────────────────────────────────────────────────┤
│ 楼栋筛选 | 状态筛选 | 仅看阻塞 | 月/双周/周 | 流程板/台账    │
├──────────────────────────────────────────────────────────────┤
│        时间轴： 8月          9月          10月              │
├──────────────────────────────────────────────────────────────┤
│ 准备阶段        [图审通过] -> [资料齐套]                     │
├──────────────────────────────────────────────────────────────┤
│ 专项验收        [消防]   [人防]   [规划]   [电梯]            │
│                 └───── 分流 / 并行 / 合流结构 ─────┘        │
├──────────────────────────────────────────────────────────────┤
│ 单位工程验收    [单位工程竣工验收]                           │
├──────────────────────────────────────────────────────────────┤
│ 备案归档        [竣工备案]                                   │
├──────────────────────────────────────────────────────────────┤
│ 交付收口        [分户验收] [交付准备]                        │
└──────────────────────────────────────────────────────────────┘
```

## 3.3 卡片设计

每个验收项卡片仅展示高频信息。

建议展示字段：

- 验收项名称
- 所属范围
- 当前状态
- 计划完成日期
- 实际完成日期
- 准备度
- 叠加标记

建议卡片层级：

- 主标题：`1#楼消防验收`
- 副信息：`楼栋级 / 消防`
- 状态徽标：`整改中`
- 时间：`计划 09-12`
- 进度：`准备度 80%`
- 标签：`受阻`、`前置未满足`

卡片不直接堆：

- 全部资料清单
- 所有依赖列表
- 全部问题记录
- 长文本备注

## 3.4 关系表达

## 前后关系

前后关系用简单箭头表达：

- 统一样式
- 从左到右
- 不区分虚实线语义

## 并行关系

并行关系不通过文案说明，而通过结构表达：

- 一个节点后进入 `分流`
- 并行节点横向或纵向并列
- 并行项完成后汇入 `合流`

建议视觉形式：

- 分流 / 合流点使用小圆点或小菱形
- 连接线使用直角短线
- 同一并行组共用一段淡色背景引导线，但不出现大面积容器框

这样既保留图形关系，又避免“画太满”。

## 3.5 关系展示规则

为了保证清晰度，流程板遵循 5 条约束：

1. 所有关系只从左向右
2. 默认只显示相邻依赖
3. 跨阶段长链只在选中节点时高亮
4. 一屏默认聚焦当前阶段与相邻阶段
5. 同楼栋或同专项过多时优先折叠，不把几十张卡平铺到底

## 4. 图上轻编辑

## 4.1 设计原则

图上可以编辑，但不能自由摆放。

用户看到的是“拖动卡片”，系统执行的是“修改业务属性并自动重排”。

因此：

- 可以拖
- 但不保存任意坐标
- 不允许脱离业务规则悬浮摆放

## 4.2 支持的轻编辑动作

允许的动作：

- 左右拖动
  调整计划完成日期
- 上下拖动到其他阶段
  申请变更阶段
- 拖到另一张卡片后方
  设置前后关系
- 拖入某并行分支
  加入并行组
- 删除依赖线

不允许的动作：

- 任意空白区摆放
- 手工拉满屏交叉线
- 在图上直接编辑全部字段
- 用像素位置代表业务逻辑

## 4.3 自动校准

拖动完成后系统自动执行：

1. 横向吸附到最近时间刻度
2. 纵向吸附到合法阶段泳道
3. 重排并行组
4. 重算分流 / 合流结构
5. 校验非法循环依赖

交互结果：

- 编辑体验接近看板
- 布局结果仍然干净

## 5. 台账视图

台账视图是主编辑入口。

建议字段列：

- 验收项名称
- 标准类型
- 阶段
- 范围
- 计划完成日期
- 当前状态
- 前置完成情况
- 责任单位
- 主管机构
- 阻塞情况
- 操作

台账建议支持：

- 批量改阶段
- 批量改日期
- 批量调整前置
- 批量设置责任单位
- 自定义项新增

说明：

- 结构性编辑优先在台账做
- 流程板主要负责理解与快速微调

## 6. 详情抽屉

点击任一卡片后，右侧抽屉打开。

建议分 5 个区块：

1. 基本信息
2. 前置与依赖
3. 资料与条件
4. 过程记录
5. 联动信息

### 6.1 基本信息

展示：

- 验收项名称
- 类型
- 阶段
- 范围
- 计划 / 实际日期
- 状态
- 责任单位
- 外部主管机构

### 6.2 前置与依赖

展示：

- 前置项列表
- 当前阻塞了哪些后续项
- 是否属于并行组

提供操作：

- 设为前置
- 移除前置
- 加入并行组

### 6.3 资料与条件

展示：

- 资料清单
- 图纸条件
- 证照条件
- 任务条件
- 风险 / 问题清障要求

### 6.4 过程记录

展示：

- 申报记录
- 预约记录
- 验收结论
- 整改记录
- 复验记录
- 归档记录

### 6.5 联动信息

展示：

- 关联任务
- 关联图纸
- 关联前期证照
- 关联风险
- 关联问题
- 关联预警

## 7. 默认视角与信息密度控制

为了避免页面一上来就太乱，建议采用以下默认策略：

- 默认按当前阶段展开
- 默认只显示未来 90 天窗口
- 默认只展示关键节点，不展示所有过程记录
- 默认隐藏跨屏长依赖
- 默认对大量同类节点按楼栋折叠

只有用户主动展开时，再展示更细层级。

## 8. 自定义验收项在页面中的表现

自定义项必须与标准项拥有一致的页面语义。

表现规则：

- 使用同样的卡片样式
- 明确显示 `自定义` 标签
- 必须属于某阶段
- 必须拥有前置或后置定位

新增入口建议在台账中完成，流程板仅提供“插入到某节点前后”的辅助动作。

## 9. 与当前实现的差异

相较当前 `AcceptanceTimeline.tsx`，后续页面原型的变化点如下：

- 取消 `ForceDirectedGraph` 作为主视图
- 取消自由拖拽坐标保存
- 强化阶段与时间双轴
- 关系继续保留在图上，但只保留简单箭头和分流 / 合流
- 详情处理从弹窗式转为统一抽屉式
- 台账成为主编辑入口

## 10. 移动端 / 小屏适配

小屏下不建议完整保留流程板大画布。

建议降级方式：

- 默认进入台账视图
- 流程板切为横向可滚动阶段板
- 抽屉改为全屏面板
- 节点关系只在选中项时显示上下游摘要

移动端重点保证：

- 能看状态
- 能看日期
- 能处理单个节点

不强求完整大图编辑体验。

## 11. 首批实现建议

如果后续进入开发，页面首批建议做这些：

1. 先完成流程板静态布局和台账双视图
2. 再完成详情抽屉
3. 再补拖拽改日期和改阶段
4. 最后再补图上关系轻编辑

顺序原因：

- 先解决“能看清”
- 再解决“能改动”
- 最后再解决“图上轻编辑”

## 12. 与当前应用视觉风格的统一要求

当前应用已经形成较稳定的页面壳与设计语言，后续验收流程轴不应另起一套视觉体系，而应在现有系统风格上延展。

需要对齐的现有视觉基线：

- 页面外层沿用 `PageHeader` + `Breadcrumb` + `shell-surface`
- 主底色继续使用 `bg-slate-50`
- 主要内容卡片继续使用 `bg-white`
- 边框延续 `border-slate-200`
- 文本层级延续 `text-slate-900 / text-slate-500`
- 主要强调色继续使用当前系统蓝色体系
- 标签延续 `badge-base`
- 空状态、占位态继续使用浅灰虚线框与 `shadow-sm`

验收页应避免的视觉偏差：

- 不单独引入重色深背景
- 不改成独立品牌色主题
- 不使用过多渐变、发光、荧光描边
- 不做自由画布式脑图视觉
- 不把关系线做成高噪音信息层

建议的视觉落点：

- 顶部概览区保持当前系统的白底卡片式摘要
- 阶段泳道采用轻度底色区分，但透明度要低，不能压过卡片
- 验收卡片保持白底 + 浅边框 + 状态徽标
- 关系线采用统一中性灰蓝，不做多线型语义
- 当前选中节点通过 `ring`、边框加强或浅蓝背景高亮
- 受阻、临期、逾期通过标签表达，不通过大面积红黄底强行抢视觉

推荐的页面视觉结构：

- 页面级：`shell-surface`
- 摘要卡片：与 `Dashboard`、`RiskManagement` 同一层级语言
- 流程板主体：白底容器 + 清晰网格 + 轻泳道背景
- 抽屉：延续当前 `Dialog / Drawer` 白底、圆角、浅边框体系

结论：

- 验收流程轴是“复杂业务视图”
- 但视觉上仍应看起来像“当前系统的一页”

## 14. 结论

验收流程轴的主页面不应再是复杂自由关系图，而应是“带时间轴的阶段流程板”。

这样可以同时保证：

- 业务顺序清楚
- 并行关系直观
- 时间节奏可见
- 页面信息密度可控
- 后续轻编辑可自动校准

这套原型也能和前面确认的字段模型、状态机、联动规则自然对齐。
