## 一、责任人与责任单位体系增强

### 1.1 任务表增加 `assignee_user_id` 可选外键

- 在 `tasks` 表新增 `assignee_user_id UUID REFERENCES users(id) NULL`
- 保留现有 `assignee`（字符串显示名）作为兼容字段
- 任务表单选择责任人时，优先从项目成员列表（`project_members`）中选择，自动填入 `assignee_user_id`；也允许手动输入名字（此时 `assignee_user_id` 为空）
- 前端展示统一使用 `assignee`（显示名）；`assignee_user_id` 用于后端关联和精准推送

### 1.2 预警送达精准化

- `warningService` 生成通知时：
  - 若任务有 `assignee_user_id`，通知直接送达对应用户
  - 若 `assignee_user_id` 为空，退化为送达项目 `owner` 责任组
- 延期审批通知、关账超期升级通知等管理动作通知不受影响，仍按 `owner` 责任组送达

### 1.3 责任单位跳转打通

- Dashboard 中 `UnitProgressCard`（责任单位快照）的"查看全部"跳转目标从 `GanttView` 改为**责任主体子视图**（§1.8），并默认定位到责任单位维度

### 1.4 责任人与责任单位联动查询

- TaskSummary 的责任主体分析（含责任人和责任单位两个维度）属于业务流程优化清单本身要落地的内容（清单 §4140-4188），不在本文档重复定义
- 本文档仅定义 v2 增量：在清单落地的责任主体分析基础上，支持**交叉查询**——选择某个责任人时能看到其所属责任单位整体表现，选择某个责任单位时能看到其下所有责任人明细
- 交叉查询的展示位置为**责任主体子视图**（§1.8），不再放在 TaskSummary 中
- 数据基础：任务上的 `assignee_user_id`（人）+ `participant_unit_id`（单位，10.8b 正在落地）

### 1.5 责任人与系统用户关联收口

- 当前系统中，任务的 `assignee`、风险/问题的 `responsible_person`、阻碍的相关人等字段均为纯字符串，与 `users` 表无关联
- 收口策略：所有需要"知道这个人是谁"的字段，统一提供可选的 `_user_id` 外键字段关联到 `users.id`
  - 任务：`assignee_user_id`（§1.1 已定义）
  - 验收：`responsible_user_id`（§1.6 已定义）
  - 风险/问题：后续版本视需求补充，v2 不强制要求
- 字符串显示名保留作为兼容和展示用途
- 关联后系统可实现：预警精准送达、按用户维度汇总工作量、操作记录追溯到具体系统用户

#### 实施方案：智能匹配 + 待关联提示

**阶段 1：任务分配阶段（混合输入模式）**

- GanttView 任务表单中，责任人输入框改为 Combobox 组件
- 下拉选项优先展示项目成员列表（从 `project_members` 加载）
- 允许自由输入不在成员列表中的姓名（如外部协作方、还未加入的成员）
- 数据填充规则：
  - 选择项目成员：同时填充 `assignee_name`（显示名）和 `assignee_user_id`（外键）
  - 自由输入：仅填充 `assignee_name`，`assignee_user_id` 为 null
- 当 `assignee_user_id` 为 null 时，表单下方显示提示："该责任人尚未关联账号，可稍后在项目设置中关联"

**阶段 2：成员加入阶段（自动匹配提示）**

- 当新成员通过 `POST /api/projects/:id/members` 加入项目时，后端自动检测未关联的责任人
- 检测逻辑：查询 `tasks` 表中 `assignee_user_id IS NULL` 且 `assignee_name` 与新成员姓名相似的记录
- 前端收到响应后，如果 `suggestedMatches` 不为空，弹出 Toast 提示：
  - "系统发现 5 个任务的责任人为'张三'，是否关联到此成员？"
  - 提供"立即关联"按钮，点击后批量更新相关任务的 `assignee_user_id`

**阶段 3：项目设置（待关联面板）**

- 在 Header 用户菜单的"团队管理" Drawer 中新增 Tab："待关联责任人"
- 位置：与"团队成员"Tab 并列，放在第二个 Tab
- 面板内容：
  - 列出所有 `assignee_user_id IS NULL` 的责任人（按 `assignee_name` 去重）
  - 每行显示：责任人姓名、负责任务数、关联操作下拉框
  - 下拉框选项：当前项目的所有成员
  - 选择成员后，调用 `POST /api/projects/:id/members/link-assignee` 批量更新
- 如果所有责任人均已关联，显示："✓ 所有责任人均已关联账号"
- 符合 §5.2 的设计思路：所有项目配置通过 Header 用户菜单 → Drawer 访问

**废弃文件清理**

- `client/src/pages/Settings.tsx` 已废弃（路由重定向到 `/notifications`），应删除
- `client/src/pages/__tests__/Settings.test.tsx` 对应的测试文件也应删除
- Settings 页面中的"数据质量权重"和"数据导入导出"功能暂不提供前端入口（后端 API 已实现，按需后续补充）

**API 新增**

```typescript
// POST /api/projects/:id/members/link-assignee
// 批量关联责任人到系统用户
{
  assigneeName: string,  // 责任人姓名
  userId: string         // 要关联的用户 ID
}
// 返回：{ linkedTaskCount: number }
```

### 1.6 验收流程轴责任人挂钩

- 验收字段模型中增加可选的 `responsible_user_id UUID REFERENCES users(id) NULL`
- 保留 `responsible_person`（字符串）作为显示名和兼容
- 验收项目的推进责任人可与 TaskSummary 的责任主体分析打通
- 验收的外部机构（质监站、消防局等）不入 `participant_units`，保持为验收域自有字段

### 1.7 施工图纸与前期证照

- **施工图纸**：不补 `user_id` 外键。图纸域的主体是设计单位/设计院，与项目内部责任体系是不同维度
- **前期证照**：不补 `user_id` 外键。审批部门是外部政府机构，与项目内部责任体系无关

### 1.8 责任主体独立子视图

#### 定位与入口

- 责任主体分析从 TaskSummary 迁出，作为**任务管理导航组下的独立子视图**，与 GanttView（任务列表）、TaskSummary（任务总结）并列
- TaskSummary 不再包含责任主体分析区，专注于"结果总结 → 月度兑现"主线
- 导航结构：

```
任务管理（侧边栏）
  ├── 任务列表（GanttView）—— 执行维度
  ├── 任务总结（TaskSummary）—— 结果维度
  └── 责任主体 —— 人/单位维度
```

#### 页面内容结构

页面采用纵向两区块结构（同页纵向 + 吸顶导航 + 锚点跳转），顶部提供责任人/责任单位维度切换。

**参建单位管理（按钮 + 弹窗模式）**

- 保持当前按钮弹窗的交互模式，不单独占一个页面区块
- PageHeader 区提供"参建单位管理"按钮，点击弹出 Dialog/Drawer，承接 CRUD 操作
- 数据：`participant_units` 表

**区块一：责任主体监控（v2 新增 4 个能力）**

> 监控在上、分析在下——项目经理打开页面先看"谁现在需要关注"，再下钻看"历史表现如何"。

1. **健康画像与趋势**
   - 每个责任主体展示核心指标（按时完成率、延期任务数、活跃阻碍数）
   - 增加**趋势箭头**：本周 vs 上周的按时完成率变化方向（上升/下降/持平）
   - 数据来源：`task_progress_snapshots` 按周聚合

2. **责任主体异常预警**
   - 在 `warningService` 中新增责任主体级别的汇总预警规则
   - 触发条件（满足任一）：
     - 某责任人/单位的按时完成率连续两个统计周期低于 60%
     - 某责任人/单位的活跃延期任务数 ≥ 3
     - 某责任人/单位本月承诺的重点项中未兑现数 ≥ 2
   - 预警送达 owner 责任组
   - 预警内容为汇总级别（如"张三近两周按时完成率为 45%，低于项目平均 72%"），不是逐条任务通知

3. **关注标记**
   - 项目经理可对表现异常的责任人/单位标记"关注"
   - 标记后系统每周推送该主体的表现变化摘要
   - 项目经理手动取消关注，或指标恢复健康水平后系统建议取消

4. **恢复确认**
   - 责任主体异常预警在指标恢复后自动消除（复用 §6.2 的通知自动消除机制）
   - 恢复标准：连续一个统计周期保持健康水平，不是单次好转即消除

**区块二：责任主体分析（从 TaskSummary 迁移）**

- 清单 §4140-4188 定义的责任主体分析功能整体迁移至此
- 含责任人和责任单位两个维度切换（与区块一共享顶部维度切换）
- 排行、重点关注列表、明细下钻、交叉查询（§1.4）

#### 页面 UI 规范

**维度切换（PageHeader 区）**
- 复用现有 ResponsibilityAnalysisPanel 切换按钮样式：`rounded-lg border bg-white p-1` 内含两个 Button
- 切换影响区块一和区块二的数据维度（责任人 or 责任单位）

**区块一：监控面板**
- 卡片样式：`Card border-amber-100 bg-amber-50/40`（amber 调标识监控/预警属性）
- 监控列表每行：主体名称 | 核心指标 + 趋势箭头（↑ emerald / ↓ rose / ─ slate） | 延期数 | 状态标签（正常/异常）| 操作（标记关注）
- 异常行底色 `bg-rose-50`
- 关注列表：已标记关注的主体单独列出，展示本周表现变化摘要

**区块二：分析面板**
- 卡片样式：`Card border-blue-100 bg-blue-50/40`（blue 调标识分析属性，沿用现有 ResponsibilityAnalysisPanel）
- 摘要指标卡：grid `cols-2 → cols-3 → cols-6` 响应式，复用现有 MetricCard 组件
- 排行图 + 重点关注列表：grid `xl:grid-cols-[1.2fr_0.8fr]`，复用现有布局
- 选中主体明细表：底部展开

#### Dashboard 联动

- Dashboard 的 `UnitProgressCard` 保留为轻量摘要卡，"查看全部"跳转到此子视图（§1.3）
- Dashboard DashboardHero 区新增**数据置信度指标**（§8.5），与健康度并列展示

## 二、甘特图（横道图报表）

### 2.1 定位与入口

- 横道图是任务列表的另一种**只读查看形式**
- 数据 100% 来自任务列表，不可编辑任何内容
- 入口：在 GanttView 页面内增加视图切换按钮（`列表视图 / 横道图视图`）
- 切换后数据源不变，仅改变展示形态

### 2.2 横道图展示内容

#### 左侧固定列

| 列 | 说明 |
|----|------|
| 任务名称 | 缩进体现 WBS 层级，支持折叠/展开 |
| 责任单位 | 取 `participant_unit_name` 或 `assignee_unit` |
| 责任人 | 取 `assignee` |

#### 右侧时间轴区

| 元素 | 说明 |
|------|------|
| **计划时间条**（主条） | `start_date → end_date`，实心色条 |
| **实际时间条**（对比条） | `actual_start_date → actual_end_date`（若有），细条叠在主条下方 |
| **进度填充** | 计划时间条内按 `progress %` 填充深色 |
| **颜色区分** | 正常：灰蓝；延期：红；已完成：绿 |
| **里程碑标记** | 菱形节点标在对应日期位置 |
| **依赖线** | 任务间前置后继关系用箭头连线 |
| **关键路径高亮** | 关键路径上的任务条用红色/橙色边框标识 |
| **横轴缩放** | 支持天/周/月三档缩放 |

#### 双线对比说明

- **第一条**（主条）：项目计划时间（`start_date ~ end_date`），按进度填充
- **第二条**（细条）：实际时间（`actual_start_date ~ actual_end_date`），有值时显示
- 两条线的错位直观反映计划与实际的偏差
- 不增加月计划第三条线——月度兑现偏差的可视化已归入 `Reports > 进度偏差分析 > 月度兑现偏差`

### 2.3 交互

- 只读为主，点击横道可打开任务详情弹窗（复用现有弹窗）
- 不支持拖拽编辑（编辑留给列表视图）
- 支持按楼栋/标段、专业、责任单位筛选（复用列表视图的筛选器）

### 2.4 项目基线查看态横道图

- 基线确认后为冻结的只读快照，增加横道图展示"当初确认的计划全貌"
- 双线模式：
  - 主条：基线时间（`baseline_start ~ baseline_end`）
  - 对比条：当前计划时间（`start_date ~ end_date`）
- 直观展示基线漂移

### 2.5 月度计划查看态

- **不增加横道图**。月度计划的核心是"本月承诺了哪些、兑现了多少"，用列表 + 兑现率指标回答更高效。月度计划的时间跨度只有一个月，横道图信息密度太低

### 2.6 样式规范与技术要求

> 横道图的色值体系与任务列表（GanttView）保持一致，确保用户在列表视图和横道图视图之间切换时，颜色语义不变。

#### 技术选型

| 维度 | 方案 |
|------|------|
| **渲染方式** | SVG——任务数通常 < 500，SVG 支持交互事件更自然 |
| **第三方库** | 自研——需求为只读展示，不需要拖拽编辑，第三方库的大部分能力用不上反而增加包体积和样式冲突风险 |
| **打印/导出** | v2 不做，后续按需增加 |

#### 色值体系（对齐任务列表）

| 元素 | 样式 | 对应任务列表色值 |
|------|------|----------------|
| **计划时间条 — 正常** | 高度 20px，圆角 4px，底色 `bg-gray-100`，进度填充 `bg-blue-500` | 对齐任务列表进度条 `h-2 bg-gray-100` + `bg-blue-500` |
| **计划时间条 — 已完成** | 底色 `bg-emerald-100`，进度填充 `bg-emerald-500` | 对齐已完成状态色 `bg-emerald-100 text-emerald-700` |
| **计划时间条 — 延期** | 底色 `bg-rose-100`，进度填充 `bg-rose-500` | 对齐延期/逾期状态色 `bg-red-100 text-red-700` |
| **计划时间条 — 受阻** | 底色 `bg-amber-100`，进度填充 `bg-amber-500` | 对齐受阻状态色 `bg-amber-100 text-amber-700` |
| **实际时间条（对比条）** | 高度 8px，圆角 2px，`bg-blue-400`，叠在主条下方间距 2px | — |
| **里程碑菱形** | 12×12px 旋转 45°，`bg-amber-500`，边框 `border-white` 2px | — |
| **依赖线** | 1px `stroke-slate-400`，箭头三角，贝塞尔曲线连接 | — |
| **关键路径边框** | 主条增加 2px `border-rose-500` | 对齐关键路径标记 `bg-red-50 border-red-300 text-red-700` |
| **今日线** | 1px 竖虚线 `stroke-rose-400`，标注"今天" | — |

#### 布局尺寸

| 元素 | 规范 |
|------|------|
| **左侧固定列** | 宽度 360px（xl+），lg 收窄至 280px |
| **行高** | 每行 40px（含主条 + 对比条 + 间距） |
| **时间轴标尺** | 顶部固定，`bg-slate-50`，文字 `text-xs text-slate-500` |
| **任务名缩进** | 16px / 层级 |
| **字号** | 任务名 `text-sm`，责任人/单位 `text-xs text-gray-600` |

#### 缩放档位

| 档位 | 横轴刻度 | 适用场景 |
|------|---------|---------|
| 天 | 每格 = 1 天，标注日期 | 短期查看（1-2 周范围） |
| 周 | 每格 = 1 周，标注"W1""W2" | 中期查看（1-3 月范围） |
| 月 | 每格 = 1 月，标注月份 | 全局查看（整个项目周期） |

#### 性能边界

| 场景 | 处理方式 |
|------|---------|
| < 200 任务 | 全量渲染 |
| 200-500 任务 | 虚拟滚动（只渲染可见区域的行） |
| > 500 任务 | 提示用户缩小筛选范围 |

#### 响应式

| 断点 | 行为 |
|------|------|
| xl+ (≥1280px) | 完整展示（左侧列 360px + 右侧时间轴） |
| lg (1024-1279px) | 左侧列收窄至 280px |
| < 1024px | 提示"横道图建议在桌面端查看"，不渲染 |

## 三、登录与权限系统优化

### 3.1 项目级角色体系

保持现有三级体系不变：

| 角色 | 权限范围 |
|------|---------|
| **owner** | 全部权限 + 审批 + 强制操作（强制关账/解锁/确认）+ 延期审批 + 独立修正审批 + 项目删除/转让。每个项目至少一个 owner |
| **editor** | 编辑任务 + 提交延期申请 + 日常操作 |
| **viewer** | 只读查看（当前代码 `permission_level` 已支持 `viewer`） |

- `project_members.permission_level` 合法值保持 `['owner', 'editor', 'viewer']`
- JWT 的 `role` 字段不需要改（它只标识登录身份，项目级权限靠 `project_members.permission_level`）
- 权限中间件 `requireProjectEditor` 检查 `owner / editor`；`requireProjectOwner` 检查 `owner`

### 3.2 Owner 转让

- 后端已实现 `POST /api/members/:projectId/transfer-owner`，逻辑为：旧 owner 降为 editor，新 owner 升为 owner，同时更新 `projects.owner_id`
- **需补前端 UI**：在项目成员管理页面，owner 对其他成员提供"转让负责人"操作入口
- 转让后原 owner 自动降为 editor，不离开项目

### 3.3 公司层面全局角色

- `users` 表新增 `global_role` 字段，取值为 `'company_admin' | 'regular'`，默认 `'regular'`
- 公司驾驶舱数据查询按 `global_role` 过滤：
  - `company_admin`：看到所有项目汇总
  - `regular`：只看到自己作为 `project_members` 的项目汇总
- 首个注册用户自动设为 `company_admin`；后续用户由 `company_admin` 在用户管理中授权

### 3.4 Viewer 权限收口

- 当前代码中 `project_members.permission_level` 已包含 `viewer` 取值
- 需确保所有写操作的权限中间件（`requireProjectEditor` 等）正确拦截 `viewer` 角色
- 前端需对 `viewer` 角色隐藏所有编辑/提交按钮

### 3.5 登录体验总览

```
登录（用户名 + 密码 → JWT）
  │
```

### 6.4 通知冗余清理

#### 冗余一：`snapshot_gap` 双路写入（删除 OperationalNotificationJob 中的该类型）

- **现状**：`OperationalNotificationJob`（每 2 小时）和 `DataQualityJob`（每天）均检测"进行中任务 7 天无进度快照"，写入不同存储路径，用户在通知中心会收到两条内容几乎相同的消息
- **保留**：`DataQualityJob` 的 `SNAPSHOT_GAP` finding（写入 `data_quality_findings`，走数据质量聚合路径）
- **删除**：`OperationalNotificationJob` 中的 `detectSnapshotGapSignals()` 调用及 `task_snapshot_gap` 类型的通知写入
- **理由**：数据质量路径已含聚合摘要+阈值控制，直接逐条推送 `snapshot_gap` 通知信息密度更低、无聚合保护

#### 冗余二：`progress_trend_delay` 与 `data_quality_digest` 部分重叠（调整为互斥）

- **现状**：`ConditionAlertJob` 每小时产生逐条 `progress_trend_delay` 通知；`DataQualityJob` 每天产生 `data_quality_digest` 聚合摘要，同一滞后任务在两条链路中均出现
- **调整**：`ConditionAlertJob` 中的 `scanProgressTrendWarnings()` 只对**关键路径任务**（`is_critical = true`）逐条推送；非关键路径任务的进度滞后信号统一由 `data_quality_digest` 聚合推送
- **理由**：非关键路径任务滞后是管理参考信息，聚合呈现足够；关键路径任务滞后是紧急信号，需即时逐条触达

#### 冗余三：延期审批流程通知叠加（合并 submitted + reminder）

- **现状**：同一延期申请依次触发 3 条通知送达 owner——`delay_request_submitted`（提交即发）→ `delay_request_reminder`（≥3天）→ `delay_request_escalation`（≥5天）
- **调整**：`delay_request_submitted` 通知仅在**审批人未读**时有效；`delay_request_reminder` 改为**仅当 submitted 通知已读但未处理**时才发送（在 `DelayRequestNotificationService` 中查询 submitted 通知的 `is_read` 状态作为前置条件）
- **保留**：`delay_request_escalation`（第 5 天升级）逻辑不变，这是流程升级，不是重复提醒
- **理由**：若 submitted 通知已读但审批人 3 天内仍未处理，催促合理；若 submitted 通知本身都还没读，再发 reminder 只是重复打扰

#### 业务链级冗余：同一延期任务触发多路预警（收敛发送范围）

- **现状**：一个延期中的关键路径任务可同时触发：`critical_path_stagnation`（7 天无进度）+ `progress_trend_delay`（偏差比低）+ `delay_exceeded`（延期次数多）+ 延期审批通知链（共 3 条）——同一任务可累积 5-6 条通知送达 owner
- **调整**：同一任务在同一扫描周期内，按**最高严重级通知唯一化**原则去重：
  - 若已有 `critical_path_stagnation`（critical 级），同周期的 `progress_trend_delay`（warning 级）不再重复发送给 owner
  - `delay_exceeded` 与 `critical_path_stagnation` 针对同一任务同一天重复时，合并为单条摘要通知
- **不变**：延期审批通知链独立运行（属于流程驱动，不参与去重）

## 七、进度监控与数据质量体系

> 建筑工程的特殊性决定了解决问题发生在线下工地，线上系统的闭环角色是"看见、盯住"——主动暴露问题、精准送达责任人、跟踪是否好转、确认恢复正常。数据质量是所有分析和预警的基础，填报不及时、不准确会导致整个管理闭环失效。本章建立三层防线：系统自动监控（过程预判）、异常数据检测（发现虚假）、制度校验（强制纠偏）。

### 7.1 进度滞后趋势预警（系统自动监控）

#### 问题

当前系统的进度预警全部是**结果触发型**——任务到期了才告警（延期预警）、进度 7 天没更新才提醒（快照断层 S1）。缺少**过程监控型**预警——还没到期，但按当前进度推算大概率要出问题。

#### 机制

在 `warningService` 中新增"进度滞后趋势预警"规则，基于**时间消耗率 vs 进度完成率**的偏差比进行预判：

```
偏差比 = 进度完成率 / 时间消耗率
时间消耗率 = 已过天数 / 计划工期天数
进度完成率 = 当前 progress / 100

示例：计划工期 20 天，已过 14 天，进度 30%
  时间消耗率 = 14/20 = 70%
  偏差比 = 30% / 70% = 0.43 → 严重滞后
```

#### 预警级别

| 级别 | 触发条件 | 说明 |
|------|---------|------|
| 关注 | 偏差比 < 0.7 且剩余工期 ≥ 3 天 | 还有调整空间，提醒关注 |
| 警告 | 偏差比 < 0.5 且剩余工期 ≥ 3 天 | 进度严重滞后，需要干预 |
| 严重 | 偏差比 < 0.5 且剩余工期 < 3 天 | 几乎不可能按期完成 |

#### 适用条件

- 仅适用于 `status = 'in_progress'` 的任务
- 仅适用于有明确 `start_date` 和 `end_date` 的任务
- 计划工期 ≤ 3 天的短任务不触发（波动太大，无参考价值）
- 关键路径任务（`is_critical = true`）的阈值更严格：偏差比 < 0.8 即触发关注级

#### 与现有预警的区别

| 预警类型 | 触发时机 | 性质 |
|---------|---------|------|
| 任务延期预警（已有） | 当前日期 > end_date 且未完成 | 结果已发生 |
| 关键路径停滞（已有） | is_critical=true 且 7 天无进度变化 | 行为异常 |
| **进度滞后趋势（新增）** | 偏差比 < 阈值且尚未到期 | **趋势预判** |

#### 预警基准

基准为**计划时间线本身**，不需要额外数据源。`start_date`、`end_date`、`progress` 三个已有字段即可计算。

### 7.2 异常数据检测规则与基准

#### 问题

操作者填报进度时可能出现：填报不及时（长期不更新）、填报不准确（随便填一个数字）、怕麻烦粗填（多个任务填相同值）。当前系统能检测"不填"（快照断层 S1），但不能检测"乱填"。

#### 单任务检测规则

| 规则 | 检测逻辑 | 基准定义 | 说明 |
|------|---------|---------|------|
| **进度跳变** | 单次进度变化 ≥ N% 且此前 14 天无更新 | N = 14 × (100 / 计划工期天数)，即 14 天合理进度累积上限 | 长期不填后突然大幅填报，疑似补填 |
| **工期与进度不匹配** | 偏差比 < 0.3（极端滞后）或 > 2.0（极端超前） | 基准同 §7.1 的时间消耗率 | 区别于 §7.1 的预警阈值，此处阈值更宽松，标记的是"数据很可能不真实" |
| **批量相同值** | 同一责任人单次提交中 ≥ 4 个任务进度值完全相同 | 基准为同次提交行为 | 疑似批量粗填 |
| **逆序进度** | 子任务已完成但父任务进度 < 子任务加权均值 - 20pp | 基准 = 子任务按计划工期加权的进度均值 | 父子数据矛盾 |
| **前后置矛盾** | 前置任务进度 = 0% 但后续任务进度 > 0% | 基准 = 依赖关系定义的执行顺序 | 前置未开始但后续已推进 |

#### 交叉检测规则

| 规则 | 检测逻辑 | 说明 |
|------|---------|------|
| **里程碑与前置任务矛盾** | 里程碑标记已达成，但其关联的前置任务中仍有 progress < 100% 的 | 里程碑达成的前提条件不满足 |
| **条件未满足但已开始** | 任务前置条件未全部满足（`conditions_met_count < conditions_total_count`），但进度 > 0% | 逻辑矛盾：条件未满足不应开工 |
| **同一责任人工作量异常** | 同一责任人同时段负责的进行中任务数 × 平均剩余工期超过物理可能值 | 疑似责任人分配不合理或数据虚填 |
| **同一工作面工序矛盾** | 同一楼栋/位置的两个任务时间重叠，按工序逻辑后者应已开始但进度为 0%，前者已接近完成 | 工序顺序与进度不匹配 |

### 7.3 异常数据处理方式

#### 处理原则

异常数据检测的信息不够重要到值得即时打扰，但不能完全不管。处理方式按"给谁看"分为两条路径：

#### 路径一：聚合推送给项目经理（单任务检测 + 交叉检测的汇总结果）

- 后端定时检测（每日一次），结果写入内部记录表，**不直接在页面上标记**，避免视觉杂乱
- 累积到阈值后，推送**一条聚合通知**给项目经理（owner）
- 推送阈值（满足任一）：
  - 同一责任人/单位的异常条数 ≥ 3
  - 同一工作包/楼栋范围内异常任务占比 ≥ 20%
- 通知内容为范围和严重度，不是逐条明细：如"本周 XX 栋主体结构相关任务存在较多进度异常（涉及 8 条任务、2 个责任人），建议下次现场巡检时重点核实"
- **关键路径任务例外**：关键路径上的任务出现数据异常时即时推送，不等累积阈值

#### 路径二：即时提示给填报人（交叉检测结果）

- 交叉检测发现的矛盾（前后置矛盾、条件未满足但已开始、父子不一致等）**在任务列表的非常态提示栏展示**
- 提示面向填报操作者而非项目经理，目的是让填报人自行发现并修正数据错误
- 提示形式：任务列表提示栏显示"当前有 N 条任务存在数据矛盾需要确认"，点击展开具体条目和矛盾说明
- 操作者修正后提示自动消失
- 与现有非常态提示（月计划待确认、关账超期等）共用提示栏位置

### 7.4 数据置信度

#### 定义

数据置信度是对当前项目进度数据整体真实性和完整性的量化评估，用于告知项目经理"这些数据有多可信"。

#### 计算维度

| 维度 | 计算方式 | 权重建议 |
|------|---------|---------|
| 填报及时性 | 当月进行中任务出现快照断层（≥7 天无更新）的占比 | 高 |
| 异常检测命中率 | 当月触发异常检测规则的任务数 / 当月进行中任务总数 | 高 |
| 交叉一致性 | 当月存在交叉矛盾且未修正的任务数占比 | 中 |
| 进度跳变率 | 当月出现进度跳变的任务数占比 | 中 |
| 更新覆盖率 | 当月至少更新过一次进度的任务数 / 当月进行中任务总数 | 高 |

#### 展示位置

- **Dashboard DashboardHero 区**：与健康度并列展示，如 `健康度: 78 分  |  数据置信度: 85%`。项目经理一眼看到健康度和支撑健康度的数据质量
- **Reports 偏差分析页顶部**：作为分析结论的前提条件标注，如"当月偏差分析基于数据置信度 72% 的数据集"

#### 后果

- 置信度低于阈值时，关账可以完成，但当月兑现率旁标注"数据置信度低，仅供参考"
- 不阻塞关账流程，但影响分析结论的可信度标注

#### 阈值调整

- v2 初始权重和阈值硬编码行业经验默认值
- 系统每月生成置信度维度分解（展示各维度对置信度下降的贡献），帮助项目经理理解哪个维度影响最大
- owner 可在项目设置中手动调整权重（如某项目天然更新频率低，可降低填报及时性权重）
- 后续版本可基于多项目数据积累提供"建议权重"

### 7.5 月末关账数据质量标记

#### 定位

与 Dashboard 和 Reports 中展示数据置信度的逻辑一致——告知项目经理本月数据有无异常，在关账结果上标记置信度。

#### 机制

- 关账完成时，系统自动将当月数据置信度写入关账记录
- 置信度低于阈值的月份，兑现率旁永久标注"数据置信度低，仅供参考"
- 不增加额外的预检步骤，不要求项目经理逐条确认或修正
- 本质与 Dashboard/Reports 中展示置信度相同：让项目经理知道数据质量如何，标记留痕

## 九、性能与技术债务优化

> 以下优化项来自对现有代码的性能审计，按数据一致性风险和用户体感影响排序。

### 9.1 任务删除级联清理（数据一致性）

#### 问题

任务删除时不清理关联记录，导致孤立数据：
- `task_conditions`、`task_obstacles`、`task_progress_snapshots` 中残留已删除任务的记录
- `task_dependencies` 中被删任务仍作为其他任务的前置/后继依赖存在，导致关键路径计算异常

#### 方案

- 在数据库层为 `task_conditions`、`task_obstacles`、`task_progress_snapshots`、`task_dependencies` 的外键设置 `ON DELETE CASCADE`
- 应用层 DELETE tasks 接口中增加防御性清理：`DELETE FROM task_dependencies WHERE predecessor_id=? OR successor_id=?`
- 确保删除后不影响其他任务的依赖关系展示

### 9.2 GanttView 重复数据请求消除

#### 问题

`GanttView.loadTasks()` 在组件 `useEffect` 中独立发起 `/api/tasks` 请求，而 `useProjectInit` 已经加载了相同数据到 store，导致双倍网络请求和双倍 store reset。

#### 方案

- 删除 GanttView 内独立的 `loadTasks` 副作用
- 改为直接消费 `useProjectInit` 已加载到 store 中的 tasks 数据
- 保留手动刷新入口（如用户主动触发重新加载）

### 9.3 GanttView Store 细粒度订阅

#### 问题

GanttView 整体解构 `useStore()`，store 中任何字段变化（如 `currentProject`、`conditions`、`obstacles` 等）均触发整个 3400+ 行组件重渲染。

#### 方案

- 拆分为细粒度 selector：`useStore(s => s.tasks)`、`useStore(s => s.conditions)` 等
- 使用 `useShallow` 或 zustand 的 shallow 比较避免引用变化导致的无效重渲染
- 对计算密集的 `useMemo`（如 `wbsTree`、`flatList`、`filteredFlatList`）确保依赖项精确

### 9.4 图纸包详情接口串行查询优化

#### 问题

图纸包详情接口存在 11 次串行 DB 查询，响应时间约 550ms。

#### 方案

- 将独立表查询改为 `Promise.all([query1, query2, ...])`
- 预计响应时间从 ~550ms 降至 ~100ms

### 9.5 executeSQL LIMIT 正则匹配修复

#### 问题

`executeSQL` 的 LIMIT 处理逻辑中，正则表达式只匹配 `LIMIT ?` 参数化形式，忽略了 `LIMIT 1` 等字面量写法，导致实际执行全表查询后在应用层取 `[0]`。

#### 方案

- 修复正则同时匹配 `LIMIT ?` 和 `LIMIT \d+`
- 或为"取单条"场景提供专用方法（如 `executeSQLOne`），从 DB 层保证只返回一条

### 9.7 同步队列去重

#### 问题

`syncQueue` 与 `pendingOps` 双队列独立 flush，同一实体可能被重放两次，导致数据不一致或重复写入。

#### 方案

- 在 flush 前用 `Map<entityId>` 对队列去重，保留最新操作
- 改动面小于合并为单一队列的方案，且能解决核心问题

### 9.9 GanttView 子组件 React.memo 优化

#### 问题

GanttView 3700+ 行无任何 `React.memo` 使用，子组件（行、单元格、弹窗）在父组件每次重渲染时全部跟着重渲染。

#### 方案

- 对 GanttTaskRows 的行组件、单元格组件、弹窗组件包裹 `React.memo`
- 高计算量的 derived data 用 `useMemo` 缓存
- **与 §9.3 Store 细粒度订阅配套实施**，selector + memo 双管齐下才能真正减少无效渲染

### 9.10 建议执行顺序

```
第一批（数据一致性 + 核心性能，约 7.5h）：
  §9.1 级联删除（2h）→ §9.2 去重请求（1.5h）→ §9.3 + §9.9 Store selector + memo（3h）→ §9.4 Promise.all（2h）

第二批（隐蔽 bug + 体验，约 3h）：
  §9.5 LIMIT 正则（1h）→ §9.7 队列去重（2h）
```

## 十、无障碍优化

### 10.1 交互元素语义标注

#### 问题

全站仅 18 处 `aria-label`、9 处 `role=`，大量 icon-only 按钮缺乏语义标注，屏幕阅读器用户无法理解按钮用途。

#### 方案

- 为所有 `<Button size="icon">` 添加 `aria-label`（如"关闭"、"删除"、"展开筛选"）
- Dialog 组件添加 `aria-labelledby` / `aria-describedby`
- **分批实施**：优先覆盖高频交互页面（GanttView 操作按钮、Dialog 关闭按钮、Header 导航按钮），再扩展到其他页面

### 10.2 动画无障碍适配

#### 问题

无 `prefers-reduced-motion` 媒体查询检测，`animate-pulse`、`transition-all` 等动画可能导致前庭障碍用户不适。

#### 方案

- 在 Tailwind 配置中确认 `motion-reduce:` 变体已启用
- 对 `animate-pulse`、`transition-all`、`page-enter` 等动画添加 `motion-reduce:animate-none` 或 `motion-reduce:transition-none`

### 10.3 数据表格语义化

#### 问题

数据表格（图纸台账、风险列表等）部分使用 `<td>` + 加粗代替 `<th>`，缺少 `<thead>` 和 `scope="col"` 属性。

#### 方案

- 检查所有 shadcn `<Table>` 使用，确保表头使用 `<th scope="col">` 而非 `<td>`
- 确保 `<thead>` 包裹表头行

### 10.4 跳过导航链接

- 在 `<body>` 顶部添加隐藏的 skip-link，Tab 聚焦时可见，跳转到 `<main>`
- 样式：`sr-only focus:not-sr-only focus:absolute focus:top-4 focus:left-4 focus:z-50`

### 10.5 Toast 通知语义修正

- 错误级别 toast 改为 `role="alert"` + `aria-live="assertive"`，确保屏幕阅读器立即播报
- 普通信息 toast 保持 `aria-live="polite"`

### 10.6 Dialog 焦点恢复

#### 问题

shadcn Dialog（基于 Radix）已内置 focus trap，但自定义抽屉组件（如 `DrawingDetailDrawer`）关闭后焦点未恢复到触发按钮。

#### 方案

- 自定义 Drawer/Sheet 组件中增加 `returnFocusRef`，打开时记录触发元素，关闭时恢复焦点

## 十二、前端健壮性优化

### 12.1 localStorage 容量保护

#### 问题

前端 58 处 `localStorage.setItem/getItem` 无容量检查，5MB 上限溢出后静默丢数据。

#### 方案

- 写入时 try-catch，捕获 `QuotaExceededError` 后提示用户清理或切换 Supabase 模式
- 比 `navigator.storage.estimate()` 更直接可靠

### 12.2 关键页面请求取消机制（最小闭环）

#### 问题

`AbortController` 仅在少量位置使用；Dashboard/GanttView 的多个 `useEffect` 请求无取消机制，快速切换页面时导致"组件已卸载但 setState 仍被调用"。

#### 方案

- 在 `apiClient.ts` 封装 `fetchWithAbort(url, signal)`
- **最小闭环**：先覆盖 Dashboard、GanttView 等高频页面，在对应 `useEffect` 中创建 `AbortController` 并在 cleanup 中 abort
- 暂不要求全站一次性补齐

### 12.3 Tab 休眠同步保护

#### 问题

`useDataSync` 的 `setInterval` 轮询 + `storageService` 的 `syncQueue` flush 可能在 Tab 休眠后突然批量触发。

#### 方案

- 使用 `document.visibilitychange` 事件：Tab 隐藏时暂停同步，恢复时先做一次全量对账再恢复增量同步

### 12.4 定时任务重试机制

#### 问题

后端定时任务失败仅 `console.error`，无重试机制、无失败告警。

#### 方案

- `scheduler.ts` 中为每个 job 增加 try-catch + 最多 3 次指数退避重试
- 失败后写入 `job_failures` 表，定期清理避免无限增长

### 12.5 GanttView setTimeout 清理

- 第 181/186 行的 `setTimeout` 未在组件卸载时完整清除
- 将 timer ID 存入 `useRef`，在 `useEffect` 的 cleanup 函数中 `clearTimeout`

### 12.6 backup interval 重启

- `backup.ts` 的 `autoBackupInterval` 是模块级变量，刷新页面后不会自动重启
- 在 `App.tsx` 挂载时检查是否已有 backup interval 运行，否则重新初始化

### 12.7 Suspense 细粒度包裹

#### 问题

`Suspense` 仅在 `App.tsx` 顶层使用（1 处），lazy 组件加载失败时整个应用白屏。

#### 方案

- 每个 `<Route>` 的 lazy 组件独立包裹 `<Suspense fallback={<PageLoader/>}>`，避免单点故障

### 12.8 Vite 分包优化

- `vite.config.ts` 的 `manualChunks` 增加细粒度分包：
  - `charts: ['chart.js']`
  - `icons: ['lucide-react']`
  - `utils: ['date-fns', 'uuid', 'zod']`
- 减少首屏加载体积

### 12.9 运行时错误上报

#### 问题

`ErrorBoundary` 仅 `console.error`，生产环境错误无法感知。

#### 方案

- 自建 `POST /api/client-errors` 端点，在 `componentDidCatch` 中上报
- 本地部署环境不依赖外网服务，比 Sentry 更可控
- 后端写入日志表，支持按时间/页面/错误类型查询

## 十三、数据库与数据安全优化

### 13.1 高频查询字段补建索引

#### 问题

初始迁移仅建了 5 个索引（tasks/risks/milestones/invitations/members 的 `project_id`），后续 77 个迁移虽有补充但大量高频查询字段仍缺索引。

为以下高频查询列补建索引：
- `task_conditions(task_id, is_satisfied)`
- `acceptance_requirements(plan_id)`
- `drawing_packages(project_id, discipline_type)`
- `certificate_work_items(certificate_id)`
- `task_dependencies(predecessor_id, successor_id)`

### 13.3 clearAllData 保护

#### 问题

`clearAllData()` 直接 `localStorage.clear()` 无二次确认、无选择性清除，会清除其他应用的 localStorage。

#### 方案

- 改为仅清除本应用前缀的 key（如 `pm_*` 或 `workbuddy_*`）
- 调用前强制弹出确认对话框，输入项目名称才允许执行

### 13.4 数据导出走后端 API

#### 问题

`dataExport.ts` 仅从 localDb 读取，当 `VITE_STORAGE_MODE=backend` 时导出的是空数据。

#### 方案

- 新增后端聚合接口 `GET /api/projects/:id/export`，打包返回 tasks/conditions/obstacles/milestones 等

### 13.6 数据保留策略

操作日志、任务快照、健康历史等无过期清理，数据无限增长。

#### 方案

  - 清理 365 天前已归档的 `change_logs`
- **关账记录和基线快照不清理**——审计留痕需永久保留

## 十五、业务健壮性补全

### 15.1 核心写链路乐观锁补齐（最小闭环）

#### 问题

GanttView 已实现 version 字段 + ConflictDialog 冲突解决，但 `risks`、`milestones`、`drawings` 等高频多人编辑链路仍可能未做 version 校验，后保存者会静默覆盖前者修改。

#### 方案

- **最小闭环**：先覆盖 `risks`、`milestones`、`drawings` 三类高频写链路，统一 `UPDATE ... WHERE version = ?` + 返回 `409`
- 前端先接入统一冲突提示或轻量 `ConflictDialog`，不要求所有页面一次性做完整冲突工作流
- 其他低频域后续版本再跟进

### 15.2 离线状态提示（最小版）

#### 问题

网络断开后所有操作立即失败，无 UI 反馈。

#### 方案（v2 最小版，不做离线写入）

- 检测网络状态（`navigator.onLine` + `networkMonitor`）
- 离线时在顶部显示"离线模式 — 请恢复网络后操作"条幅
- 阻止写操作并 toast 提示"当前处于离线状态，无法保存"
- 恢复在线后自动隐藏条幅
- **不做离线写入和本地缓存**——完整离线模式留后续版本

### 15.3 表单未保存保护

#### 问题

用户在编辑表单时意外关闭/刷新页面，未保存的数据直接丢失。

#### 方案

- 在表单脏状态时注册 `window.addEventListener('beforeunload', handler)`
- 路由切换使用 React Router 的 `useBlocker` 拦截并弹出确认对话框
- 表单保存成功后移除拦截

## 十七、进度数据消费增强

> 系统已具备完整的进度采集链路（任务快照、变更日志、定时扫描），但数据的消费深度和呈现方式仍有提升空间。本章围绕"让采集到的数据产生更大管理价值"展开，不涉及采集侧变更。

### 17.1 项目整体偏差计算改进（P1）

#### 问题

`server/src/routes/dashboard.ts` 中 `calculateOverallProgress()` 使用**简单算术平均**计算项目整体进度：

```typescript
// 当前实现（有缺陷）
overallProgress = tasks.reduce((sum, t) => sum + (t.progress || 0), 0) / totalTasks
```

这导致一个计划工期 3 天、进度 100% 的小任务与一个计划工期 90 天、进度 10% 的主体结构任务拥有相同权重，项目整体进度严重失真。

#### 方案

改为**工期加权平均**：

```
整体进度 = Σ(task_progress × planned_duration) / Σ(planned_duration)
```

- `planned_duration` = `planned_end_date - planned_start_date`（天数），无日期的任务取默认值 1 天
- 公式含义：工期越长的任务对整体进度的贡献越大，符合项目管理直觉
- 影响范围：仅改变**项目整体进度的聚合公式**（多任务 → 一个项目级数字），单个任务的 `progress` 字段不受影响。所有展示"项目整体进度"的位置因为调用同一个 `calculateOverallProgress()` 函数，结果会自然更新，包括：Dashboard Hero 区、公司驾驶舱项目卡片、周度进度简报、健康度计算中的进度维度、Reports 偏差分析页
- 后端修改点：`calculateOverallProgress()` 函数本身，约 10 行改动；调用方无需修改（输入输出接口不变，只是返回值更准确）

#### 示例

| 任务 | 工期（天） | 进度 | 简单平均贡献 | 加权贡献 |
|------|-----------|------|------------|---------|
| 基坑支护 | 30 | 100% | 50% × 1/2 = 50% | 30×100% = 3000 |
| 主体结构 | 90 | 10% | 10% × 1/2 = 5% | 90×10% = 900 |
| **合计** | 120 | — | **55%**（失真） | **32.5%**（准确） |

### 17.2 周度进度简报（P1）

#### 问题

项目经理需要定期了解项目整体进展、偏差态势和需要关注的异常，但当前系统只有逐条预警通知，缺少汇总性的进度简报。项目经理要么逐个页面查看，要么自行整理信息，效率低。

#### 方案

新增 `WeeklyDigestService`，每周一 09:00 自动生成**周度进度简报**，推送给项目 owner。

#### 简报内容结构

| 区块 | 内容 | 数据来源 |
|------|------|---------|
| **整体状态** | 项目整体进度（加权）、健康度、本周进度变化量 | `calculateOverallProgress()`（§17.1 改进后）、`project_health_scores` |
| **本周完成** | 本周完成的任务数 + 里程碑达成情况 | `task_progress_snapshots`（event_type = task_completed / milestone_completed） |
| **关键路径状态** | 关键路径任务数、受阻数、最近的关键里程碑及其偏差天数 | `tasks.is_critical`、关键路径计算结果 |
| **Top 5 偏差任务** | 偏差天数最大的 5 个任务（标题 + 责任人 + 偏差天数） | `tasks` 表计划日期 vs 当前日期 |
| **责任主体异常** | 本周处于异常状态的责任人/单位名单 | `responsibility_alert_states` |
| **新增风险/阻碍** | 本周新增的风险和阻碍条数 + 最高级别摘要 | `risks`、`task_obstacles`（created_at 在本周范围内） |

#### 技术实现

- 后端：`server/src/services/weeklyDigestService.ts`，定时任务注册到 `scheduler.ts`
- 简报生成后写入独立存储表（如 `weekly_digests`），不写入 `notifications` 表——Dashboard 面板直接查询该表渲染，不走通知链路
- 数据源全部来自已有表和服务，不新增采集逻辑

### 17.3 关键路径偏差升级（P2）

#### 问题

当前 `warningService.scanCriticalPathStagnationWarnings()` 仅检测关键路径任务**7 天无进度变化**的情况，触发单一级别的预警。但关键路径任务的延期天数本身就是重要信号——延期 3 天和延期 15 天的紧迫程度完全不同，应按天数梯度升级预警等级。

非关键路径任务的逐条延期提醒信息量过大，不需要每条都通知；只有影响关键路径的偏差才值得升级。

#### 方案

在 `warningService` 中扩展关键路径预警规则，增加**按延期天数梯度升级**：

| 延期天数 | 预警级别 | 动作 |
|---------|---------|------|
| ≥ 5 天 | info（关注） | 提醒中心通知，送达任务责任人 |
| ≥ 10 天 | warning（警告） | 提醒中心通知，送达项目 owner |
| ≥ 20 天 | critical（严重） | 提醒中心通知 + 周度简报中高亮，送达 owner |

#### 适用条件

- 仅适用于 `is_critical = true` 的任务
- 仅适用于 `status != 'completed'` 的任务
- 延期天数 = `当前日期 - planned_end_date`（正值表示已延期）
- 非关键路径任务不触发此规则（其延期信息通过周度简报 Top 5 偏差任务间接呈现）

#### 与现有预警的关系

| 预警 | 触发条件 | 保留/新增 |
|------|---------|----------|
| 关键路径停滞（已有） | is_critical=true 且 7 天无进度变化 | 保留，检测"不填"行为 |
| 进度滞后趋势（§7.1） | 偏差比 < 阈值且尚未到期 | 保留，检测"大概率要延期" |
| **关键路径偏差升级（新增）** | is_critical=true 且已延期 ≥ 阈值天数 | 新增，检测"已延期多严重" |

三者互补：停滞检测发现"不填"，趋势预警预判"要出问题"，偏差升级评估"已经出了多大问题"。

## 十八、任务列表进度滞后状态

### 18.1 背景与动机

当前任务业务状态体系（`taskBusinessStatus.ts`）中的"受阻"状态（`blocked`）由**是否有活跃阻碍记录**驱动，与进度是否真正受影响无关——任务可以有阻碍记录但进度正常，也可以没有阻碍但进度严重滞后。这使得"受阻"状态对项目经理的决策价值有限。

### 18.2 方案：以进度滞后替代受阻作为主状态

#### 状态码变更

将 `blocked` 状态码替换为 `lagging`（进度滞后），判定逻辑从"有活跃阻碍"改为"偏差比低于阈值"。

阻碍数量降级为行内辅助信息（现有 `obstacleCount` chip），不再决定主状态。

#### 判定逻辑

在 `client/src/lib/taskBusinessStatus.ts` 新增纯客户端函数 `isLaggingTask(task)`，复用以下公式（与 `dataQualityService.detectTrendFindings()` 保持一致）：

```
时间消耗率 = 已过天数 / 计划工期天数
偏差比     = (progress / 100) / 时间消耗率
```

**适用条件**（与服务端趋势预警对齐）：
- `status = 'in_progress'`
- 有明确的 `start_date` 和 `end_date`
- 计划工期 > 3 天（短任务波动大，不触发）
- 尚未到期（已逾期的走现有逾期逻辑，不重叠）

**三档阈值**：

| 级别 | 触发条件 | 标签 |
|------|---------|------|
| 轻度 | 偏差比 < 0.7（关键路径任务 < 0.8） | 进度偏慢 |
| 中度 | 偏差比 < 0.5 | 进度慢 |
| 严重 | 偏差比 < 0.5 且剩余工期 < 3 天 | 进度严重滞后 |

#### getTaskBusinessStatus() 优先级链调整

```
① completed          → status='completed' 或 progress≥100
② lagging（严重）    → 偏差比 < 0.5 且剩余工期 < 3 天
③ lagging（中度）    → 偏差比 < 0.5
④ lagging（轻度）    → 偏差比 < 0.7（关键路径 < 0.8）
⑤ in_progress        → status='in_progress'，无滞后
⑥ pending_conditions → status=todo 且有未满足开工条件
⑦ ready              → status=todo 且开工条件全部满足
⑧ pending            → 兜底
```

### 18.3 视觉呈现规范

#### 主状态标签

| 级别 | 标签文案 | 标签色 |
|------|---------|--------|
| 轻度 | 进度偏慢 | `bg-amber-50 text-amber-600 border border-amber-200` |
| 中度 | 进度慢 | `bg-amber-100 text-amber-700 border border-amber-200` |
| 严重 | 进度严重滞后 | `bg-orange-100 text-orange-700 border border-orange-200` |

#### 行级视觉（整行）

| 级别 | 行样式 |
|------|--------|
| 轻度（进度偏慢） | 不加行底色，仅主状态标签变化 |
| 中度（进度慢） | `border-l-4 border-l-amber-400 bg-amber-50/40` |
| 严重（进度严重滞后） | `border-l-4 border-l-orange-500 bg-orange-50/40` |

轻度不加行底色，避免行数多时视觉噪音过大；中度和严重通过左边框+浅底色提供整行扫视锚点，与现有"逾期"行处理方式一致。

### 18.4 与服务端通知链路的关系

| 维度 | 客户端状态（本章） | 服务端通知链路（§7.1） |
|------|-------------------|----------------------|
| 触发时机 | 实时，任务列表加载时即计算 | 每日定时批处理 |
| 数据源 | 前端已加载的任务字段 | 后端数据库扫描 |
| 输出 | 任务行状态标签 + 行底色 | 通知推送给项目负责人 |
| 持久化 | 无 | 写入 `data_quality_findings` |

两者公式相同、阈值对齐，但完全独立运行，互不依赖。

### 18.5 实施范围

- **修改**：`client/src/lib/taskBusinessStatus.ts`
  - 新增 `isLaggingTask(task)` 函数及 `LaggingLevel` 类型
  - `getTaskBusinessStatus()` 中替换 `blocked` 为三档 `lagging`
  - `TASK_STATUS_THEME` 中替换 `blocked` 条目为三档 lagging 条目
- **修改**：`client/src/pages/GanttView.tsx` 的 `getBusinessStatus()`
  - `lagging` 分支返回对应标签和样式
- **修改**：`client/src/pages/GanttViewRows.tsx` 行样式判定
  - 移除 `status='blocked'` 的行底色条件，新增 lagging 中度/严重的行底色条件
- **不修改**：服务端 `warningService`、`dataQualityService`、通知链路

## 十九、阻碍类型关联跳转

### 19.1 背景

当前 8 种阻碍类型（人员/材料/设备/环境/设计/手续/资金/其他）均为孤立标签，无法从阻碍记录直接导航到系统中处理该问题的对应页面。录入阻碍后，处理路径完全依赖用户自行导航。

### 19.2 关联跳转范围

| 阻碍类型 | 关联目标 | 跳转入口位置 | 理由 |
|---------|---------|------------|------|
| **设计** | 施工图纸台账 | 阻碍详情展开区 → "查看相关图纸" | 设计类阻碍通常指向具体图纸问题，系统已有图纸台账页面，跳转直接有效 |
| **人员** | 不新增跳转 | — | 人员阻碍的处置在线下（协调/调配），系统无对应操作页面 |
| **手续** | 不新增跳转 | — | 手续阻碍对应前期证照流程，但证照页面是独立管理域，阻碍与具体证照的关联关系不确定，强行跳转价值低 |
| **资金** | 不新增跳转 | — | 系统无资金管理模块，无对应页面 |
| **材料 / 设备 / 环境** | 不新增跳转 | — | 物理/现场问题，系统内无对应管理页面，链接无意义 |
| **其他** | 不新增跳转 | — | 兜底类型，无法预判跳转目标 |

### 19.3 交互形式

- 跳转入口放在**阻碍详情展开区**（现有阻碍行点击展开的详情部分），不在阻碍列表行直接显示
- 形式：文字链接按钮（如 `↗ 查看相关图纸`），不占用额外列宽
- 跳转时携带上下文：
  - 设计类 → 跳转到施工图纸台账，并按任务所属专业（`specialty_type`）预筛选

- **修改**：`client/src/pages/RiskManagement.tsx` 阻碍详情展开区，根据 `obstacle_type` 条件渲染跳转链接
- **不修改**：后端数据结构、`obstacle_type` 枚举值

## 二十一、材料管控模块

### 21.1 定位与背景

材料管控服务于建筑工程后期大量专项工程和安装工程（幕墙、机电、消防、装饰装修等）的材料到场跟踪。土建阶段材料标准化程度高，不是重点；专项工程材料品类多、分包商各异、到场时间直接影响施工推进，是当前管理盲区。

材料管控不是采购管理系统，核心职责是：**让项目经理在任务进度视角下感知材料到场风险**。

- 入口：侧边栏专项管理组，与施工图纸台账、前期证照并列
- 材料责任主体：参建单位（分包商），而非个人或专项类型
- 与阻碍挂钩方式：任务阻碍 `obstacle_type='材料'` 的详情区，提供跳转链接至本模块，与 §19 设计→图纸的挂钩方式完全对称；不自动生成阻碍

### 21.2 数据模型

新建 `project_materials` 表：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `project_id` | UUID | 项目 |
| `participant_unit_id` | UUID nullable | 参建单位（分包商），删除时置 null 保留记录 |
| `material_name` | TEXT | 材料名称，自由文本 |
| `specialty_type` | TEXT nullable | 所属专项类型，用于模板预填和筛选 |
| `requires_sample_confirmation` | BOOLEAN | 是否需要定样，默认 false |
| `sample_confirmed` | BOOLEAN | 定样是否完成 |
| `expected_arrival_date` | DATE | 预计到场日期（核心字段） |
| `actual_arrival_date` | DATE nullable | 实际到场日期 |
| `requires_inspection` | BOOLEAN | 是否需要送检，默认 false |
| `inspection_done` | BOOLEAN | 送检是否完成 |

**参建单位删除处理**：`participant_unit_id` 置 null，材料项保留，在列表末尾以"无归属单位"分组显示，提示用户重新关联。

### 21.3 状态推导（纯前端计算）

| 状态 | 判定条件 |
|------|---------|
| **待定样** | `requires_sample=true` AND `sample_confirmed=false` |
| **待到场** | 定样完成或不需要定样 AND `actual_arrival_date` 为空 AND `expected_arrival_date >= 今天` |
| **逾期未到** | `actual_arrival_date` 为空 AND `expected_arrival_date < 今天` |
| **本周到场** | `actual_arrival_date` 在本周内 |
| **待送检** | 已到场 AND `requires_inspection=true` AND `inspection_done=false` |
| **已完成** | 已到场 AND（不需要送检 OR `inspection_done=true`） |

### 21.4 页面设计

#### PageHeader

标题"材料管控"，副标题"跟踪专项工程材料到场状态，关联分包责任主体"，右侧 `[+ 新增材料]` 按钮。

#### 摘要卡（6 张）

`grid-cols-3 xl:grid-cols-6`，依次：

| 待到场 | slate |

### 21.5 新增材料交互

#### 材料清单模板（按专项类型预填）

新增材料时，选择专项类型后，系统提供该专项的常用材料清单供勾选（如选"幕墙工程"自动列出：铝合金型材、Low-E 玻璃、结构胶、五金件）。用户勾选后批量创建，可删减调整。

预置模板覆盖常见专项：幕墙、机电、消防、装饰装修、电梯、给排水。

#### 批量录入模式

提供表格式批量录入 UI（逐行填写），一次可录入多条，与逐条弹窗模式并存。适合分包商合同签订后集中录入的场景。

### 21.6 到场提醒（自动推算提前量）

后端每日扫描，找该参建单位下最早将开始（`status='todo'/'in_progress'`）的任务，计算：

```
缓冲天数 = 任务计划开始日期 - expected_arrival_date
```

当缓冲天数 ≤ 7 天时，触发提醒，推送给项目 owner，内容示例：

> 【材料未到场】铝合金型材 / 绿城幕墙 · 相关任务将在 5 天后开始

- 若该参建单位无关联任务，退化为到场前 5 天固定提醒
- 同一参建单位多条材料在同一天聚合为一条通知；对尚未逾期、仅进入提醒窗口的材料项，同一材料项同一自然周内只发一次提醒
- 逾期未到场时每日推送，直到填写 `actual_arrival_date`

### 21.7 Reports 集成

在 Reports 页面新增"材料到场率"分析区块：

- 按参建单位：各单位材料按时到场率（`actual_arrival_date <= expected_arrival_date` 的比例）
- 按月：近 6 个月材料到场准时率趋势折线（按 `expected_arrival_date` 归月，准时率 = `actual_arrival_date <= expected_arrival_date` 的材料数 / 该月预计到场总数）
- 与现有阻碍类型分析并列，不单独占页

### 21.8 阻碍关联跳转

在 `GanttViewDialogs.tsx` 任务弹窗阻碍展开区，当 `obstacle_type === '材料'` 时，显示跳转链接：

```tsx

  <Link
    to={`/projects/${projectId}/materials?unit=${task.participant_unit_id}`}
    className="inline-flex items-center gap-1 text-xs text-blue-600 hover:text-blue-800"
  >
    ↗ 查看材料清单
  </Link>

```

跳转携带 `unit` 参数，材料管控页按该参建单位预筛选。

### 21.9 权限

- owner / editor：完整 CRUD
- viewer：只读
- company_admin：默认可读所有项目材料页与材料到场率分析，无需额外加入项目成员；不拥有写权限，材料修改仍以项目级 `owner / editor` 为准

### 21.10 实施边界

**不做**：
- 材料数量 / 库存管理
- 价格 / 成本跟踪
