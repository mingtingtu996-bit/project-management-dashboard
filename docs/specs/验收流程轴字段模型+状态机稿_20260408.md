# 验收流程轴字段模型 + 状态机稿_20260408

> 适用范围：`证照管理 / 验收时间轴`
>
> 文档定位：承接 `验收流程轴产品改造稿_20260408.md`，用于统一后续字段设计、数据边界、状态机与迁移方向。

## 1. 文档目标

本稿解决四类核心问题：

- 当前 `AcceptanceTimeline` 把“标准类型、项目实例、依赖关系、过程记录、展示坐标”混在同一对象里，语义不清。
- 现有 `type_id / type_name / acceptance_type / name` 口径并存，前后端容易错位。
- 状态过于粗糙，既不能支撑工程实际过程，也不能支撑后续预警、联动和统计。
- 关系图当前依赖 `position` 保存坐标，不适合升级为“阶段-时间双轴流程板”。

本稿结论：

- 验收流程轴后续至少拆成 `标准目录`、`项目实例`、`依赖关系`、`前置条件`、`过程记录` 五层对象。
- `受阻 / 临期 / 逾期` 不作为主状态，而作为叠加标记。
- 页面位置不再作为核心业务字段存储，展示位置改为由业务属性自动计算。

## 2. 总体业务模型

验收流程轴采用两层业务对象：

1. `标准目录层`
   用于定义“有哪些验收类型”。
2. `项目实例层`
   用于定义“当前项目里真实发生了哪些验收项，以及它们之间如何推进”。

围绕这两层，再补三类支撑对象：

1. `依赖关系`
2. `前置条件`
3. `过程记录`

建议的核心对象如下：

- `acceptance_catalog`
- `acceptance_plan`
- `acceptance_dependency`
- `acceptance_requirement`
- `acceptance_record`

如后续需要做资料模板与预警规则，可追加：

- `acceptance_requirement_template`
- `acceptance_warning_rule`

但这两项不作为首批重构前置。

## 3. 对象设计

## 3.1 `acceptance_catalog`

定位：标准验收类型目录。

一条目录记录只回答“这是什么验收类型”，不代表项目里的某一次真实验收。

建议字段：

- `id`
- `code`
  建议稳定英文码，如 `fire_acceptance`
- `name`
  标准名称，如 `消防验收`
- `short_name`
  短名，用于卡片与标签
- `category`
  类型归类
- `default_phase_code`
  默认所属阶段
- `default_scope_level`
  默认作用范围
- `is_system`
  是否系统内置
- `is_required_by_default`
  默认是否启用
- `sort_order`
- `description`
- `applicable_project_types`
  适用业态，如住宅、商业、综合体
- `enabled`

建议的 `category` 枚举：

- `special_acceptance`
  专项验收，如消防、人防、规划、电梯、防雷
- `unit_completion`
  单位工程类验收
- `completion_filing`
  备案归档类
- `delivery_closeout`
  交付收口类
- `custom`
  自定义项

建议的 `default_scope_level` 枚举：

- `project`
- `building`
- `unit`
- `discipline`

## 3.2 `acceptance_plan`

定位：项目中的一条真实验收实例。

一条实例记录回答的是“哪个项目、哪个楼栋、哪一次验收、计划何时完成、当前走到哪一步”。

建议字段：

- `id`
- `project_id`
- `catalog_id`
  对应 `acceptance_catalog.id`
- `acceptance_name`
  项目实例名，如 `1#楼消防验收`
- `phase_code`
  当前所在阶段
- `scope_level`
- `scope_ref_id`
  例如楼栋 ID、单位工程 ID
- `building_id`
  楼栋级项目建议冗余存储
- `sort_key`
  同阶段内排序键
- `planned_start_date`
- `planned_finish_date`
- `baseline_finish_date`
  基线日期，供偏差分析
- `actual_submit_date`
- `actual_accept_date`
- `actual_finish_date`
- `status`
  主状态
- `progress_percent`
  准备度或推进度，0-100
- `owner_org_name`
  内部牵头单位
- `owner_user_name`
  责任人
- `external_authority_name`
  外部主管或验收机构
- `summary`
  一句话说明
- `remarks`
- `is_custom`
- `is_required`
- `is_blocked`
  叠加标记，非主状态
- `block_reason_summary`
- `warning_level`
- `created_at`
- `updated_at`
- `created_by`

说明：

- `catalog_id` 负责识别“标准类型”。
- `acceptance_name` 负责表达项目实例。
- 不再使用模糊的 `name` 同时承担类型名和实例名。
- 不再把 `position.x / position.y` 作为业务核心字段。

## 3.3 `acceptance_dependency`

定位：验收实例之间的推进关系。

它不负责展示位置，只负责表达“谁依赖谁”。

建议字段：

- `id`
- `project_id`
- `predecessor_plan_id`
- `successor_plan_id`
- `dependency_kind`
- `branch_group_key`
  用于表达并行分流组
- `merge_group_key`
  用于表达并行合流组
- `sort_order`
- `remarks`
- `created_at`
- `created_by`

建议的 `dependency_kind`：

- `hard`
  硬前置，前项未满足不能推进到下一步
- `soft`
  软依赖，可以并行准备，但不能最终关闭

页面展示上不需要用虚线区分，但数据层仍建议保留。

## 3.4 `acceptance_requirement`

定位：某条验收实例的前置条件或资料要求。

这层是“能不能申报”的关键。

建议字段：

- `id`
- `acceptance_plan_id`
- `requirement_type`
- `source_type`
- `source_ref_id`
- `name`
- `description`
- `is_required`
- `is_satisfied`
- `satisfied_at`
- `satisfied_by`
- `due_date`
- `status_note`
- `created_at`
- `updated_at`

建议的 `requirement_type`：

- `document`
- `drawing`
- `permit`
- `task`
- `milestone`
- `issue_clearance`
- `custom`

建议的 `source_type`：

- `task`
- `drawing`
- `pre_milestone`
- `risk`
- `issue`
- `manual`

## 3.5 `acceptance_record`

定位：过程事件流水。

状态是当前结果，记录是过程事实。两者必须分开。

建议字段：

- `id`
- `acceptance_plan_id`
- `record_type`
- `record_date`
- `title`
- `description`
- `result`
- `attachments`
- `operator_name`
- `operator_org_name`
- `external_party_name`
- `next_action`
- `next_action_due_date`
- `created_at`

建议的 `record_type`：

- `plan_confirmed`
- `requirement_ready`
- `submitted`
- `appointment_confirmed`
- `inspection_started`
- `inspection_finished`
- `rectification_required`
- `rectification_completed`
- `resubmitted`
- `passed`
- `filed`
- `closed`
- `comment`

## 4. 阶段模型

验收流程轴建议采用固定阶段码，项目按需裁剪，不建议每个项目都自由命名阶段。

建议的 `phase_code`：

- `preparation`
  准备阶段
- `special_acceptance`
  专项验收
- `unit_completion`
  单位工程验收
- `filing_archive`
  备案归档
- `delivery_closeout`
  交付收口

说明：

- 一个实例始终只归属一个当前阶段。
- 阶段用于页面纵向泳道和统计分层。
- 阶段不是状态，不能拿来替代主状态。

## 5. 主状态机

## 5.1 设计原则

主状态只表达“当前业务推进到哪一步”，不混入风险、阻塞、预警语义。

因此建议主状态收为 8 个：

- `draft`
  待策划
- `preparing`
  准备中
- `ready_to_submit`
  待申报
- `submitted`
  已申报
- `inspecting`
  验收中
- `rectifying`
  整改中
- `passed`
  已通过
- `archived`
  已归档

建议中文名称：

- `待策划`
- `准备中`
- `待申报`
- `已申报`
- `验收中`
- `整改中`
- `已通过`
- `已归档`

## 5.2 状态迁移

建议的标准迁移路径：

```text
待策划 -> 准备中 -> 待申报 -> 已申报 -> 验收中 -> 已通过 -> 已归档
                             \-> 整改中 -> 已申报
                             \-> 整改中 -> 验收中
```

核心迁移规则：

- `待策划 -> 准备中`
  项目确认纳入计划或系统自动生成节点
- `准备中 -> 待申报`
  所有硬前置和必备资料满足
- `待申报 -> 已申报`
  发生正式申报行为
- `已申报 -> 验收中`
  预约确认或现场验收开始
- `验收中 -> 已通过`
  验收结论通过
- `验收中 -> 整改中`
  验收未通过，需要整改
- `整改中 -> 已申报`
  整改完成重新申报
- `整改中 -> 验收中`
  直接进入复验
- `已通过 -> 已归档`
  备案、归档或资料闭环完成

## 5.3 叠加标记

以下内容不进入主状态，而是作为叠加标记展示：

- `受阻`
- `临期`
- `逾期`
- `资料缺失`
- `前置未满足`

原因：

- 它们本质是风险或提示，不是阶段结果。
- 如果把这些信息混进主状态，后续统计和筛选都会变乱。

建议规则：

- 主状态保持单值
- 叠加标记允许多值

## 6. 自动派生字段

后续页面展示建议优先使用以下派生结果，而不是直接堆底层字段：

- `can_submit`
  是否具备申报条件
- `is_overdue`
  是否逾期
- `days_to_due`
  距计划完成剩余天数
- `requirement_ready_percent`
  前置准备完成率
- `downstream_block_count`
  当前节点阻塞了多少后续节点
- `upstream_unfinished_count`
  尚未满足的前置数量
- `display_badges`
  页面标签集合

## 7. 自定义验收项规则

自定义项允许存在，但必须受控。

新增自定义项时至少要求填写：

- `catalog category`
- `phase_code`
- `scope_level`
- `acceptance_name`
- `planned_finish_date`
- `owner_org_name`
- 是否硬前置

约束：

- 自定义项必须落入某个既有阶段
- 自定义项必须能明确前置
- 自定义项必须归属某个范围
- 不允许创建“无阶段、无责任、无计划日期”的漂浮节点

## 8. 与现有系统的联动建议

## 8.1 与任务管理

- 验收项可关联准备任务
- 准备任务完成率可反推 `progress_percent`
- 关键任务未完成可触发 `前置未满足`

## 8.2 与前期证照

- 前期证照一般不直接作为验收节点
- 但可作为某些验收项的 `permit` 类前置条件
- 联动方式应是“前置满足”，而不是把两块业务混成一个流程页

## 8.3 与施工图纸

- 图纸状态可作为 `drawing` 类前置条件
- 图纸未通过或未归档时，相关验收项不能转入 `待申报`

## 8.4 与预警 / 风险 / 问题

- 验收流程轴建议与现有系统联动
- 但不建议强耦合为同一对象

建议联动口径：

- 验收项临期、逾期、前置缺失，进入预警系统
- 验收项长期卡滞，可一键转问题
- 验收项存在重大阻塞时，可关联风险条目

结论：

- 验收流程轴是“事实与进度层”
- 预警 / 风险 / 问题是“提醒与处置层”

## 9. 与当前旧模型的迁移关系

针对现有 `client/src/types/acceptance.ts`，建议迁移如下：

- 当前 `AcceptanceType.id`
  -> 新 `acceptance_catalog.code`
- 当前 `AcceptancePlan.type_id`
  -> 新 `acceptance_plan.catalog_id`
- 当前 `AcceptancePlan.type_name`
  -> 新目录快照字段，可保留冗余展示
- 当前 `AcceptancePlan.name`
  -> 新 `acceptance_plan.acceptance_name`
- 当前 `AcceptancePlan.depends_on`
  -> 拆到 `acceptance_dependency`
- 当前 `AcceptancePlan.phase`
  -> 新 `acceptance_plan.phase_code`
- 当前 `AcceptancePlan.position`
  -> 仅保留为临时兼容字段，后续废弃
- 当前 `AcceptancePlan.status`
  -> 映射到新主状态机

建议的状态映射：

- `pending`
  -> `ready_to_submit` 或 `preparing`
  需结合前置满足度判断
- `in_progress`
  -> `submitted` 或 `inspecting`
  需结合是否已预约 / 已进场判断
- `passed`
  -> `passed`
- `failed`
  -> `rectifying`
- `needs_revision`
  -> `rectifying`

## 10. 首批实施边界

如果后续进入真正开发，首批建议只做以下收口：

1. 建立新状态机与新字段命名
2. 拆出依赖表，不再把关系塞在实例对象里
3. 拆出过程记录表
4. 页面改为根据 `phase_code + planned_finish_date + dependency` 自动布局
5. `position` 仅作为兼容数据临时保留，不再新增依赖

暂不强求首批完成：

- 复杂资料模板引擎
- 自动排程优化
- 全量历史数据标准化清洗
- 风险系统深度双向回写

## 11. 结论

后续验收流程轴要稳定，关键不是先重写页面，而是先把“目录、实例、依赖、条件、记录”五层结构拉开。

只要字段边界清楚：

- 页面可以从关系图切到双轴流程板
- 自定义项不会失控
- 预警、风险、问题、任务、图纸也都能自然联动

这也是后续页面原型、前后端接口和数据库迁移的共同基线。
