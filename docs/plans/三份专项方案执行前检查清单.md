# 三份专项方案执行前检查清单

> 用途：在执行《业务流程优化实施专项方案.md》《页面导航收口实施专项方案.md》《计划编制实施专项方案.md》任一步骤前，先检查本清单中适用于该步骤的阻断项。
> 原则：若命中任一 `blocked` 条件，则当前步骤不得直接进入实现，必须先补前置动作、迁移方案、目标文件或测试口径。
> 本清单不是第四份需求真值，只是三份专项方案的执行前 gate；需求真值仍以《业务流程优化清单.md》为准。

## 使用规则

1. 开始任一步骤前，先对照“适用步骤”筛出本步相关条目。
2. 若条目状态为 `blocked`，先补前置步骤或把缺口写回 `EXECUTION_PROGRESS.json`，不得直接实现。
3. 若条目状态为 `ready`，仍需把对应的目标文件、迁移方式、测试文件纳入当前步骤范围。
4. 同一问题若已在正文中写成显式动作，以正文为准；本清单只负责防止漏掉。

## 业务流程优化域

### C01 `task_delay_history -> delay_requests` 迁移方式已锁定

- **适用步骤**：`10.2b`
- **检查项**：当前步骤必须采用 `Rename + Extend`，不是新建一张平行表后双写。
- **必须满足**
  - 迁移文件显式写 `RENAME`
  - 存量数据回填 `status = approved`
  - `projectHealthService / taskSummaryService / schedulePredictor / task-summaries / tasks` 五处旧查询纳入目标文件
- **blocked 条件**
  - 只写“切到 delay_requests”但没写迁移路径
  - 仍允许代码继续硬查 `task_delay_history`

### C02 `warningService` / `warningChainService` / 调度壳职责已拆清

- **适用步骤**：`10.2c`
- **检查项**：检测、编排、调度必须分层，不允许 warningService 继续直接兼管升级链和通知写入。
- **必须满足**

  - 升级检查挂到现有 `conditionAlertJob`
  - `notifications` 只允许由编排层统一写入
- **blocked 条件**
  - 只补字段，不拆服务职责
  - 新增第二套平行 scheduler

### C03 `issues` 独立实体与 RiskManagement 两阶段迁移已锁定

- **适用步骤**：`10.2a`
- **检查项**：问题链不能继续用 `TaskObstacle` 冒充，前端切换必须两阶段推进。
- **必须满足**
  - 先建 `issues` 表与 `/api/issues`
  - 阶段一保留旧源并行验收
  - 阶段二正式切 `问题` tab 到 `/api/issues`
- **blocked 条件**
  - 直接把 `RiskManagement` 前端硬切到新接口，但后端未完成最小回填
  - 长期保留双真值不设收口态

### C04 `actual_start_date / actual_end_date` 自动写入只挂显式保存链

- **适用步骤**：`10.2b`
- **检查项**：自动写入规则必须绑到明确的保存链，不允许散落前端或隐式批处理。
- **必须满足**
  - 规则写在 `tasks.ts` 显式保存链
  - 测试覆盖开始、完成、回退三类情况
- **blocked 条件**
  - 只在文档里提到规则，没有目标文件和测试

### C05 `participant_unit_id` 成为主引用且 fallback 链明确

- **适用步骤**：`10.8b`
- **检查项**：不能让 `assignee_unit / responsible_unit / participant_units` 三源并存无主次。
- **必须满足**
  - 新增 `participant_unit_id` 可空外键
  - 写入优先级固定为 `participant_unit_id -> responsible_unit -> assignee_unit`
  - `assignee_unit` 标记 deprecated
- **blocked 条件**
  - 只加新字段，不写回退链

### C06 提醒中心新来源类型与导航元数据联动已锁定

- **适用步骤**：`10.2f`、`14.5`
- **检查项**：提醒中心必须支持 `planning / baseline / monthly_plan / closeout`，且 label 不再硬编码。
- **必须满足**
  - `source_entity_type` 明确支持四个新值
  - `Notifications.tsx` 的 `resolveTarget` 读 `navigation.ts`

  - 只加后端来源类型，不改前端目标解析
  - 继续在提醒中心写死 `Dashboard / Reports`

### C20 关账与执行重排 `3/5/7` 日梯度必须落到 scheduler + 通知链

- **适用步骤**：`15.10`
- **检查项**：前端超期态已写，后端梯度调度也必须写实。

  - 第 `5` 日升级到 `owner / admin`
  - 第 `7` 日解锁强制关账或自动结束重排，并生成摘要/通知
- **blocked 条件**
  - 只有前端红条或入口，没有调度任务和通知写入

## 开工前最小 Gate

在任一子步骤真正开始写代码前，至少确认以下 5 点：

1. 本步适用的 `blocked` 条件都已排除。
2. 本步目标文件已经覆盖所有受影响的旧查询、旧测试、旧路由或旧文案。
3. 本步依赖的前置步骤在三份专项方案和 `EXECUTION_PROGRESS.json` 中都已对齐。
4. 本步的验证命令能覆盖迁移、编译、测试或页面 contract，而不是只跑 `tsc`。
5. 若本步涉及 `review`，已约定 `gaps` 写回格式，且 `review` 失败时父步骤保持 `blocked`。
