## G1：图纸创建失败（迁移未执行）

### 蓝图要求
蓝图 4.12 图纸事项：支持图纸新建、版本管理、状态流转。

### 现状
- 前端入口已存在：`PreMilestones.tsx` 施工图纸 tab，`handleSaveDrawing` 函数已完整实现
- 后端路由已存在：`server/src/routes/construction-drawings.ts` POST 处理器，27 列 INSERT
- 数据库迁移脚本已存在：`server/migrations/032_add_construction_drawings_table.sql`
- **但用户确认真实环境创建失败**
- `v1真实环境校验结果_20260405.md` 中没有提及 `construction_drawings` 表

### 根因分析
1. **主因**：Migration 032 极可能从未在真实数据库执行，`construction_drawings` 表不存在
2. **次因**：`dbService.ts` 的 `executeSQL` INSERT 正则 `INSERT INTO (\w+)\s*\(([^)]+)\)\s*VALUES\s*\(([^)]+)\)` 默认单行匹配，而 `construction-drawings.ts` 的 INSERT SQL 跨 27 列多行，正则可能匹配失败

### 修复步骤
1. 在真实数据库手动执行 `server/migrations/032_add_construction_drawings_table.sql`
2. 验证表是否成功创建：`SELECT COUNT(*) FROM construction_drawings`
3. 在 PreMilestones 施工图纸 tab 新建一条图纸，观察返回错误信息
4. 若仍失败且错误来自 executeSQL，将 `construction-drawings.ts` 中的 INSERT SQL 改为单行写法：
   ```sql
   -- 改前（多行）
   INSERT INTO construction_drawings (
     col1, col2, ...
   ) VALUES (?, ?, ...)
   
   -- 改后（单行）
   INSERT INTO construction_drawings (col1, col2, ...) VALUES (?, ?, ...)
   ```
5. 重新测试，确认创建成功

### 涉及文件
- `server/migrations/032_add_construction_drawings_table.sql`（需手动在真实 DB 执行）
- `server/src/routes/construction-drawings.ts`（可能需要将 SQL 改单行）
- `server/src/services/dbService.ts`（仅作诊断，不轻易修改）

### 建议归入
v1收尾清单 Phase 1 之前，单独作为 **环境前置检查** 步骤（无需等待其他 Phase 完成）

## G3：问题三类来源链路缺失

### 蓝图要求
蓝图 5.7：问题来源须支持 3 种类型
- 阻碍上卷为问题（obstacle → issue）
- 风险转为问题（risk → issue）
- 人工直接登记（manual）

### 现状
- **人工直接登记** ✅：`RiskManagement.tsx` 已有手动创建问题入口
- **阻碍上卷为问题** ❌：无任何代码实现；`riskDetector.ts` 检测到阻碍未解决时只生成风险候选，不创建问题
- **风险转为问题** ❌：无任何代码实现；`risks` 表有 `type: risk/issue` 字段区分，但无转换链路

### 根因分析
整个问题来源链路的设计是：
- obstacle（阻碍）超时未解决 → escalate → 自动或手动变为 issue
- risk（风险）确认为真实问题 → 手动确认 → 转为 issue，记录 `source_type`

但当前代码中没有 escalation 机制，也没有 `source_type` 字段。

### 修复步骤

**Step A：数据库层**
1. 检查 `risks` 表是否有 `source_type` 字段（`manual / risk_converted / obstacle_escalated`）
2. 若无，新建迁移脚本添加该字段（可选 nullable，默认 `manual`）

**Step B：风险转问题（前端入口）**
1. 在 `RiskManagement.tsx` 的风险卡片操作菜单中增加"转为问题"按钮
2. 点击后调用 PUT `/api/risks/:id`，将 `type` 从 `risk` 改为 `issue`，同时设 `source_type: 'risk_converted'`
3. 转换后该条目从风险列表移入问题列表（tab 切换联动）

**Step C：阻碍上卷为问题（规则 + 后端）**
1. 在 `server/src/utils/riskDetector.ts` 或新建 `problemEscalation.ts` 中增加规则：
   - 当某阻碍（obstacle）的 `created_at` 超过阈值天数（建议 7 天）且状态未解决时
   - 自动在 `risks` 表插入一条 `type: 'issue'`、`source_type: 'obstacle_escalated'` 记录
   - 关联字段 `related_obstacle_id` 指向原阻碍
2. 该规则在任务更新时或定时触发（复用现有 riskDetector 的调用时机）

**Step D：来源标签展示**
1. 在 `RiskManagement.tsx` 问题卡片上展示来源 badge：
   - 🖊 手动登记
   - ⚡ 风险转入
   - ⬆ 阻碍上卷

### 涉及文件
- `client/src/pages/RiskManagement.tsx`（添加"转为问题"按钮 + 来源标签）
- `server/src/routes/risks.ts`（type 转换处理）
- `server/src/utils/riskDetector.ts`（阻碍超时检测逻辑）
- `server/migrations/` — 新增迁移：添加 `source_type` 字段

### 建议归入
v1收尾清单 Phase 2（自动化恢复）内，作为 2.3 的扩展项

## G6：EditProfileDialog 绕过 AuthContext 直接写 localStorage

### 蓝图要求
用户资料更新后全局状态同步更新。

### 现状
- `client/src/components/EditProfileDialog.tsx` L47：
  ```typescript
  localStorage.setItem('user', JSON.stringify(updatedUser))
  ```
- 直接写 localStorage，未调用 `AuthContext.updateProfile()`
- 结果：Header 等使用 `useAuth()` 的组件不会感知到资料更新，需刷新页面

### 修复步骤
1. 在 `EditProfileDialog.tsx` 中引入 `useAuth()`
2. 将 localStorage 直接写改为调用 `updateProfile()`：
   ```typescript
   // 修改前
   localStorage.setItem('user', JSON.stringify(updatedUser))
   
   // 修改后
   const { updateProfile } = useAuth()
   await updateProfile(updatedUser)
   ```
3. `AuthContext.updateProfile()` 已在 L188-214 中实现，直接调用即可

### 涉及文件
- `client/src/components/EditProfileDialog.tsx` L47（主改）
- `client/src/context/AuthContext.tsx`（已有 updateProfile，无需改）

### 建议归入
