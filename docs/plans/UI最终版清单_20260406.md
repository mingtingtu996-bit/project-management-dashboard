# UI 最终版清单
更新时间：2026-04-06

## 定位

本清单只处理展示层与入口组织，不涉及后端架构、数据库治理、发布包、自动化恢复。
基准参照按优先级依次为：
- `UI设计方案_V4_最终版.html`
- `UI设计更新说明_V4.1.md`
- `milestone-optimized.html`
- `risk-page-interactive.html`
- `recent-tasks-card-demo.html`
- `责任人单位完成情况卡片优化.html`
- `tasks-phase-ab-demo.html`

## 一、必须补齐的 UI 缺口

### 1. Dashboard

必须恢复的内容：
- `项目健康度` 的 4 维进度条和分楼栋 / 分项展开区
- `问题与风险` 的 2x2 语义色网格
- `Top-5` 紧急问题展开列表
- `任务状态` 的“左侧 4 行统计 + 右侧环形图”组合区
- `最近待完成任务` 的 `border-l-4` 语义色系统和底部统计栏
- `责任人完成情况` 的语义色排名样式
- `责任单位完成情况` 的语义色排名样式、单位类型图标和排名数字语义色
- `里程碑追踪摘要` 的补全信息

应补信息：
- 里程碑下一节点的责任人
- 责任单位
- 关联任务数
- 准时率
- 恢复更接近 `key-value` 的项目基本信息组织感，而不只是 4 格摘要卡
- 如果后端真实有字段，则补 `项目位置`、`总投资` 等基本信息

### 2. CompanyCockpit

必须恢复的内容：
- 按 `V4.1` 收口后的公司驾驶舱结构
- `3` 个核心指标卡
- `项目网格`
- `3` 个图表并排
- `里程碑汇总`
- 公司级 `健康度热力图`
- 公司级 `项目矩阵地图`

必须保留的内容：
- 项目排序
- 风险矩阵
- 交付倒计时
- 跨项目里程碑汇总
- 项目进入入口

必须去掉的内容：
- 内部过程型注释感文案
- 同一层级的重复统计说明
- 过长的单侧信息栏占位

### 3. GanttView / 任务列表

必须恢复的内容：
- `5` 个统计卡片行
- `BatchActionBar`
- 父级进度的 `↑` 标注
- 关键路径 `+0d` badge
- 任务列表页内的 `WBS` 结构表达
- 可折叠筛选面板与更完整的筛选工具栏结构
- `优先级`、`关键路径`、`重置 / 应用筛选` 这些筛选入口

必须明确的产品语义：
- 当前页面应被认知为 `任务管理 / 任务列表`
- `WBS` 是任务列表内部结构，不是独立主模块
- `GanttView` 只是历史文件名和历史路由名，不应误导为单独甘特图模块

### 4. Reports / 模块化分析承接

必须补齐的内容：
- 不保留独立主导航报表页
- 把分析入口拆回各业务模块
- 每个分析页都有自己的真实内容，而不是只剩承接说明页

对应承接关系：
- `项目进度总览分析` -> `Milestones`
- `风险与问题分析` -> `RiskManagement`
- `前期证照状态分析` -> `PreMilestones`
- `验收进度分析` -> `AcceptanceTimeline`
- `WBS完成度分析` -> `TaskManagement`

### 5. Notifications / 提醒中心

必须补齐的内容：
- `设置` 面板 tab
- 导入 / 导出入口承接
- 继续保留提醒聚合、`与我相关`、模块跳转

必须明确的产品语义：
- `Settings` 不再作为主导航入口
- 导入 / 导出能力应在 `Notifications` 页面内承接

## 二、需要按最新口径修正，而不是回退原样

### 1. Milestones

正确口径：
- 保留 `左列表 + 右详情` 的双列布局
- 增加层级左边框颜色
- 增加已完成项降权
- 增加 `即将到期(7天)` 统计卡
- 维持树形感，但不强行回退成真正树视图
- 旗帜标记面板属于“功能已存在、入口不同”，应记录但不作为缺失恢复项

不应做的事：
- 不要把页面强行改回老式树形主视图
- 不要丢掉当前双列布局的优点

### 2. RiskManagement

正确口径：
- 以 `risk-page-interactive.html` 为真实参照
- 趋势图应支持折叠 / 展开
- 风险项应区分自动 / 手动来源
- 统计卡副文案应改为上下文信息
- 增加骨架屏加载

不应做的事：
- 不要把它改回最早的静态 V4 形态
- 不要把“可交互增强”误当成“重做一个新页面”

### 3. AcceptanceTimeline

正确口径：
- 当前实现已经高于 V4 原始设计
- 保持现状，不作为回退对象
- 不要把它当作缺失项强行恢复旧版

### 4. TaskSummary / TaskManagement / WBS

正确口径：
- `TaskSummary` 继续作为 `任务管理` 的子页
- `WBS` 是 `任务列表` 内部结构，不是独立子页
- `任务总结` 不回退成独立平级模块

### 5. Header

正确口径：
- 不回退 Header 的增强结构
- 但应补强“当前项目名 + 状态 badge”的直观可见性
- `项目名 / 状态` 的可读性下降属于可优化项，不应被忽略

## 三、不建议再做的设计选择

这些不应再写入“缺失恢复清单”，因为它们属于设计选择或成熟风格取舍，不是功能丢失：

- 白色侧边栏回退到暗色侧边栏
- Emoji 图标回退到 Lucide 图标
- Header 显示工期范围文本
- 导航 hover 的 `translateX(4px)` 微动效

## 四、建议保留的成熟风格

以下是当前版本已经成立、也不应被误判为缺失的部分：

- 深色侧边栏
- Lucide 图标体系
- Header 的搜索 / 通知 / 用户菜单结构
- `TaskSummary` 的分析与沉淀能力
- `AcceptanceTimeline` 的力导向时间轴版本
- `WBS` 的条件 / 阻碍交互方式
- `Notifications` 的提醒聚合定位

## 五、最终优先级

### P0

- Dashboard 关键分析卡和语义色系统补齐
- CompanyCockpit 按 V4.1 收口
- GanttView 统计卡 / 批量操作 / `↑` / `+0d` 补齐
- Reports 分析能力拆回模块
- Notifications 设置面板承接导入导出

### P1

- Milestones 的 due soon 统计和层级颜色
- RiskManagement 的可折叠趋势图和来源标签
- Dashboard 的里程碑追踪摘要补全
- Dashboard 的项目基本信息区和任务状态区结构补全
- GanttView 的筛选工具栏结构补全
- Header 的项目名 / 状态可见性补强
- CompanyCockpit 的公司级图表补强

### P2

- 重新判断历史 `/reports`、`/gantt` 等兼容路由是否还需要继续暴露
- 清理会误导用户的过时文案
- 对仍在代码里、但无入口的分析组件做最终归位

## 六、结论

这份清单补完后，UI 层的判断标准应当是：
- 先看是否恢复了该有的高频入口
- 再看是否把分析能力放回对应模块
- 最后看是否避免了把设计选择误写成缺失项

只要满足这三条，就可以认为 UI 最终版清单成立。
