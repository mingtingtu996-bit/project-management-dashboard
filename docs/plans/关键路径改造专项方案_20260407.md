# 关键路径改造专项方案

> 更新时间：2026-04-07  
> 适用范围：`GanttView` 当前关键路径逻辑升级为“自动计算 + 手动插入 / 关注 + 完整链展示 + 图形化视图”  
> 目标：把当前“关键任务集合 + 红色 badge”升级为可解释、可维护、可跨页面复用的关键路径体系  
> 执行口径：本文已作为 `v1结束阶段最终执行方案_20260406.md` 的 `11.0-11.6` 正式专项接入执行体系

## 0. 使用规则

### 0.1 适用前提与时序

- 本专项已正式对应 `EXECUTION_PROGRESS.json` 的 `11.0-11.6`
- 自动执行默认顺序为：`9.x -> 10.x -> 11.x`
- 默认情况下，只有 `10.10` 完成后才启动 `11.0`
- 若人工明确调整优先级，至少要求 `9.9 completed`，且当前不存在 `10.x` 的 `in_progress`
- 本专项允许同时改前端、后端、迁移、测试和文档

### 0.2 范围边界

- 本专项默认不与 `9.x UI 收口` 混做
- 本专项默认不与 `10.x Drawings / issue 域` 混做

### 0.3 核心口径

- `自动排程关键路径` 是客观真值，来自依赖关系 + 工期 + CPM 计算
- `手动关注` 是管理标记，不自动改变客观关键路径
- `手动插链` 是管理层定义的展示链修正，必须指定插入锚点
- 若要让手动插入真正影响总工期，必须显式同步真实依赖关系；不能只改展示

### 0.4 绝对不要做的事

- 不要继续把“关键路径”只当成无序 ID 集合
- 不要把 `is_critical` 一把并进自动结果后就当作完整路径
- 不要把“手动关注”误写成“客观关键任务”
- 不要用力导图作为关键路径主视图
- 不要在没有锚点的情况下允许“手动插入关键路径”

### 0.5 图形化展示规则

- 主视图应是 `有方向的 DAG / 时间轴式链路图`
- `CriticalPathGraph.tsx` 的输入统一是 `CriticalPathSnapshot`
- 布局优先采用“手动分层布局算法”：按 `primaryChain` / `alternateChains` 与拓扑层级计算 `x/y`，`D3` 只负责渲染、连线、缩放和高亮
- 可复用现有 `D3` 缩放、拖拽、节点高亮经验
- 不复用 `ForceDirectedGraph` 的力导向布局作为关键路径主布局

## 2. 目标状态

### 2.1 最终要达到的产品结果

1. 系统能自动计算项目的排程关键路径
2. 用户能手动标记“关注任务”
3. 用户能把任务按锚点手动插入管理关键链
4. 页面能展示一条默认 `主关键链`
5. 若存在多条同等关键链，系统能展示 `备选关键链`
6. 用户能在 `GanttView` 内看到图形化关键路径弹窗
7. `GanttView / Dashboard / Reports / 风险提醒` 能复用同一份关键路径快照

### 2.2 建议采用的数据结构

```ts
type CriticalSource = 'auto' | 'manual_attention' | 'manual_insert' | 'hybrid'

interface CriticalPathEdge {
  id: string
  fromTaskId: string
  toTaskId: string
  source: 'dependency' | 'manual_link'
  isPrimary: boolean
}

interface CriticalTaskSnapshot {
  taskId: string
  title: string
  floatDays: number
  durationDays: number
  isAutoCritical: boolean
  isManualAttention: boolean
  isManualInserted: boolean
  chainIndex?: number
}

interface CriticalChainSnapshot {
  id: string
  source: CriticalSource
  taskIds: string[]
  totalDurationDays: number
  displayLabel: string
}

interface CriticalPathSnapshot {
  projectId: string
  autoTaskIds: string[]
  manualAttentionTaskIds: string[]
  manualInsertedTaskIds: string[]
  primaryChain: CriticalChainSnapshot | null
  alternateChains: CriticalChainSnapshot[]
  displayTaskIds: string[]
  edges: CriticalPathEdge[]
  tasks: CriticalTaskSnapshot[]
  projectDurationDays: number
}
```

### 2.3 手动规则

- `manual_attention`
  - 只做高亮和摘要提示
  - 不自动加入主链顺序
- `manual_insert`
  - 必须选择锚点
  - 允许的锚点形式：
  - 插到 `A` 后面
  - 插到 `B` 前面
  - 插到 `A-B` 之间
- 现有 `is_critical = true`
  - 第一阶段迁移为 `manual_attention`
  - 暂时保留兼容读，不再作为长期主模型

### 2.4 页面与交互清单

- **新增页面**
  - 不新增独立业务页，关键路径图形化查看统一作为 `GanttView` 内弹窗能力承载。
- **重点改造页面**
  - `GanttView`：保留任务主页面定位，但只增加关键路径摘要、筛选/高亮、延期影响提示、关键路径弹窗和手动干预入口。
  - `Dashboard`：增加关键路径摘要卡片，不承担编辑职责。
  - `Reports`：不新增独立关键路径分析页，只允许在项目进度总览分析或进度偏差分析中轻量引用关键路径相关信号。
- **新增功能**
  - 自动关键路径快照。
  - 手动关注任务。
  - 手动插链并指定锚点。
  - 主链/备选链图形化展示。
  - Dashboard / Reports / 风险提醒复用同一快照。
- **页面关键按钮**
  - `查看关键路径`
  - `设为关注任务`
  - `插到前面`
  - `插到后面`
  - `取消手动标记`
  - `仅关键路径`
- **当前页面改造**
  - `GanttView` 顶部卡头从“关键任务数量”升级为“关键路径摘要”。
  - `GanttView` 不新增关键链分析面板或独立侧栏；完整链路统一只在弹窗中查看，避免把关键链信息塞回任务行或主区。
  - `Dashboard` 只新增轻量摘要卡，不新增复杂图。
  - `Reports` 不新增独立关键路径分析入口或复杂解释区，只保留摘要级引用。
- **UI 统一要求**
  - 延续当前系统的 `PageHeader + Breadcrumb + shell-surface + slate/blue/white card`。
  - 关键路径弹窗可更强调链路感，但不能脱离现有系统页壳。
  - 图形化弹窗必须支持回跳并高亮任务列表，避免变成孤立展示层。

## 5. 与 `10.10` 主数据归并的关系

- 关键路径与 `前期证照 / 施工图纸 / 验收流程轴` 不完全相同，主真值不应下放为页面本地业务状态。
- 本专项的权威来源固定为服务端 `CriticalPathSnapshot` 与 overrides 模型，前端只负责消费、展示和局部交互。
- 并入 `10.10` 时，只需要把跨页面复用所需的关键路径摘要 / 读取缓存统一到 `criticalPathSummary` 命名状态分区。
- `GanttView / Dashboard / Reports / 风险提醒` 必须读取同一份摘要口径，但不得各自再维护不同的关键路径真值副本。
- 弹窗开关、图形布局展开态、当前选中节点等瞬时 UI 状态继续留在页面本地，不进入主数据归并范围。

这次关键路径专项的核心不是“再加几个红色标签”，而是把系统从“关键任务集合”升级成“自动排程关键路径 + 管理关键链覆盖层 + 图形化视图”的完整体系；只要按 `11.1 -> 11.4` 这条主链推进，就能先把核心能力闭环，再逐步把 Dashboard / Reports / 风险提醒接上。
