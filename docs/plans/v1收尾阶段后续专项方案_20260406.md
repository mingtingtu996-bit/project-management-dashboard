### 0.6 进度回写规则

进度文件继续使用 `steps.<stepId>` 结构，后续专项新增如下子键：

```json
{
  "steps": {
    "8.1": { "status": "in_progress", "note": "8.1 已从父步骤 blocked 切换为子步骤执行" },
    "8.1.0": { "status": "pending" },
    "8.1.1": { "status": "pending" },
    "8.1.2": { "status": "pending" },
    "8.1.3": { "status": "pending" },
    "8.1.4a": { "status": "pending" },
    "8.1.4b": { "status": "pending" },
    "8.1.4c": { "status": "pending" },
    "8.2b.0": { "status": "pending" },
    "8.4.0": { "status": "pending" }
  }
}
```

回写原则：

- 父步骤只表示专项总状态
- 子步骤表示真实执行进度
- 子步骤完成说明里必须包含“改动对象 + 验证结果”
- 子步骤 `blocked` 时，说明里必须写明阻断文件或阻断原因

把 `8.1` “切出子步骤” 的意思就是：

1. 保留原有 `8.1`
2. 把 `8.1.status` 从 `blocked` 改成 `in_progress`
3. 在同一个 `steps` 对象里新增：
   - `8.1.0`
   - `8.1.1`
   - `8.1.2`
   - `8.1.3`
   - `8.1.4a`
   - `8.1.4b`
   - `8.1.4c`
4. 这些子步骤初始都写成 `pending`

“从 `8.1.0` 和 `8.1.1` 开始” 的意思是：

1. 先只做进度切换和专项基线，这一步叫 `8.1.0`
2. `8.1.0` 完成后，把它改成 `completed`
3. 再进入第一个真正改代码的页面步骤 `8.1.1`

也就是说，不是一次性同时开 7 个步骤，而是先把 `8.1` 变成“可续跑专项”，再按 `8.1.0 -> 8.1.1 -> 8.1.2 -> 8.1.3 -> 8.1.4a -> 8.1.4b -> 8.1.4c` 顺序推进。
