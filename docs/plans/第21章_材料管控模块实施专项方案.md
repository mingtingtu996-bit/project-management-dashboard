# 第21章 材料管控模块实施专项方案

## 一、专项判断

- 本章可以执行，但不建议直接无拆解开工。
- 原因：它同时涉及新表、后端 CRUD、新页面、批量录入、每日提醒、Reports 集成、阻碍跳转与权限收口，属于完整新模块。
- 本方案作为第 21 章唯一实施口径，执行时以 [v2功能增强需求文档_20260414.md](./v2功能增强需求文档_20260414.md) 第 21 章为产品真值，以本方案为工程拆解真值。

## 二、先定口径

1. `project_materials` 是本章唯一材料真值表，不额外建立“模板表”。
2. 材料状态不落库，`待定样 / 待到场 / 逾期未到 / 本周到场 / 待送检 / 已完成` 均由字段推导。
3. 前端页面状态标签由纯函数计算；后端提醒与报表不依赖状态字符串存储，按同一字段规则计算。
4. 模板预填先做“代码内置模板常量”，不在本章扩成模板后台。
5. 批量录入先做表格式逐行录入，不做 Excel 导入。
6. Reports 的“按月准时率”按 `expected_arrival_date` 所在月份归档，计算公式固定为：`actual_arrival_date <= expected_arrival_date` 的材料数 / 该月预计到场总数。
7. 同一参建单位同一天仅发送一条聚合提醒；对尚未逾期、仅进入提醒窗口的材料项，同一材料项同一自然周内只发一次提醒；逾期未到场材料仍按日提醒。
8. 参建单位删除后依赖数据库 `ON DELETE SET NULL`，材料项保留并在前端落到“无归属单位”组。
9. 权限固定为：
   - 项目级 `owner / editor`：完整 CRUD
   - 项目级 `viewer`：只读
   - 公司级 `company_admin`：默认可读所有项目材料页与材料到场率分析，无需额外加入项目成员；不拥有写权限

## 三、专项步骤

### 21.M21a 数据底座与接口合同

- **前置步骤**：无
- **目标文件**
  - `server/migrations/<next>_create_project_materials.sql`
  - `server/src/routes/project-materials.ts`
  - `server/src/index.ts`
  - `server/src/types/db.ts`
  - `server/src/__tests__/project-materials-routes.test.ts`
- **开始前定位命令**
  ```powershell
  Get-ChildItem -LiteralPath server/migrations -File | Where-Object { $_.Name -match '^(\d{3})_' } | Sort-Object Name | Select-Object -ExpandProperty Name | Select-Object -Last 12
  ```
- **具体动作**
  1. 先确认可用迁移编号后再建表，不直接假设下一个编号。
  2. 建立 `project_materials` 表，字段严格按第 21.2 节落地。
  3. `participant_unit_id` 使用 `ON DELETE SET NULL`。
  4. 建立索引：`project_id`、`participant_unit_id`、`expected_arrival_date`、`specialty_type`。
  5. 新增项目级材料 CRUD 路由：
     - `GET /api/projects/:projectId/materials`
     - `POST /api/projects/:projectId/materials`
     - `PATCH /api/projects/:projectId/materials/:materialId`
     - `DELETE /api/projects/:projectId/materials/:materialId`
  6. `GET` 接口返回列表完整字段，并通过 `LEFT JOIN participant_units` 带出 `participant_unit_name`；无归属单位时返回 `null`。
  7. 权限在路由层收口：
     - `owner / editor` 可写
     - `viewer` 可读
     - `company_admin` 可读所有项目材料页数据
- **不要做的事**
  - 不新增价格、库存、数量字段
  - 不建立模板后台表
  - 不在本步顺手做提醒或 Reports
- **验证方式**
  ```powershell
  npm run build --workspace=server
  npx vitest run --root server src/__tests__/project-materials-routes.test.ts
  ```
- **通过标准**
  - 材料表与 CRUD 路由可用
  - 删除参建单位后材料保留且 `participant_unit_id` 自动置空
  - `company_admin` 可读，非项目成员 `regular` 不可越权读取
- **进度回写示例**
  - `21.M21a completed: project_materials 表与 CRUD 路由已建立，权限与 ON DELETE SET NULL 已收口`

### 21.M21d 到场提醒调度与通知链

- **前置步骤**：`21.M21a completed`
- **目标文件**
  - `server/src/services/materialArrivalReminderService.ts`
  - `server/src/scheduler.ts`
  - `server/src/routes/jobs.ts`
  - `server/src/__tests__/materialArrivalReminderService.test.ts`
- **具体动作**
  1. 新增每日材料提醒扫描服务。
  2. 规则按第 21.6 节落地：
     - 只匹配该参建单位下最早将开始且 `status in ('todo', 'in_progress')` 的任务
     - 无计划开始日期的任务不参与缓冲天数计算
     - `缓冲天数 <= 7` 触发提醒
     - 无关联任务时退化为到场前 5 天提醒
     - `actual_arrival_date` 一旦填写即停止该材料提醒
     - 逾期未到场时每日提醒
  3. 同一参建单位的多条材料在同一天聚合成一条通知；对尚未逾期、仅进入提醒窗口的材料项，同一材料项同一自然周内只发一次提醒；逾期未到场材料仍按日提醒。
  4. 接收人固定为项目 `owner`。
- **不要做的事**
  - 不自动生成阻碍
  - 不扩成多级升级链
- **验证方式**
  ```powershell
  npm run build --workspace=server
  npx vitest run --root server src/__tests__/materialArrivalReminderService.test.ts
  ```
- **通过标准**
  - 每日扫描可跑通
  - 有关联任务与无关联任务两种路径正确
  - 聚合提醒与逾期每日提醒正确
- **进度回写示例**
  - `21.M21d completed: 材料到场提醒服务与每日调度已接入，支持按单位聚合通知`

- **具体动作**
  1. 为 Reports 增加“材料到场率”区块数据：
     - 按参建单位的按时到场率
     - 近 6 个月按月趋势（按 `expected_arrival_date` 归月，准时率 = `actual_arrival_date <= expected_arrival_date` 的材料数 / 该月预计到场总数）

## 四、全章收口标准

1. `project_materials` 真值表、CRUD、权限、路由全部落地。
2. 材料页能完成新增、批量录入、筛选、状态展示、行内更新、删除。
3. 每日到场提醒已进入调度器。
4. Reports 已显示材料到场率分析。
5. GanttView 材料型阻碍已能跳转并按单位预筛选。
6. `company_admin` 无需加入项目成员即可读取所有项目材料页与材料到场率分析，但不拥有写权限。
7. 前后端类型检查、构建、定向测试全部通过。

## 五、本章不要做的事

1. 不做采购、库存、成本、数量管理。
2. 不做材料审批流。
3. 不做自动生成阻碍。
4. 不做外部供应商门户。
5. 不做模板后台治理系统。
