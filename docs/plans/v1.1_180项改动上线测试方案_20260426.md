## 2. 测试环境与数据

### 2.1 环境要求

- 使用 staging 环境，不直连生产库。
- 前后端指向同一套 staging API / DB。
- 部署前确认环境状态：

```bash
npm run env:status
```

### 2.2 测试账号

必须准备以下账号，并记录账号 ID、角色、所属项目权限：

| 账号 | 角色 | 用途 |
|------|------|------|
| owner_user | owner | 全量流程、确认、强制处理、权限上限 |
| admin_user | admin | 管理动作、强制关账、审批类操作 |
| editor_user | editor | 常规编辑、计划编制、任务维护 |
| viewer_user | viewer | 只读权限、越权拦截 |

### 2.3 测试项目数据

至少准备 1 个全量项目，必须包含：

- 任务树：不少于 50 条任务，含父子层级、依赖关系、负责人、责任单位、范围维度。
- 关键路径：至少 1 条主链、1 条备选链、2 条依赖线、1 个手动关注任务、1 个手动插链候选。
- 计划编制：至少 1 个确认基线、1 个草稿基线、1 个月度计划草稿、1 个待关账月份。
- 月末关账：至少 5 条待处理项，覆盖建议关闭、建议滚入、需人工判断、强制处理。
- 风险问题：至少 3 条预警、3 条风险、3 条问题，含 pending_manual_close。
- 专项数据：前期证照、施工图纸、验收流程轴、材料各至少 5 条记录。
- 里程碑：至少 10 个里程碑，覆盖基态、映射待确认、已合并、偏差过大、数据不完整。
- 通知：业务预警、系统异常、流程催办各至少 2 条。

### 4.3 状态机专项

月度计划：

- 草稿生成 -> 编辑 -> 保存 -> 确认。
- 有阻断条件时确认按钮 disabled。
- 绕过前端直接调确认接口时，后端返回阻断错误。
- 空草稿进入“本月无承诺”确认流程。
- 确认月度计划后，使用 confirmed 态计划调用 batch-scope / batch-shift-dates / batch-target-progress / batch-notes API，后端必须返回 409 + `PLAN_NOT_DRAFT`。

月末关账：

- 普通关账走 `/close`。
- 强制关账走 `/force-close`。
- 一键采纳只处理后端认定的可采纳条目。
- 完成确认弹窗 4 项计数正确。

风险与问题：

- 预警升级风险。
- 阻碍升级问题。
- pending_manual_close 的“确认关闭”和“保持处理中”走专用端点。
- 关闭后链路状态、通知、列表筛选同步更新。

验收流程：

- draft -> preparing -> ready_to_submit -> submitted -> inspecting -> passed/rectifying -> archived。
- 非法跳转被前后端共同拦截。
- 状态变更后刷新页面保持一致。

并发保护：

1. 使用 owner_user 与 admin_user 同时打开同一基线草稿。
2. 两个账号同时点击确认基线，预期一个成功，另一个返回 409 + `LOCK_HELD_BY_OTHER_USER`。
3. 使用 owner_user 与 admin_user 同时打开同一月度计划草稿。
4. 两个账号同时点击确认月度计划，预期一个成功，另一个返回 409 + `LOCK_HELD_BY_OTHER_USER`。
5. 失败方刷新后必须看到最新确认状态，不允许继续基于旧草稿提交。

### 4.7 关键路径数据源一致性专项

覆盖条目：`CP9 / CP10 / CP11 / CP14 / CP15`

测试数据：

- 至少 1 条自动 CPM 主链。
- 至少 1 个 `manual_insert` 覆盖。
- 至少 1 个 `manual_attention` 关注任务。
- 至少 1 个旧 `tasks.is_critical = true` 但不在最新 snapshot `displayTaskIds` 中的任务。

测试步骤：

1. 生成或刷新关键路径 snapshot。
2. 手动插链一条任务，确认 `displayTaskIds` 顺序变化，`projectDurationDays` 不因该任务被手动插入而改变。
3. 手动关注一条非关键路径任务，确认该任务进入 `watchedTaskIds`，但不进入 `displayTaskIds`。
4. 制造一个关键路径任务延期，确认 `warningService` 只对 `displayTaskIds` 中的任务触发关键路径延期通知。
5. 制造一个仅 `manual_attention` 的任务延期，确认不触发“关键路径延期”通知；如有通知，应为独立关注任务提醒。
6. 运行数据质量、计划完整性、治理快照、周报、责任洞察、延期申请相关逻辑，确认输出以最新 `criticalPathSnapshot.displayTaskIds` 为准。
7. 检查以下后端消费者不再以 `tasks.is_critical` 作为真值源：`dataQualityService`、`planningIntegrityService`、`planningGovernanceService`、`projectExecutionSummaryService`、`weeklyDigestService`、`responsibilityInsightService`、`baselineGovernanceService`、`delayRequests`、`dbService`、`warningService`。
8. 若保留 `tasks.is_critical` 作为兼容缓存，确认 snapshot 更新后的同步机制能更新缓存，且消费者优先读 snapshot。

通过标准：

- `displayTaskIds` 表示客观关键路径和手动插链展示链，不包含纯关注任务。
- `watchedTaskIds` 表示管理关注任务。
- 下游通知、治理、统计、周报、数据质量输出与 snapshot 一致。
- `projectDurationDays` 不被手动插链或关注污染。
