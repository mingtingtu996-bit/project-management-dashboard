# 登录权限系统实施专项方案

编制日期：2026-04-15  
需求来源：`v2功能增强需求文档_20260414.md` §三 + §五

## 1. 方案目标

本专项用于把“登录身份、公司层级角色、项目层级角色、团队管理入口、密码重置”整合成一套清晰可执行的权限体系。

目标是：

1. 保持现有项目级三角色体系稳定。
2. 补齐公司层级全局角色。
3. 把团队管理从“孤立页面”收口到 Header 用户菜单。
4. 补齐 owner 转让前端入口和密码重置能力。
5. 对 viewer 做前后端双重权限收口。

## 2. 当前现状与缺口

### 2.1 已有底座

- 项目成员体系已经存在，后端已支持 `owner / editor / viewer`。
- 后端已有 `POST /api/members/:projectId/transfer-owner`，owner 转让服务端逻辑已具备。
- 登录采用用户名 + 密码 + JWT，本地部署链路已成熟。
- Header 已有用户菜单、修改密码等入口，可复用交互壳子。

### 2.2 关键缺口

- `users` 表当前没有 `global_role`，公司驾驶舱无法按公司级角色过滤。
- `TeamMembers.tsx` 仍是独立全页面，入口不自然，且没有融入 Header 菜单。
- owner 转让虽有后端接口，但前端未形成明确操作入口。
- 没有管理员重置他人密码的能力。
- viewer 的后端校验与前端隐藏未完全收口。
- Header 用户菜单尚未展示“当前全局角色 + 当前项目角色”。

## 3. 目标权限模型

## 3.1 两层角色

### 3.1.1 公司层角色

新增 `users.global_role`：

- `company_admin`
- `regular`

职责：

- `company_admin`：查看所有项目汇总、管理用户授权、重置密码。
- `regular`：仅查看自己参与的项目汇总。

### 3.1.2 项目层角色

维持现有 `project_members.permission_level`：

- `owner`
- `editor`
- `viewer`

职责不变：

- `owner`：全部权限、审批、强制操作、项目删除/转让。
- `editor`：日常编辑与提交流程。
- `viewer`：只读查看。

## 3.2 登录体验总览

```text
登录成功
  ├── 先确定 global_role
  │   ├── company_admin -> 公司驾驶舱看全部项目
  │   └── regular -> 公司驾驶舱只看自己参与的项目
  └── 进入具体项目后，再按 project_members.permission_level 控制按钮和接口
```

## 4. 实施设计

## 4.1 数据库与鉴权改造

### 4.1.1 users 表

- 增加 `global_role TEXT NOT NULL DEFAULT 'regular'`。
- 首个注册用户自动设为 `company_admin`。
- 后续用户默认 `regular`，由 `company_admin` 进行提升。

### 4.1.2 JWT 与会话信息

JWT 原有 `role` 字段可保留。

建议补充返回体中的 `global_role`，供前端直接消费：

- 登录响应返回 `globalRole`。
- `/me` 或当前用户信息接口同步返回 `globalRole`。

这样可以避免前端每次进入公司驾驶舱时再额外拉取角色详情。

## 4.2 后端能力改造

### 4.2.1 公司驾驶舱过滤

所有公司级项目汇总接口统一按 `global_role` 过滤：

- `company_admin`：不加项目成员限制。
- `regular`：只查询本人在 `project_members` 中存在记录的项目。

### 4.2.2 密码重置

新增接口：

- `POST /api/auth/reset-password`

约束：

- 仅 `company_admin` 可调用。
- 参数：目标 `username`。
- 返回：系统生成的临时密码。
- 被重置用户使用临时密码登录后，走现有“修改密码”流程。

不做邮箱、短信、自助找回。

### 4.2.3 Viewer 权限收口

统一审计所有写操作接口，确保：

- `requireProjectEditor` 仅放行 `owner / editor`。
- `requireProjectOwner` 仅放行 `owner`。
- `viewer` 绝不允许绕过前端直接写入。

需重点复查：

- 任务、风险、图纸、验收、里程碑、审批、导出导入、成员管理。

## 4.3 前端交互改造

## 4.3.1 Header 用户菜单收口

目标结构：

- 我的账户
- 当前角色说明
- 团队管理
- 修改密码
- 退出登录

其中“团队管理”仅在项目上下文中显示，并受权限控制：

- `owner`：完整可见并可操作。
- `editor / viewer`：不显示。

### 4.3.2 团队管理 Drawer

不再保留独立全页面作为主入口，改为 Header 菜单唤起 Drawer / Sheet。

Drawer 承载：

- 成员列表。
- 添加成员。
- 调整权限。
- 移除成员。
- 转让负责人。
- 重置密码。
- 邀请码管理。

建议组件拆分：

- `TeamManagementDrawer`
- `MemberListTable`
- `MemberRoleEditor`
- `TransferOwnerDialog`
- `ResetPasswordDialog`
- `InvitationCodePanel`

### 4.3.3 角色展示

Header 用户菜单中增加两行角色信息：

- 全局角色：`company_admin / regular`
- 当前项目角色：`owner / editor / viewer`

目的：

- 让用户明确知道自己“现在能做什么”。
- 降低“为什么我看不到按钮”的理解成本。

### 4.3.4 登录弹窗补充文案

登录弹窗继续保留 Header 弹窗形态。

补充文案：

- “忘记密码？请联系项目负责人重置”

前提是管理员重置能力同步上线。

## 4.4 现有页面与路由调整

- `TeamMembers.tsx` 不再作为主入口承载团队管理。
- 原路由可暂时保留一版兼容跳转，避免旧链接失效。
- 后续版本再决定是否彻底移除该页面。

## 4.5 数据与文案兼容

需要同步收口的旧命名：

- 前端本地类型若仍使用 `admin`，要统一替换为 `owner`，避免项目级角色语义混乱。
- 所有成员管理弹窗、菜单、提示文案统一使用：
  - 项目负责人：`owner`
  - 编辑成员：`editor`
  - 只读成员：`viewer`

## 5. 分阶段执行

## 5.1 第一阶段：权限底座

- 增加 `global_role`。
- 登录态回传 `global_role`。
- 公司驾驶舱接口按全局角色过滤。
- Viewer 后端写权限复查。

交付结果：

- 公司级与项目级权限边界清晰。

## 5.2 第二阶段：团队管理收口

- Header 菜单增加团队管理入口。
- 团队管理 Drawer 上线。
- 支持增删改成员与邀请码管理。
- 前端补 owner 转让入口。

交付结果：

- 项目成员管理入口从“独立页”收口到用户菜单。

## 5.3 第三阶段：密码与体验补齐

- 管理员重置密码接口与弹窗。
- 登录弹窗忘记密码提示。
- Header 菜单展示双层角色信息。

交付结果：

- 登录权限链路从“能用”升级为“可解释、可运维”。

## 7. 验收标准

1. `users` 表存在 `global_role`，并可区分 `company_admin / regular`。
2. 首个注册用户自动成为 `company_admin`。
3. 公司驾驶舱数据按 `global_role` 正确过滤。
4. owner 可在前端对成员执行“转让负责人”。
5. 团队管理从 Header 用户菜单唤起 Drawer，不再依赖独立导航入口。
6. owner 可调整成员权限、移除成员、管理邀请码。
7. `company_admin` 可重置用户密码，并返回临时密码。
8. Header 菜单能展示当前全局角色与当前项目角色。
9. viewer 无法通过前端或直接请求接口执行写操作。

## 8. 风险与注意事项

- `company_admin` 与项目 `owner` 不是同一概念，界面文案必须区分清楚。
- 密码重置返回临时密码属于敏感操作，日志里不能明文持久化。
- 团队管理 Drawer 替换独立页后，需留一段兼容期，避免旧入口失效。
- Viewer 权限收口必须以后端为准，不能只做前端按钮隐藏。

## 9. 建议优先级

- P1：`global_role`、公司驾驶舱过滤、owner 转让前端、密码重置。
- P1：团队管理收口到 Header。
- P2：viewer 权限全面审计与 Header 角色展示。
