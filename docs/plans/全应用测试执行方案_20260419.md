### 3.6 关键路径域

- 查看关键路径快照：`primaryChain + alternateChains`。
- 手动覆盖：`manual_attention`、`manual_insert`。
- 延期审批后 CPM 重算，关键路径更新。
- 关键路径任务停滞 7 天触发 warning。
- `primaryChain` 选链规则：总工期最长优先；并列时一级里程碑更多优先；再并列时结束时间更晚优先。
- `manual_insert` 必须指定锚点（插到 A 后面 / B 前面 / A-B 之间），无锚点请求被后端拒绝。

- 仅关键路径筛选基于 `displayTaskIds` 而非 `is_critical` 字段，筛选结果与快照一致。
- 关键路径弹窗节点颜色：自动关键 `border-red-500`、手动关注 `border-amber-400 border-dashed`、手动插链 `border-orange-400`。

## 4. 第三轮：边界与异常

### 4.1 升级链边界

- 删除 warning 后，risk 标记 `is_deleted` 但仍可访问。
- 删除 risk 后，issue 标记 `is_deleted` 但仍可访问。
- `chain_id` 串联完整，可从 issue 追溯到原始 warning。
- 升级后原记录状态变为 `escalated`。
- 降级场景：risk 关闭后 warning 不自动恢复。

### 4.2 延期审批边界

- 同一任务不能有多个 `pending` 延期申请。
- 延期审批超时第 3 天提醒、第 5 天升级。
- 延期审批通过后，任务时间更新 + CPM 重算。
- 延期审批驳回后，任务时间不变。
- 关键路径任务延期，通知级别升级。

### 4.3 条件 / 阻碍边界

- 首次进度 `0 -> >0` 豁免条件拦截。
- 后续进度推进必须满足条件。
- 任务完成后自动满足依赖任务的条件。

- 障碍超时 severity 升级。

### 6.4 发布阻塞项：基础设施与边界

#### 6.4.1 browserStorage / 本地缓存安全

- `isQuotaExceededError` 可识别 `QuotaExceededError` 与 `NS_ERROR_DOM_QUOTA_REACHED`。
- `safeStorageSet` 在配额写满时返回 `false`，并派发存储告警事件。
- `bindStorageWarningToToast` 只绑定一次，重复调用不产生重复监听。
- `safeStorageGet` 底层 `getItem` 抛异常时返回 `null`，不导致页面崩溃。
- `safeJsonParse` 对非 JSON、空字符串、`null`、`undefined` 均返回 fallback。
- `listAppStorageKeys` 仅返回应用前缀 key，不泄露无关存储数据。
- `clearAppStorage` 在确认文本不匹配时拒绝执行清理。
- 所有存储工具函数在 SSR 或 `window === undefined` 场景下安全降级。

#### 6.4.2 storageService / useDataSync / useConflictDetection

- 三种存储模式 `LOCAL / SYNC / READONLY` 下读写行为正确。
- 同步队列在 `create / update / delete` 后正确入队，初始状态为 `pending`。
- `processSyncItem` 成功后状态变为 `completed`，失败时 `retries + 1`。
- 达到最大重试次数后状态变为 `failed`，不再无限重试。
- 网络状态 `ONLINE / SLOW / OFFLINE` 检测与自动切换正确。
- 离线恢复在线后自动尝试清空待同步队列。
- 同步状态监听注册、注销正确，回调携带在线状态、最后同步时间、待同步数量。
- `applyMergedData / forceUpdate / applyServerData` 三种冲突处理路径写入结果正确。
- `getPendingSyncCount` 返回 `pending + failed` 总数。
- `useDataSync` 在在线且标签可见时每 `30s` 自动同步。
- 离线或标签不可见时不触发同步；标签恢复可见后立即补同步一次。
- `isSyncing` 生效，不发生重复并发同步。
- 队列中单项同步失败时，不阻塞其他待同步项继续处理。
- 开启冲突检测时会联动 `detectConflicts`，并按 `merge / keepLocal / keepServer` 正确落库。
- 冲突检测对相同版本返回空结果，忽略 `id / version / created_at / updated_at / user_id / project_id` 等系统字段。
- 六类实体 `project / task / risk / milestone / member / invitation` 均可做冲突检测。

#### 6.4.3 useProjectInit / 项目初始化

- 项目初始化数据通过并发加载，不因单一路失败阻塞其他成功项。
- 单一路加载失败时，该路数据以空数组或本地降级结果回填，其余数据正常写入共享 store。
- 项目 `404` 时进入 `not_found` 状态；项目接口失败但本地有缓存时可回退到缓存数据。
- `hydratedProjectId` 防止同一项目重复初始化。
- 快速切换项目时旧请求会被中止，不出现项目数据串项。
- 里程碑可从任务数据正确推导构建。
- 快照接口失败时可从任务、条件、障碍等本地数据降级构建。
- 任务、条件、障碍、预警、问题、延期请求等字段标准化逻辑正确。
- `SharedSliceStatus` 在加载前为 `loading=true`，完成后正确写入 `error` 或 `null`。
- `reloadToken` 或等效重试机制可触发重新加载。

#### 6.4.4 边界异常 / 极端场景

- 本地存储接近浏览器配额上限时，写入失败有明确提示且应用不崩溃。
- 无痕或隐私模式下本地存储抛异常时，全应用安全降级。
- 手动篡改 localStorage 为非 JSON 字符串时，应用以 fallback 值继续运行。
- 超长任务标题、Unicode、Emoji、跨年与闰年日期、空日期、`progress=0 / 100 / >100` 等边界值正确处理。
- 空项目与大项目都能渲染：空项目显示空态，大项目保持可操作且无严重卡顿。
- 双击或快速重复提交时，按钮禁用或服务端幂等保护有效。
- 双标签页同时编辑同一记录时，后保存一方触发版本冲突提示。
- 页面刷新后能从 URL 恢复项目上下文；浏览器前进后退保持路由与状态一致。
- 深度链接如 `/projects/:id/gantt?view=timeline` 可直接加载。
- 无效项目 ID 返回 `not_found` 状态，而不是页面崩溃。
- 超时或非 JSON API 响应能被优雅解析并反馈给用户。
- `offline -> online` 恢复后，同步队列与 WebSocket 会自动恢复。
- `sessionStorage` 仅做当前标签页内持久化，关闭标签页后正确清除。

### 8.1 如何判断“测试到位”

不是看“测了很多”，而是看是否形成完整证据链。满足以下条件，才可判定本轮测试基本到位：

3. 回归优先顺序：`B线执行主链 -> C线计划治理 -> A线壳层协作 -> D线专项业务 -> E线基础设施复核`。
4. 同一时间只允许一条执行线修改共享层，其余执行线只记录问题，不立即改。

### 9.6.1 第4章模型的抢占式修复规则

- `A4/B4/C4/D4/E4` 默认执行第 `4` 章模块内一致性。
- 主控一旦派单，这些模型必须立即暂停当前第 `4` 章测试，优先处理分配给自己的缺陷。
- 独占模块问题由对应 `A4/B4/C4/D4` 修复。
- 共享层问题只允许 `E4` 修复。
- 修复完成后，修复模型只把状态改成 `待回归` 或 `已修复待回归`，不直接宣布通过。
- 原发现问题的第 `3` 章模型负责回归，并新增 `回归通过 / 回归失败` 记录。

### 9.7 边测边修执行规则

如果你希望测试时发现问题就直接修，可以这样执行，而且不会把台账弄乱：

#### 9.7.1 哪些问题可以直接修

- `A2/B2/C2/D2` 可以直接修本线独占模块内的问题，例如单页面 UI、单页交互、页面内局部状态、单个接口展示错误。
- `E2` 可以直接修本线独占的基础设施与 job / Jobs API 问题。
- 明显不影响其他执行线的局部逻辑问题可以直接修，例如文案、空态、按钮可见性、筛选条件、局部弹窗交互。

#### 9.7.2 哪些问题不能直接抢修

- 涉及共享层的改动不能由所有执行线同时改。
- 典型共享层包括：`store`、`project init`、`apiClient`、`WebSocket`、`storage/localDb`、`scheduler`、`核心写链`、`upgrade chain`、`delayRequests`。
- 一旦问题命中共享层，先记录缺陷，再指定唯一修复责任人 `E2`，其余执行线继续测不受影响的部分。
- `A1/B1/C1/D1/E1` 不允许因为“顺手”直接改代码。

#### 9.7.3 边测边修的推荐状态流转

- 第一步：测试失败，台账先记为 `失败`，生成缺陷编号。
- 第二步：如果属于当前执行线可直接修的范围，状态改为 `修复中`，并记录修复责任人 `A2/B2/C2/D2/E2`。
- 第三步：修复提交后，不改原失败记录，新增一条同编号的 `待回归` 或 `回归通过/回归失败` 记录。
- 第四步：若修复涉及共享层，把所有受影响测试项批量标记为 `待回归`。

#### 9.7.4 最小落地做法

每次边测边修都至少记录这 4 个东西：

- `缺陷编号`
- `修复责任人/模型`
- `修复提交版本`
- `需要回归的测试编号范围`

#### 9.7.5 推荐操作顺序

1. 先测出问题并落台账，不要先改再补记录。
2. 能局部修就局部修，修完立刻回归本编号。
3. 命中共享层就停在记录，不要多条线同时改。
4. 共享层修复合入后，按优先级补跑：`执行主链 -> 计划治理 -> 壳层协作 -> 专项业务 -> 基础设施复核`。

## 10. 本轮特别关注

结合当前代码与近期改造，本轮最需要盯紧的高风险区域：

- `GanttView` 与共享 store 接线后的跨页一致性。
- 预警/风险/问题链新增字段、跳转、升级链可追溯性。
- 计划编制的草稿值、显式保存、批量操作、锁机制。
- 定时任务驱动的通知、催办、自动消除是否真的生效。
- 删除保护、乐观锁、事务原子性在真人操作下是否保持稳定。

### 10.1 覆盖边界说明

| 维度 | 说明 | 影响程度 |
| --- | --- | --- |
| 页面内嵌子组件 | 例如 `ValidationPanel`、`DiffDrawer`、`GovernanceTreeTable` 等由页面直接 import 的独立组件，其内部渲染分支和细节交互，如 tooltip 文案、空态占位、骨架屏动画，可能仍有少量未被单独逐项展开 | 低：主流程已在页面用例中覆盖，剩余多为 UI 展示细节 |
| 后端 API handler / 定时任务 | 如月度催办 cron、closeout 强制关闭定时器、baseline realignment 队列处理等后端逻辑，页面源码通常只体现调用端，未天然等价于后端处理链已经逐条穷举验证 | 中：如果要做集成或 E2E 级别的发布把关，需要补后端侧专项用例或联动作业验证 |
| 纯样式 / 响应式 / 无障碍 | 如移动端断点、暗色模式、键盘导航完整性、屏幕阅读器标签等 | 低：属于专项测试类别，通常独立于主功能测试计划执行 |
