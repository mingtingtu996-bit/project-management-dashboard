# 数据质量体系实施专项方案

编制日期：2026-04-15  
需求来源：`v2功能增强需求文档_20260414.md` §七

## 1. 方案目标

本专项用于建立“进度监控 + 异常检测 + 置信度评估 + 关账标记”的数据质量闭环，让系统不仅能记录进度，还能判断这些进度是否可信。

目标是形成三层防线：

1. 系统自动监控：提前识别可能延期的任务。
2. 异常数据检测：识别长期不填、突击补填、逻辑矛盾、批量粗填。
3. 制度化标记：把数据置信度写进 Dashboard、Reports、关账结果。

## 2. 当前现状与缺口

### 2.1 已有底座

- `warningService` 已存在，可复用预警投递底座。
- `task_progress_snapshots` 已存在，具备周/月聚合与趋势计算基础。
- Dashboard 已有健康度相关摘要位，后续可并排展示数据置信度。
- 关账、月度兑现、偏差分析链路已存在，可承接“置信度仅供参考”标记。

### 2.2 当前缺口

- 现有预警偏“结果触发”，缺少“尚未到期但大概率要延期”的趋势预警。
- 当前只能较好识别“不更新”，还不能识别“乱更新”“粗填”“逻辑矛盾”。
- 项目经理缺少一张“当前数据可信不可信”的总量化指标。
- 数据质量结果没有写入月末关账记录，事后追溯无痕。

## 3. 目标业务形态

## 3.1 趋势预警

针对进行中任务，基于：

- `start_date`
- `end_date`
- `progress`

计算：

- 时间消耗率
- 进度完成率
- 偏差比

在任务未到期前，就能判断是否出现明显滞后趋势。

## 3.2 异常检测

同时覆盖两类问题：

1. 单任务异常
   - 进度跳变
   - 工期与进度极端不匹配
   - 逆序进度
   - 前后置矛盾

2. 交叉异常
   - 里程碑与前置任务矛盾
   - 条件未满足但已开始
   - 同一责任人工作量异常
   - 同一工作面工序矛盾

## 3.3 两条处理路径

- 聚合推送给项目经理：用于管理干预。
- 即时提示给填报人：用于现场修正。

## 3.4 数据置信度

对单项目生成统一的百分比评分，成为 Dashboard、Reports、关账结果的共用基础指标。

## 4. 实施设计

## 4.1 数据结构设计

建议新增三类表：

### 4.1.1 `data_quality_findings`

用于记录每日扫描结果。

建议字段：

- `id`
- `project_id`
- `task_id`
- `rule_code`
- `rule_type`：`trend | anomaly | cross_check`
- `severity`
- `dimension_key`
- `summary`
- `details_json`
- `detected_at`
- `resolved_at`
- `status`：`active | resolved | ignored`

### 4.1.2 `data_confidence_snapshots`

用于保存按项目、按月份的置信度结果。

- `id`
- `project_id`
- `period_month`
- `confidence_score`
- `timeliness_score`
- `anomaly_score`
- `consistency_score`
- `coverage_score`
- `jumpiness_score`
- `weights_json`
- `computed_at`

### 4.1.3 关账记录扩展

为月度关账记录增加：

- `data_confidence_score`
- `data_confidence_flag`
- `data_confidence_note`

这样可以把“仅供参考”永久写入留痕。

## 4.2 预警规则设计

## 4.2.1 进度滞后趋势预警

规则：

- 偏差比 < `0.7` 且剩余工期 >= 3 天：关注。
- 偏差比 < `0.5` 且剩余工期 >= 3 天：警告。
- 偏差比 < `0.5` 且剩余工期 < 3 天：严重。
- 关键路径任务阈值更严格，偏差比 < `0.8` 即可触发关注。

适用范围：

- `status = in_progress`
- 存在 `start_date`、`end_date`
- 计划工期 > 3 天

## 4.2.2 单任务异常检测

建议固化为规则引擎配置：

- `PROGRESS_JUMP`
- `PROGRESS_TIME_MISMATCH`
- `BATCH_SAME_VALUE`
- `PARENT_CHILD_INCONSISTENT`
- `DEPENDENCY_INCONSISTENT`

每条规则输出：

- 级别
- 说明
- 是否即时提示填报人
- 是否进入项目经理聚合推送

## 4.2.3 交叉异常检测

重点覆盖：

- 里程碑达成但前置任务未完成。
- 条件未满足但任务已开始。
- 同责任人同时段负荷异常。
- 同工作面工序顺序矛盾。

## 4.3 服务层设计

建议新增 `DataQualityService`，对外输出 4 组能力：

1. `scanTrendWarnings(projectId)`
2. `scanAnomalyFindings(projectId)`
3. `buildProjectConfidence(projectId, month)`
4. `buildOwnerDigest(projectId)`

调度方式：

- 每日一次全量扫描。
- 关键路径相关异常允许增量即时扫描。
- 月末关账前/关账时再做一次置信度固化。

## 4.4 通知与页面展示

## 4.4.1 推送给项目经理

规则：

- 同责任人/单位异常条数 >= 3，或
- 同楼栋/工作包异常占比 >= 20%

则推送一条聚合通知给 `owner`。

通知风格：

- 只给范围和严重度，不堆逐条任务。
- 如“本周 XX 楼栋存在较多进度异常，建议巡检时重点核实”。

关键路径任务例外：

- 一旦命中异常，直接即时推送。

## 4.4.2 提示给填报人

交叉矛盾类异常不直接打断流程，而是在任务列表非常态提示栏提示：

- 当前有 N 条任务存在数据矛盾需要确认。
- 点击展开后展示任务名、规则名、建议修正方向。
- 填报人修正后自动消失。

## 4.4.3 Dashboard 与 Reports

Dashboard：

- Hero 区与健康度并排显示数据置信度。

Reports：

- 偏差分析页顶部展示“本次分析基于 xx% 数据置信度”。

## 4.4.4 月末关账

- 关账完成时写入当月 `data_confidence_score`。
- 若低于阈值，在兑现率旁永久标注“数据置信度低，仅供参考”。
- 不阻塞关账，但保留审计痕迹。

## 4.5 权重与阈值

v2 建议先采用硬编码默认值：

| 维度 | 建议权重 |
| --- | --- |
| 填报及时性 | 30% |
| 异常检测命中率 | 25% |
| 交叉一致性 | 20% |
| 进度跳变率 | 10% |
| 更新覆盖率 | 15% |

后续支持在项目设置中调整权重。

## 5. 分阶段执行

## 5.1 第一阶段：趋势预警

- 实现进度滞后趋势预警。
- 接入 `warningService`。
- 关键路径阈值单独收紧。

交付结果：

- 系统从“延期后才报”升级为“提前预判”。

## 5.2 第二阶段：异常检测

- 落地单任务异常与交叉异常。
- 新增 `data_quality_findings`。
- 任务列表非常态提示栏接入交叉矛盾提示。
- 项目经理聚合通知上线。

交付结果：

- 系统能识别“填得不真实”的问题。

## 5.3 第三阶段：置信度与关账留痕

- 输出 `data_confidence_snapshots`。
- Dashboard / Reports 展示。
- 关账记录写入置信度标记。

交付结果：

- 数据质量成为所有分析结论的前置解释。

## 6. 涉及模块

后端：

- `server/src/services/warningService*`
- 新增 `server/src/services/dataQualityService.ts`
- 任务快照、任务依赖、里程碑、条件、阻碍查询链路
- 关账与报表相关服务

前端：

- `client/src/pages/Dashboard.tsx`
- `client/src/pages/Reports.tsx`
- `client/src/pages/GanttView.tsx`
- 非常态提示栏相关组件
- 通知中心、项目 Hero 指标区

## 7. 验收标准

1. 系统能对进行中任务生成“进度滞后趋势预警”。
2. 关键路径任务使用更严格阈值。
3. 系统能检测并记录单任务异常和交叉异常。
4. 项目经理收到聚合型数据质量通知，而非逐条刷屏。
5. 填报人在任务列表可看到即时矛盾提示，并在修正后自动消失。
6. Dashboard 展示数据置信度。
7. Reports 在偏差分析顶部展示数据置信度说明。
8. 月末关账记录写入当月数据置信度，并在低置信度时标记“仅供参考”。

## 8. 风险与注意事项

- 数据质量规则太敏感会产生噪声，v2 必须先控制误报率，再逐步扩规则。
- 异常检测不应直接堵死填报流程，否则现场人员会强烈抵触。
- 置信度是管理参考指标，不是审批门槛，避免被误用成“不能关账”。
- 关键路径任务即时扫描要控制执行成本，避免每次小更新都全项目重扫。

## 9. 建议优先级

- P1：趋势预警。
- P2：异常检测 + 两条处理路径。
- P2：数据置信度 + 关账标记。
