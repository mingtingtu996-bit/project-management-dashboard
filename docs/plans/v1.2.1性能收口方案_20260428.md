# v1.2.1 性能收口方案

编制日期：2026-04-28

## 1. 背景

v1.2 BI 化已经完成上线，但生产构建暴露出首包偏大的性能问题：

- 生产仅生成一个主入口脚本：`index-BAPuQuL2.js`
- 原始体积约 `2.76 MB`
- gzip 后约 `773 KB`
- brotli 后约 `599 KB`

这会导致多页面首次打开和 BI 钻取链路出现可感知等待，与 v1.2 BI 化“分析 -> 钻取 -> 行动”的体验目标存在偏差。

## 2. 收口目标

v1.2.1 不新增业务能力，只做性能收口：

- 恢复路由级代码分割，降低首屏入口 JS 体积
- 重型能力按需加载，避免 Excel / 图表 / 拖拽库进入无关页面首包
- 增加构建体积门禁，防止后续功能继续撑大入口包
- 保持 v1.2 BI 口径、趋势、切片、钻取能力不变

## 3. 本轮实施范围

### 3.1 前端构建拆包

- 移除 `inlineDynamicImports: true`
- 保留 `modulePreload: false`，避免公网首屏并发预拉取过多静态资源
- 保留 Rollup 对 React / Radix / 通用依赖的自动分包，避免手动 vendor 切分形成循环依赖
- 对重型能力增加 `manualChunks`：
  - `vendor-dnd`
  - `vendor-charts`
  - `vendor-xlsx`

### 3.2 Excel 能力按需加载

- `Reports` 导出 Excel 时再动态加载 `xlsx`
- `BaselinePage` 解析导入文件时再动态加载 `xlsx`

### 3.3 构建体积门禁

新增 `scripts/check-client-bundle-budget.mjs`：

- 入口脚本 gzip 默认不超过 `350 KB`
- 任一 JS chunk gzip 默认不超过 `900 KB`
- 总 JS gzip 默认不超过 `1600 KB`

部署流水线在前端构建后执行该检查。

## 4. 暂不纳入本轮

以下内容收益高但改动面更大，放入后续性能专项：

- Gantt / Reports / Risk / Materials / MonthlyPlan 页面 bootstrap 聚合接口
- 甘特、材料、风险、验收台账虚拟列表
- API GET 请求短缓存与前端请求去重策略
- 服务端慢查询日志和索引专项
- 静态资源 brotli 预压缩

## 5. 验收命令

本轮本地验收至少执行：

```bash
pnpm --dir client run lint
pnpm --dir client run typecheck
pnpm --dir client run test:run
pnpm --dir client run build
node scripts/check-client-bundle-budget.mjs client/dist --entry-gzip-kb 350 --max-chunk-gzip-kb 900 --total-gzip-kb 1600
npm run verify:workflow-contract
```

部署验收：

- GitHub Actions `Deploy Project Management System` 成功
- Browser Checks 三组与 Overview 成功
- 自托管服务器部署成功
- API 健康检查返回 `{"status":"ok"}`

## 6. 放行标准

- v1.2 BI 功能回归不退化
- 前端入口脚本 gzip 小于 `350 KB`
- 构建产物拆为多个可缓存 chunk
- 部署流水线全部成功
- 生产健康检查通过
