### A1 验收目录管理 — UI 壳已有，CRUD 未接后端

- **后端**：`/api/acceptance-catalog`（单数）— 验收 5 层模型中的 catalog 层 CRUD

- **实现方式**：
  1. 将 `acceptanceApi.getCustomTypes` / `createCustomType` / `deleteCustomType` 的假实现替换为真实调用 `/api/acceptance-catalog` GET/POST/DELETE

  3. 前端 UI 表单字段与上述后端字段一一对齐，`catalog_name` 为必填
- **涉及文件**：`client/src/services/acceptanceApi.ts`（第 550 行附近）、`client/src/pages/AcceptanceTimeline.tsx`
- **注意**：后端挂载为 `/api/acceptance-catalog`（单数，见 `server/src/index.ts:266`），前端调用时须与此一致

  - `ResponsibilityView.tsx` — 关注名单操作
  - `planning/*.tsx` — 基线确认、月计划确认/关账
- **后端配合**：审计所有写操作路由的 `requireProjectEditor` 中间件是否到位

### CP13 CPM 失败降级策略与规格不一致

  - 不查询 `critical_path_snapshots` 表历史记录

- **实现方式**：
  1. CPM 计算失败时：查询 `critical_path_snapshots` 表获取最近一条 `status = 'success'` 的快照

  4. 移除确定性 fallback 排序逻辑，避免生成误导性的伪关键路径
- **`critical_path_snapshots` 表 schema**：`id UUID PK` / `project_id UUID NOT NULL` / `snapshot_json JSONB NOT NULL`（含 displayTaskIds / watchedTaskIds / edges / projectDurationDays） / `status TEXT NOT NULL`（`'success' | 'failed'`） / `error_message TEXT` / `created_at TIMESTAMPTZ DEFAULT now()` / `created_by UUID`
- **涉及文件**：`server/src/services/projectCriticalPathService.ts`
- **严重度**：P1 — 降级策略可能生成误导性数据

### CP14 manual_attention 覆盖污染 displayTaskIds — 管理意图与客观计算混合

- **规格要求**：关键路径改造专项方案 §CP.10 定义 `displayTaskIds` 为"客观关键路径任务列表"，下游消费者（通知/治理/统计）据此判定任务是否在关键路径上。`manual_attention`（关注标记）是管理层面的主观意图，不应与 CPM 自动计算结果混合
- **后端现状**：`projectCriticalPathService.ts:871-877` 在生成 `displayTaskIds` 时将 `manual_attention` 覆盖的任务 ID 直接 append 到数组末尾：
  ```
  const manualAttentionTaskIds = overrides
    .filter((override) => override.mode === 'manual_attention')
    .map((override) => override.task_id)
  const displayTaskIds = unique([
    ...orderDisplayTaskIds(autoTaskIds, ..., manualInsertOverrides),
    ...manualAttentionTaskIds,  // ← 管理意图混入客观列表
  ])
  ```
- **影响**：
  - CP10（warningService）：延期通知会将"仅被关注但不在 CPM 主链上"的任务误判为关键路径延期，发送错误通知
  - CP11（10 个服务）：数据质量阈值提升（`is_critical ? 0.8 : 0.7`）、治理快照关键任务计数膨胀、周度简报关键路径统计偏高
  - CP8（甘特图着色）：`manual_attention` 任务虽应以琥珀色区分，但下游消费者无法区分来源
  - CP9（projectDurationDays）：manual_attention 任务被纳入工期计算
- **实现方式**：
  1. `displayTaskIds` 仅包含 CPM 自动计算和 `manual_insert` 类型的任务（这两类影响主链结构）
  2. `manual_attention` 任务 ID 存入 snapshot 的独立字段 `watchedTaskIds: string[]`，不混入 `displayTaskIds`
  3. 前端着色（CP8）使用 `watchedTaskIds` 判定琥珀色；下游消费者（CP10/CP11）不消费 `watchedTaskIds`
  4. 若产品需要 `manual_attention` 任务也触发某些通知，应作为独立的"关注任务提醒"通知类型，而非复用"关键路径延期"通知
- **涉及文件**：`server/src/services/projectCriticalPathService.ts`（第 871-877 行）
- **严重度**：P0 — 管理意图污染客观数据源，影响全局通知/统计/治理判定
- **与清单交叉**：CP8（着色区分）、CP9（工期计算）、CP10（通知判定）、CP11（10 服务消费）均受此影响

### CP15 tasks.is_critical 与 criticalPathSnapshot 无同步机制

- **规格要求**：关键路径改造专项方案 §CP.10 要求 `tasks.is_critical` 降级为"缓存/兼容字段"，由 snapshot 生成后同步更新
- **后端现状**：`projectCriticalPathService.ts` 在生成新 snapshot 后**不回写** `tasks.is_critical` 字段。同时 `baselineGovernanceService.ts`（lines 228/238/243/251/252）仍直接 `UPDATE tasks SET is_critical = true/false`，形成双向不一致：
  - snapshot 更新 → `tasks.is_critical` 不跟随更新 → CP11 中 10 个服务读到旧值
  - `baselineGovernanceService` 直接写 `is_critical` → snapshot 不跟随更新 → 弹窗/图谱展示与表字段不一致
- **实现方式**：
  1. 在 `recalculateProjectCriticalPath` 函数末尾增加 post-hook：根据新 snapshot 的 `displayTaskIds` 批量同步 `UPDATE tasks SET is_critical = (id IN displayTaskIds) WHERE project_id = ?`
  2. `baselineGovernanceService.ts` 中直接写 `is_critical` 的操作改为触发 `recalculateProjectCriticalPath(projectId)` 重算 snapshot，由 post-hook 同步 `is_critical`
  3. 最终目标（CP11 全量切换后）：移除 post-hook 和 `is_critical` 字段，所有消费者直接查 snapshot
  4. 过渡期：post-hook 确保 CP11 中尚未迁移的服务读到正确的缓存值
- **涉及文件**：`server/src/services/projectCriticalPathService.ts`、`server/src/services/baselineGovernanceService.ts`
- **严重度**：P1 — 缓存字段与真值源无同步机制，CP11 切换前所有消费者读到过期数据

### CP16 CriticalPathGraph 互斥逻辑阻止同一任务设置双类型覆盖

- **规格要求**：数据库 `critical_path_overrides` 表 unique key 为 `(project_id, task_id, mode)`，即同一任务可同时拥有 `manual_attention`（关注标记）和 `manual_insert`（手动插链）两个覆盖记录
- **前端现状**：`CriticalPathGraph.tsx` 使用 `selectedOverride` 变量做互斥判定：
  - line 319：`selectedOverride = props.overrides.find((override) => override.task_id === selectedTaskId)`，取第一个匹配的 override（若有两个只取一个）
  - line 368：`!selectedOverride` 作为 `canCreateInsert` 的条件 — 已有任何 override 则不允许创建插链
  - line 1107：`Boolean(selectedOverride)` 作为"标记为关注"按钮的 `disabled` 条件 — 已有任何 override 则不允许标记关注
  - 结果：已标记关注的任务无法再手动插链，已手动插链的任务无法再标记关注
- **决策**：选择方案 B（允许共存）。理由：数据库 unique key `(project_id, task_id, mode)` 已明确允许同一任务拥有不同 mode 的覆盖记录，前端应与 DB 策略对齐
- **实现方式**：
  1. `CriticalPathGraph.tsx:319` 将 `selectedOverride` 改为 `selectedOverrides: Override[]`，使用 `props.overrides.filter((override) => override.task_id === selectedTaskId)` 获取所有匹配的 override
  2. `canCreateInsert`（line 368）判定改为 `!selectedOverrides.some(o => o.mode === 'manual_insert')`（仅在已有插链覆盖时阻止，关注标记不阻止）
  3. "标记为关注"按钮 disabled 条件（line 1107）改为 `selectedOverrides.some(o => o.mode === 'manual_attention')`（仅在已有关注标记时阻止，插链覆盖不阻止）
  4. 已有覆盖的展示区需同时渲染多个 override 卡片（当前仅渲染一个 `selectedOverride`）
  5. DB unique key `(project_id, task_id, mode)` 保持不变，无需迁移
- **涉及文件**：`client/src/components/CriticalPathGraph.tsx`（第 319、368、1107 行）
- **严重度**：P2 — 前后端策略不一致，但不影响核心功能（仅限制了高级操作组合）

### CP17 overrides 端点缺少 PUT/PATCH — 无法编辑已有覆盖记录

- **规格要求**：手动覆盖记录应支持编辑操作（如修改 reason 备注、调整 anchor_type/left_task_id/right_task_id 锚点位置）

  - `POST /:id/critical-path/refresh`（line 50）— 重算快照

- **涉及文件**：`server/src/routes/critical-paths.ts`、`server/src/services/projectCriticalPathService.ts`
- **严重度**：P2 — 功能增强，非核心阻断
