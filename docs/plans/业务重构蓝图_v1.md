# 业务重构蓝图 v1

## 1. 总目标

基于现有应用进行系统性修缮，而不是重新开发一套新产品。

目标包括：
- 保留当前应用中你认可的核心业务逻辑
- 修复当前高频 bug、接口契约不一致、数据源混用、统计口径不统一的问题
- 统一架构，提升稳定性、性能和可维护性
- 在不增加现场录入负担的前提下，补齐缺失功能、合并重复功能、删减低价值功能

## 2. 产品总原则

- 任务列表 / WBS 是现场唯一主录入入口，尽量把日常更新集中到这里。
- 同一份底层数据，要能从不同维度反映项目绝大部分情况以及存在问题。
- 同一份底层数据，要能对不同层级反映不同层级所关心的事项。
- 现场尽量少填，系统尽量自动汇总、自动判断、自动展示。
- 不追求把线下解决过程完整搬进系统，只记录必要状态和结果。
- 不把应用做成复杂审批系统或重型工单系统。

## 3. 三层视图

### 3.1 公司驾驶舱

面向公司级管理层，查看多个项目的整体情况。

必须回答两类问题：
- 当前每个项目大致推进到哪了
- 哪些项目存在明显异常、偏差或风险压力

核心关注：
- 项目整体进展
- 健康度
- 关键里程碑偏差
- 重大风险 / 问题 / 阻碍
- 临期交付项目
- 跨项目排序与预警

### 3.2 项目 Dashboard

面向项目经理，查看单个项目的多维情况。

必须回答两类问题：
- 当前项目大致进展到哪了
- 这个项目当前哪里有问题、哪里偏了

核心关注：
- 当前阶段
- 总体进展
- 关键里程碑
- 延期情况
- 风险 / 问题 / 阻碍
- 证照、验收、图纸准备度
- 最近几天的变化

### 3.3 任务列表 / WBS

面向现场和一线管理，是最主要的录入入口。

核心职责：
- 每日更新进度
- 记录开工条件
- 记录阻碍
- 标记关键节点
- 沉淀任务时间线
- 提取任务总结素材

## 4. 最终业务模型

### 4.1 项目

一个独立建设、考核、交付的工程项目。

### 4.2 项目范围维度

项目范围不是严格线性树，而是管理维度集合。

可同时包含：
- 分期
- 标段
- 楼栋
- 地块
- 区域
- 专业

允许任务同时挂在多个维度上，不强行压成单线层级。

### 4.3 参建单位

参与项目的组织主体，例如：
- 建设单位
- 总包
- 监理
- 设计院
- 分包
- 供应商
- 顾问

### 4.4 项目成员

项目中的具体人，既包含其归属单位，也包含其在项目里的职责角色。

### 4.5 任务

一切可计划、可推进、可跟踪的工作项。

### 4.6 里程碑

关键节点或特殊关注事件，不是普通进度条的别名。

### 4.7 开工条件

任务开始前未满足的前提条件。

### 4.8 阻碍

任务推进过程中已经发生并造成影响的卡点。

### 4.9 风险

尚未发生，但已被正式识别并需要跟踪的潜在事件。

### 4.10 问题

已经发生、需要闭环关注的实际事项。

### 4.11 证照事项

前期报建、施工许可、预售许可、备案等办理事项，不只是文件附件。

### 4.12 图纸事项

围绕专业、阶段、版本、审图状态、发版状态管理的设计成果。

### 4.13 验收事项

项目里真实存在的一次验收活动，验收类型只是分类字典。

### 4.14 任务总结

任务从开始到结束期间的关键更新、阻碍、条件变化、进度变化的自动沉淀结果。

## 5. 关键业务规则

- 叶子任务允许人工更新进度；父级任务、WBS、项目进度全部由系统统一汇总。
- 里程碑有三种定义方式：
  - 模板预置
  - 任务一键设为里程碑
  - 手动新增事件型里程碑
- 里程碑达成有三种方式：
  - 绑定任务开始即达成
  - 绑定任务完成即达成
  - 手动确认达成并填写实际日期
- 风险确认只保留两种入口：
  - 系统候选风险一键确认
  - 人工直接登记
- 风险解除尽量自动化；不能自动化时允许一键关闭，不强制填写复杂处理过程。
- 问题来源只保留三类：
  - 阻碍上卷
  - 风险转问题
  - 人工直接登记
- 问题状态尽量简化为：
  - 处理中
  - 已解除
- 延期重点看：
  - 累计延误天数
  - 延误次数
  - 影响到哪些关键节点
- 验收以“验收事项”为主，不强制用户理解复杂计划层；复杂项目再展开依赖关系。
- 证照、验收、图纸都支持并联 / 网状关系展示，不按死板线性串行呈现。
- 图纸的核心目标是回答：
  - 当前阶段是否图纸齐套
  - 现场使用版本是否正确

## 6. 模板策略

- 第一批模板不依赖历史项目沉淀。
- 模板来源是：
  - 现有应用逻辑
  - 中国工程建设通用流程骨架
  - 业务经验
- 模板先做成“通用骨架 + 可调整节点”，后续再从项目实践中反向沉淀优化。
- 模板重点覆盖：
  - 房建住宅
  - 商业综合体
  - 工业厂房
  - 前期证照
  - 施工图阶段
  - 专项验收 / 联合验收

## 7. 现有应用保留 / 合并 / 删减 / 补强

### 7.1 保留

- 公司驾驶舱
- 项目 Dashboard
- 任务列表 / WBS
- 里程碑
- 风险问题
- 证照
- 验收
- 图纸
- 任务总结

### 7.2 合并

- 多处重复的进度算法
- 多处重复的健康度算法
- 多处重复的里程碑完成率算法
- 多处重复的风险统计口径

### 7.3 删减

- 重型表单
- 重复录入
- 对现场价值低、但操作负担高的步骤

### 7.4 补强

- 任务时间线
- 开工条件 / 阻碍 与 风险 / 问题 的轻量联动
- 图纸与任务阶段联动
- 验收 / 证照 / 图纸的网状展示
- 公司级与项目级指标口径统一

## 8. 第一阶段技术重构范围

- 统一后端 schema 与 route contract
- 统一前端 types 与 store
- 统一任务、风险、证照、验收、图纸等主对象的接口契约
- 统一公司驾驶舱、Dashboard、Gantt / 任务列表的统计与汇总口径
- 去掉“后端 + localStorage + 页面自算”三套并存带来的状态漂移
- 修复当前高频 `400 / 500`、保存失败、页面间数据不一致问题
- 保证现有页面继续存在，但底层数据和逻辑被统一

## 9. 实施顺序

1. 统一任务 / WBS / 里程碑 / 条件 / 阻碍主链路
2. 统一风险 / 问题 / 预警轻流程
3. 统一 Dashboard 与公司驾驶舱汇总逻辑
4. 统一证照、验收、图纸的业务契约和展示关系
5. 补任务时间线与任务总结自动沉淀
6. 清理重复功能、删除低价值操作、补测试与稳定性验证

## 10. 说明

这份蓝图确认的是：
- 业务方向
- 页面层级
- 保留 / 合并 / 删减 / 补强原则
- 第一阶段改动边界

不需要确认的是：
- 技术命名
- 表字段命名
- 前后端实现细节
- store 拆分方式
- route 具体写法
