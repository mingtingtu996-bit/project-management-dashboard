# v1.2.2 性能收口方案

编制日期：2026-04-28

## 1. 背景

v1.2.1 已完成首包拆分，入口脚本 gzip 已从约 `773 KB` 降至约 `125 KB`。当前用户仍感知到页面切换和业务操作慢，主要瓶颈已经从“首包过大”转为“进入项目后的多接口业务数据加载”。

## 2. 收口目标

v1.2.2 不新增业务功能，只做加载链路收口：

- 项目壳层初始化由多路 GET fan-out 收敛为一个 bootstrap 请求
- 保留旧多接口加载作为异常回退，避免单点失败导致页面不可用
- 不改变 v1.2 BI 口径、风险/问题/预警数据语义和甘特轻量初始化策略
- 用测试固定性能契约，防止后续又退回多接口初始化

## 3. 实施范围

### 3.1 项目初始化聚合接口

新增 `GET /api/projects/:id/bootstrap?changeLogLimit=100`，一次返回：

- `project`
- `tasks`
- `risks`
- `conditions`
- `obstacles`
- `warnings`
- `issues`
- `delayRequests`
- `changeLogs`
- `taskProgressSnapshots`

### 3.2 前端优先 bootstrap

`useProjectInit` 在非甘特、非材料模式下优先请求 bootstrap：

- 成功：一次性写入共享 store 并进入 `loaded`
- 404：进入 `not_found`
- 其他异常：回退旧多接口 fan-out，保证兼容性

### 3.3 保留轻量路径

- 甘特页继续保留轻量初始化和任务预取
- 材料页继续保留轻量项目上下文
- Dashboard、Risk、Reports 等项目页复用 bootstrap 后的共享切片

## 4. 本轮不纳入

- Dashboard 独立摘要接口二次聚合
- 甘特、风险、材料、验收、月计划虚拟列表
- Supabase 慢查询索引专项
- Brotli 预压缩与 CDN/HTTP2 调整

## 5. 验收命令

```bash
npm run build:client
npm run verify:client-bundle-budget
npm run typecheck --workspace=server
npx --prefix server vitest run --root server src/__tests__/projectBootstrapRoute.test.ts
npx vitest run --root client src/pages/__tests__/Wave8PerformanceContracts.test.ts
```

部署验收：

- GitHub Actions `Deploy Project Management System` 成功
- 自托管服务器健康检查返回 `{"status":"ok"}`
- 风险管理页面旧栏位不回归

## 6. 放行标准

- 前端入口脚本 gzip 继续小于 `350 KB`
- 项目初始化主路径请求数从 9 个业务 GET 收敛为 1 个 bootstrap GET
- bootstrap 异常时仍可回退旧加载路径
- 本地测试、构建、体积门禁和部署流水线全部通过
