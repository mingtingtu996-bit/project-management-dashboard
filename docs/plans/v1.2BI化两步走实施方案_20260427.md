## 2. 总体策略

采用“两步走”：

1. `第一步（1-2周）`：收口，为 BI 化扫清障碍
2. `第二步（2-3周）`：建设最小可用的业务驱动型 BI 能力层

本方案明确不走“一步到位建设完整指标平台 + 全量事实表 + 全量维表”的路线。

本方案的目标不是做通用 BI 平台，而是尽快完成：

- 指标口径统一
- 趋势可看
- 维度可切
- 图表可钻
- 分析可回到业务动作

#### B. 把 CompanyCockpit 聚合下沉到后端

当前问题：

- `CompanyCockpit` 先拿全项目摘要，再在前端计算平均健康度、平均进度、关注项目数、低健康项目数、里程碑逾期项目数、排名
- `CompanyCockpit` 还单独调用 `/api/health-score/avg-history` 获取公司级健康度历史趋势

目标改造：

- 新增公司级聚合查询服务
- 前端不再自己做公司级二次统计
- `CompanyCockpit` 只负责展示、筛选、跳转
- `company-summary` 必须直接复用 `getAllProjectExecutionSummaries()` 的结果做聚合，不允许再重复查询同一批业务表

性能注意事项：

- [getAllProjectExecutionSummaries()](</C:/Users/jjj64/WorkBuddy/20260318232610/server/src/services/projectExecutionSummaryService.ts:885>) 当前一次会拉取多组业务表数据，是公司级摘要热点路径
- `company-summary` 应在这份摘要结果上做二次聚合，而不是新增第二轮全表查询

建议输出：

- 新增公司级聚合端点  
  可选命名：
  - `GET /api/dashboard/company-summary`
  - 或 `GET /api/analytics/company-overview`

至少返回：

- 项目总数
- 平均健康度
- 平均进度
- 关注项目数
- 低健康项目数
- 逾期里程碑项目数
- 排名数据（含 healthScore，前端可用于热力图配色）
- `healthHistory`

切换要求：

- [CompanyCockpit.tsx](</C:/Users/jjj64/WorkBuddy/20260318232610/client/src/pages/CompanyCockpit.tsx>) 当前对 `/api/health-score/avg-history` 的单独调用必须并入新的公司级聚合端点
- 切换完成后，`CompanyCockpit` 不再单独请求 `avg-history`
- 也就是：公司级摘要卡片和公司级健康趋势统一由 `company-summary / company-overview` 一次返回

建议契约骨架：

```ts
interface CompanySummaryResponse {
  projectCount: number
  averageHealth: number
  averageProgress: number
  attentionProjectCount: number
  lowHealthProjectCount: number
  overdueMilestoneProjectCount: number
  healthHistory: {
    current: number
    previous: number
    change: number
    periods: Array<{ period: string; value: number }>
  }
  ranking: Array<{
    projectId: string
    projectName: string
    healthScore: number
    overallProgress: number
    attentionRequired: boolean
  }>
}
```

`ranking` 字段规格：

- 排序：按 `healthScore` **升序**（最差在前），健康度相同时按 `attentionRequired=true` 优先
- 数量：返回全部项目，不截断。前端决定展示 Top-N
- 用途：支撑 CompanyCockpit 现有的项目卡片列表排序，替代前端自排

**不再单独返回 heatmap 字段**。当前 CompanyCockpit 没有独立的热力图组件，`ranking` 已包含 `healthScore`，前端需要热力图配色时直接从 `ranking` 读取。避免两个几乎相同的数组造成契约冗余。

其中 `关注项目数` 必须下沉当前前端复合判定逻辑，而不是简单计数：

- `attentionRequired = true`
- 或 `healthScore < 60`
- 或 `milestoneOverview.stats.overdue > 0`

#### C. 扩展快照链，开始沉淀项目日数据

这一步不建五张事实表，只建一张项目级宽表：

- `project_daily_snapshot`

定位：

- 它是当前阶段的最小项目级日快照事实层
- 它承担总纲中 `fact_project_daily` 的实际落地职责

字段来源：

- 直接映射 [projectExecutionSummaryService.ts](</C:/Users/jjj64/WorkBuddy/20260318232610/server/src/services/projectExecutionSummaryService.ts>) 的输出

建议字段至少包括：

- `project_id`
- `snapshot_date`
- `health_score`
- `health_status`
- `overall_progress`
- `task_progress`
- `delay_days`
- `delay_count`
- `active_risk_count`
- `pending_condition_count`
- `active_obstacle_count`
- `active_delay_requests`
- `monthly_close_status`
- `attention_required`
- `highest_warning_level`
- `shifted_milestone_count`
- `critical_path_affected_tasks`

写入方式：

- 由调度任务每日写入
- 写入时复用现有摘要服务计算结果
- 先保证“项目 + 日期”唯一
- 写入策略明确为：`ON CONFLICT (project_id, snapshot_date) DO UPDATE`
- 同一天重跑时覆盖，不跳过、不重复插入、不报错

建议新增：

- `projectDailySnapshotService`
- 调度入口挂到 [scheduler.ts](</C:/Users/jjj64/WorkBuddy/20260318232610/server/src/scheduler.ts>)
- Supabase 迁移脚本：`server/migrations/115_create_project_daily_snapshot.sql`
- 编号依据：截至 2026-04-27，`server/migrations` 已存在 `107` 到 `114`，因此 v1.2 BI 当前使用下一个可用编号 `115`
- 执行方式：优先走现有 migration runner / GitHub Actions `database-migration`；若 `SUPABASE_MIGRATION_URL` 未配置或 CI 无法连通 Supabase，则通过 `Supabase Dashboard SQL Editor` 手动执行并留存执行记录
- 一次性回填脚本：`server/scripts/backfill-project-daily-snapshot.mjs`

与现有快照的关系：

- 当前项目级健康历史表实际为 `project_health_history`，不是新的项目日快照主表
- 新增 `project_daily_snapshot` 后，项目级健康度的定时写入应合并到 `projectDailySnapshotService`
- 现有 `/api/health-score/record-snapshot` 与 `healthHistorySnapshotJob` 不再作为独立项目级快照写入链长期保留
- `GET /api/health-score/avg-history` 应改为优先从 `project_daily_snapshot` 聚合读取；若保留 `project_health_history`，则仅作为兼容性只读结果或迁移过渡读模型
- 若仍需月度历史展示，可由 `project_daily_snapshot` 聚合得到，或将 `project_health_history` 退化为兼容性只读结果
- `task_progress_snapshots` 继续保留，因为它是任务粒度共享真值，不与项目级日快照重复

初始回填要求：

- `project_daily_snapshot` 建表后不能长期为空，否则趋势端点无数据可用
- 需做一次性历史回填：
  - 从 `project_health_history` 回填项目级健康度历史
  - 从 `task_progress_snapshots` 聚合回填可追溯的项目进度相关字段
- 对无法可靠追溯的字段允许填 `NULL`，不做伪造回填
- 回填完成后再开放 `project-trend / company-trend` 给前端正式消费
- 回填动作必须落成可执行脚本，不接受人工临时 SQL 口头操作

旧 job 切换时序：

- [scheduler.ts](</C:/Users/jjj64/WorkBuddy/20260318232610/server/src/scheduler.ts>) 中现有 `healthHistorySnapshotJob` 不应在新快照上线当天立刻停掉
- 建议新旧 job 并行运行 `7` 天做对账验证
- 对账内容至少包括：
  - 健康度分数
  - 健康状态
  - 公司级历史趋势
- 并行期验证通过后，再停掉旧 `healthHistorySnapshotJob`
- 停旧 job 的同时，把 `/api/health-score/avg-history` 的读取正式切到 `project_daily_snapshot`

## 4. 第二步：最小可用 BI 能力层（2-3周）

### 4.1 目标

在不建设完整数据仓库和元数据系统的前提下，让系统具备真正可用的业务型 BI 能力：

- 有趋势
- 能切片
- 能钻取
- 能回到业务动作

### 4.2 要做的事

#### 执行顺序

第二步同样需要按依赖顺序推进，推荐顺序如下：

0. 先完成维度字段排查和钻取目标页能力核对
1. 先补 `project-trend / company-trend` 查询端点
2. 并行补钻取目标页的 URL 参数解析
3. 再升级 `Reports` 的指标/时间/维度筛选器
4. 最后打通“筛选后的图表 -> 业务页”的钻取链路

原因：

- 第 `0` 步先确认“维度字典”和“业务数据标签”是否真的可用，否则筛选范围会失真
- 第 `1` 步是前端筛选器的后端前提
- 第 `2` 步可以与 `Reports` 改造并行，但必须先于最终钻取联调完成
- 第 `3` 步先让图表真正可按时间/维度变化
- 第 `4` 步建立最终闭环

补充说明：

- 如果第 `0` 步发现某些分析链没有维度标签，第二步应缩小该链的切片承诺，而不是在前端做空切片
- `GanttView / Materials` 已有 URL 参数基础，可直接纳入钻取第一批
- `RiskManagement / AcceptanceTimeline` 需要先补 query 参数解析，再进入钻取联调

第 `0` 步交付物必须形成核对清单，至少包含：

- 维度字典来源
- 底层业务表维度字段
- 每个分析链支持的切片维度
- 每个钻取目标页当前已支持 / 未支持的 URL 参数
- 需要补齐的路由参数改造项

#### A. 只用一张项目级日快照宽表支撑趋势分析

第二步继续使用：

- `project_daily_snapshot`

而不是一次性扩展为：

- `fact_project_daily`
- `fact_task_daily`
- `fact_risk_daily`
- `fact_material_daily`
- `fact_acceptance_daily`

这一张表先承担：

#### B. 维度切片复用已有 scope-dimensions

当前不新建九张维度表。

直接复用现有范围维度：

- `building`
- `specialty`
- `phase`
- `region`

也就是：

- `Reports` 做项目级切片时，优先复用现有 `scope-dimensions`
- 不在这一阶段建设完整 `dim_*` 体系

但在正式开工前必须先做两项前置排查：

- `Reports` 需要先接入 `scope-dimensions` 读取链路，拿到当前项目已配置维度
- 任务、风险、材料、验收等分析底层数据必须确认已具备可用于切片的维度标签字段；当前已确认部分链路已有 `building_id / specialty_type / phase_id`，但仍需补一轮完整字段映射核对，避免出现“维度字典存在但业务数据未打标签”导致切片为空

建议最小支持：

- `GET /api/analytics/company-trend`

用途：

- 支撑 `CompanyCockpit` 的时间窗口切换
- 支撑全项目聚合趋势，不再回退到前端拼装

建议契约骨架：

```ts
interface ProjectTrendPoint {
  date: string
  value: number | null
  group?: string | null
}

interface ProjectTrendResponse {
  projectId?: string
  metric: string
  from: string
  to: string
  groupBy: 'none' | 'building' | 'specialty' | 'phase' | 'region'
  granularity: 'day' | 'week' | 'month'
  points: ProjectTrendPoint[]
}

interface CompanyTrendResponse {
  metric: string
  from: string
  to: string
  granularity: 'day' | 'week' | 'month'
  points: Array<{
    date: string
    value: number | null
    projectCount: number
  }>
}
```

`company-trend` 与 `project-trend` 的区别：
- 不接受 `projectId`（聚合所有项目）
- 不接受 `groupBy`（公司级不按维度切分）
- 每个点额外返回 `projectCount`（参与聚合的项目数，便于前端判断数据覆盖度）
- `value` 为所有项目该指标的均值（与 company-summary 的 `averageHealth` 口径一致）

## 10. 架构护栏：防止新功能脱轨

### 10.1 问题

BI 两步走统一了后端数据源，但没有任何机制强制未来的新功能必须走统一体系。新增一个路由时，在路由层随手 `.reduce()` 就能出数据，不会报错——直到两套数字对不上。

### 10.2 三道防线

| 层级 | 手段 | 何时落地 | 具体做法 |
|------|------|----------|----------|
| **约定层** | CLAUDE.md 数据聚合规范 | 第一步验收后 | 追加规则：任何对外输出的汇总/统计/分数指标，**必须**从 `projectExecutionSummaryService` 或 `project_daily_snapshot` 取值；路由层禁止自行 `.reduce()` 出新指标 |
| **静态检查层** | ESLint / CI `rg` 或 Node 检查 | BI 完成后 | `server/src/routes/` 目录中禁止出现 `.reduce(`，如确需则加 `// eslint-disable-next-line -- route-level-aggregation-approved` 注释 |
| **运行时层** | metricRegistry 白名单 | 16.9 步 | analytics 路由只接受已注册的 metric key，未注册的 key 返回 400；新加指标必须先在注册表登记 |

### 10.3 新功能接入检查清单

未来每次新增功能时，按以下清单自查：

1. 新功能是否产生了新的聚合指标？→ 必须注册到 `metricRegistry.ts`
2. 新功能的统计数字从哪来？→ 必须从 `projectExecutionSummaryService` 或 `project_daily_snapshot` 取，禁止路由层自算
3. 新功能是否引入了前端侧 `.reduce()` / `.filter().length` 聚合？→ 改为消费后端端点
4. 新功能使用的 `isCompletedTask` / `normalizeStatus` / `getHealthStatus` 等判定函数，是否引用了公共模块？→ 禁止本地重新定义

## 11. BI 口径统一前置项

以下不统一项直接影响 BI 指标的正确性——如果底层判定函数定义不一致，snapshot 写入的数据、端点聚合的数字都不可信。这些是 BI 改造的**前置依赖**，需在对应步骤中一并解决。

### 11.1 `isCompletedTask` — 9 份独立实现，完成判定标准不一致

**「任务是否完成」是所有下游指标（progress / delayDays / healthScore）的基石。** 前后端共 9 个独立实现，彼此不一致：

| 位置 | 完成状态集合 | 进度兜底 |
|------|-------------|----------|
| `server/src/services/projectExecutionSummaryService.ts:166` | `completed, done, 已完成` | `progress >= 100` |
| `server/src/services/dataQualityService.ts:240` | `completed, done, 已完成` | `progress >= 100` |
| `server/src/services/operationalNotificationService.ts:76` | `completed, 已完成`（**缺 `done`**） | `progress >= 100` |
| `server/src/services/dbService.ts:912` | `completed, 已完成`（**缺 `done`**） | `progress >= 100` |
| `server/src/services/responsibilityInsightService.ts:148` | `completed, 已完成`（**缺 `done`**） | `progress >= 100`（**额外检查 `actual_end_date`**） |
| `server/src/services/taskWriteChainService.ts:101` | `completed, 已完成`（**缺 `done`，无 progress 兜底**） | ❌ 不检查 |
| `server/src/routes/task-summaries.ts:811` | `已完成, completed`（**无 progress 兜底**） | ❌ 不检查 |
| `client/src/lib/taskBusinessStatus.ts:110` | `completed`（**缺 `done` 和 `已完成`**） | `progress >= 100` |
| `client/src/lib/dashboardStatus.ts:3` | `已完成, completed`（**缺 `done`**） | `progress === 100`（**严格等于，不含 >100**） |

**BI 影响**：snapshot 写入的 `overall_progress` / `delay_days` / `health_score` 全部依赖此判定。定义不一致 = 指标不可信。

### 11.5 `taskBusinessStatus`（前端）vs `businessStatusService`（后端）— 双轨判定

| 端 | 文件 | 功能 |
|----|------|------|
| 前端 | `client/src/lib/taskBusinessStatus.ts` | `buildProjectTaskProgressSnapshot`、`isLaggingTask`、`isActiveRisk`、`normalizeStatus` |

**BI 影响**：前端有自己的 `isActiveRisk` 和 `normalizeStatus`，与后端不一致。当 BI 端点返回的活跃风险数和前端自算的不同，用户会看到矛盾的数字。

> **v1.1 新增分叉**：`dashboardStatus.ts` 的 `isActiveRisk` 已被 v1.1 更新为 `['closed', '已关闭']`，但 `taskBusinessStatus.ts:121` 的 `CLOSED_RISK_STATUSES` 仍为旧枚举 `['resolved', 'closed', 'mitigated', '已解决']`。两个前端文件的 `isActiveRisk` 又产生了分叉。16.12 清理 `dashboardStatus.ts` 后此分叉随之消除，但 `taskBusinessStatus.ts` 的旧枚举仍需在后续迭代中修正。

**解决方式**：后续迭代中，前端 `buildProjectTaskProgressSnapshot` 改为消费后端 API 返回值，不再自行计算。
