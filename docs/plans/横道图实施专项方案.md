# 横道图实施专项方案

编制日期：2026-04-15  
需求来源：`v2功能增强需求文档_20260414.md` §二

## 1. 方案目标

本专项用于在现有任务列表页内补齐“列表视图 / 横道图视图”双视图能力，使横道图成为任务列表的只读时间表达方式，而不是新建一套独立任务系统。

目标是：

1. 与现有任务数据完全同源。
2. 强化计划、实际、进度、依赖、关键路径的时序表达。
3. 保持只读，避免与列表编辑逻辑冲突。
4. 在 500 条任务以内维持可接受的性能和交互体验。

## 2. 当前现状与缺口

结合代码现状，当前系统具备以下基础：

- 任务列表、筛选、层级展开、任务详情弹窗、关键路径摘要已经存在。
- 关键路径图谱弹窗已完成接线，任务详情链路可复用。
- 基线、里程碑、依赖、进度等数据字段已经在任务域内存在。

但横道图需求当前仍未落地：

- `GanttView` 仍以表格/列表表现为主，没有真正的时间轴横道图查看态。
- 没有“列表 / 横道图”视图切换。
- 没有计划条、实际条、今日线、依赖线、关键路径边框的图形化表达。
- 没有基线对比横道图的只读查看态。

## 3. 业务定位

横道图不是新的编辑页面，而是现有 `GanttView` 的只读可视化模式。

边界定义：

- 数据源：完全复用任务列表当前查询结果。
- 编辑：不支持拖拽改期、不支持直接在图上修改任务。
- 打开详情：点击条形块后仍复用现有任务详情抽屉/弹窗。
- 月度计划：不做横道图，保持列表表达。

## 4. 实施设计

## 4.1 入口与视图状态

在 `GanttView` 页头增加视图切换：

- `列表视图`
- `横道图视图`

推荐状态管理：

- URL query：`?view=list|timeline`
- 本地记忆：最近一次视图模式缓存到 `localStorage`

这样可以支持：

- 刷新后保持用户上次视图。
- 从公司驾驶舱、Dashboard、提醒中心跳转时携带指定视图。

## 4.2 数据映射

横道图单行数据统一由任务实体转换而来：

| 字段 | 用途 |
| --- | --- |
| `name` | 左侧任务名称 |
| `level` / 父子关系 | 缩进与折叠 |
| `participant_unit_name` / `assignee_unit` | 左侧责任单位 |
| `assignee` | 左侧责任人 |
| `start_date` / `end_date` | 计划主条 |
| `actual_start_date` / `actual_end_date` | 实际对比条 |
| `progress` | 主条内部进度填充 |
| `status` | 正常 / 延期 / 完成 / 受阻色值 |
| `is_milestone` | 菱形里程碑标记 |
| `is_critical` | 关键路径边框 |
| 前后置依赖 | 依赖连线 |
| `baseline_start` / `baseline_end` | 基线对比查看态 |

## 4.3 前端组件拆分建议

建议将横道图从 `GanttView` 主文件中拆出，避免继续堆大组件。

推荐组件：

- `TaskTimelineView`
- `TimelineToolbar`
- `TimelineScaleHeader`
- `TimelineLeftColumns`
- `TimelineCanvas`
- `TimelineDependencyLayer`
- `TimelineRow`
- `BaselineTimelineView`

其中：

- `GanttView` 只负责筛选器、视图切换、任务详情弹窗、数据装配。
- 时间轴渲染逻辑全部下沉到独立组件。

## 4.4 渲染策略

### 4.4.1 技术选型

- 渲染方式：SVG。
- 原因：
  - v2 以只读交互为主。
  - 任务量通常小于 500。
  - 依赖线、里程碑、今日线、hover 命中在 SVG 下更自然。

不建议引入第三方甘特库，原因：

- 当前需求不需要拖拽编辑。
- 自研更容易对齐现有配色、关键路径语义和详情弹窗。
- 可避免第三方库样式冲突与包体积膨胀。

### 4.4.2 性能边界

- `< 200` 条任务：全量渲染。
- `200 - 500` 条任务：行级虚拟滚动。
- `> 500` 条任务：提示用户先缩小筛选范围。

推荐实现：

- 左右区域共享垂直滚动容器。
- 只对行做虚拟化，横向时间轴保持连续。

## 4.5 交互设计

### 4.5.1 基本交互

- 点击任务条：打开任务详情。
- hover 条形块：显示任务名、日期、进度、责任主体。
- 展开/折叠：沿用现有 WBS 展开状态。
- 筛选器：沿用列表视图，不另造一套。

### 4.5.2 时间轴能力

- 缩放档位：天 / 周 / 月。
- 今日线：默认显示。
- 横向滚动：时间轴独立滚动，左侧固定列保持锁定。
- 关键路径：用边框高亮，不额外改动现有关键路径筛选逻辑。

### 4.5.3 基线查看态

在“查看基线”场景下切换为基线横道图：

- 主条：`baseline_start ~ baseline_end`
- 对比条：当前 `start_date ~ end_date`

用途：

- 直观看到基线漂移。
- 不允许编辑。

## 4.6 视觉规范

需与当前任务列表色值保持一致：

- 正常：灰蓝系。
- 已完成：绿色系。
- 延期：红色系。
- 受阻：琥珀色系。
- 关键路径：红/橙描边。
- 里程碑：琥珀色菱形。

布局规范：

- 左侧固定列：桌面端 360px，`lg` 收窄到 280px。
- 行高：40px。
- 顶部标尺固定。
- 小于 `1024px` 时不渲染横道图，仅提示建议桌面端查看。

## 5. 后端与接口要求

本专项原则上不新增独立横道图接口，直接复用任务查询接口。

需要确认的后端条件：

1. 任务接口返回依赖关系、里程碑标识、关键路径标识。
2. 基线查看场景能提供 `baseline_start`、`baseline_end`。
3. 筛选后的任务结果顺序稳定，便于横道图与列表共享排序。

如果现有 `/api/tasks` 返回字段不够，可补一个“任务时间轴投影字段”但仍走同一资源，不单独创建 `/gantt-report` 类新接口。

## 6. 实施步骤

## 6.1 第一阶段：最小可用横道图

- 在 `GanttView` 增加视图切换。
- 左侧固定列 + 右侧计划条。
- 任务详情点击。
- 天/周/月缩放。

交付结果：

- 用户可以在同一页切到横道图查看项目计划全貌。

## 6.2 第二阶段：增强对比能力

- 实际时间细条。
- 今日线。
- 里程碑菱形。
- 关键路径高亮。
- 依赖连线。

交付结果：

- 横道图具备真正的执行分析价值。

## 6.3 第三阶段：基线查看态与性能收口

- 基线对比横道图。
- 200+ 行任务虚拟滚动。
- 超大数据量提示与筛选引导。

交付结果：

- 兼顾管理层全局查看和大型项目性能边界。

## 7. 涉及模块

前端重点：

- `client/src/pages/GanttView.tsx`
- 新增 `client/src/pages/GanttView/TaskTimelineView.tsx`
- 新增 `client/src/pages/GanttView/TimelineScaleHeader.tsx`
- 新增 `client/src/pages/GanttView/TimelineDependencyLayer.tsx`
- 任务详情抽屉与关键路径组件

后端重点：

- `server/src/routes/tasks*`
- 关键路径汇总/依赖计算服务
- 基线数据查询链路

## 8. 验收标准

1. `GanttView` 页面可在“列表视图 / 横道图视图”之间切换。
2. 横道图左侧展示任务名称、责任单位、责任人。
3. 横道图右侧展示计划条、进度填充，并按状态着色。
4. 若存在实际时间，能展示实际对比条。
5. 里程碑、依赖线、关键路径高亮可见。
6. 筛选器与列表视图共享，同一数据集切换时结果一致。
7. 点击横道图任务条可打开现有任务详情。
8. 基线查看态可显示“基线 vs 当前计划”双线对比。
9. 小屏不渲染横道图，只显示桌面端提示。

## 9. 风险与注意事项

- `GanttView` 当前体量已经很大，横道图必须拆组件，否则后续维护成本会明显上升。
- 依赖线和虚拟滚动同时存在时，坐标同步要提前设计，否则容易出现错位。
- 关键路径、基线、依赖字段若来源不统一，会导致横道图与列表解释不一致。
- 横道图为只读能力，不能在 v2 范围内演变为拖拽编辑，以免放大复杂度。

## 10. 建议优先级

- P2：最小可用横道图。
- P2：依赖线、里程碑、关键路径增强。
- P3：基线对比和大数据量性能优化。
