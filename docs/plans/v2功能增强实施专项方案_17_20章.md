## 17.1 项目整体偏差计算改为工期加权平均（P1）

**需求锚点**：需求文档 §17.1

**影响文件**：`server/src/routes/dashboard.ts`（函数 `calculateOverallProgress`，约第 77 行）

### 开始前定位

```bash
grep -n "calculateOverallProgress\|totalTasks\|overallProgress" server/src/routes/dashboard.ts
```

### 具体动作

1. 定位 `calculateOverallProgress(tasks)` 函数体中计算 `currentProgress` 的行：
   ```typescript
   // 当前（有缺陷）
   const currentProgress = Math.round(
     tasks.reduce((sum, t) => sum + (t.progress || 0), 0) / totalTasks
   )
   ```
2. 替换为工期加权平均：
   ```typescript
   // 工期加权平均
   const totalWeight = tasks.reduce((sum, t) => {
     const dur = (t.planned_start_date && t.planned_end_date)
       ? Math.max(1, Math.round(
           (new Date(t.planned_end_date).getTime() - new Date(t.planned_start_date).getTime())
           / 86400000
         ))
       : 1
     return sum + dur
   }, 0)
   const currentProgress = Math.round(
     tasks.reduce((sum, t) => {
       const dur = (t.planned_start_date && t.planned_end_date)
         ? Math.max(1, Math.round(
             (new Date(t.planned_end_date).getTime() - new Date(t.planned_start_date).getTime())
             / 86400000
           ))
         : 1
       return sum + (t.progress || 0) * dur
     }, 0) / (totalWeight || 1)
   )
   ```
3. 函数输出接口（`currentProgress / targetProgress / progressDeviation`）不变，无需改调用方

- `calculateOverallProgress` 返回的 `currentProgress` 改为工期加权值
- 无日期任务取权重 1 天，不影响现有无日期数据的兼容性
- TypeScript 编译通过，无新增类型错误

### 具体动作

   - 查询 `is_critical = true AND status != 'completed'` 的任务
   - 计算 `delayDays = daysBetween(planned_end_date, today)`（正值=已延期）

   - 优先取最高级别（≥20 天只生成 `critical`，不重复生成 `info`/`warning`）
   - `delayDays ≤ 0` 的任务跳过（未延期不触发）

3. 在 `scanAllWarnings(projectId?)` 的聚合调用中，将新方法加入并行扫描列表（与 `scanCriticalPathStagnationWarnings` 并列）

4. 周度简报（17.2）中标注 `level = 'critical'` 的关键路径延期任务为高亮：17.2 步骤直接从 `weekly_digests` 写入时读取此批数据

CREATE TABLE IF NOT EXISTS weekly_digests (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  week_start DATE NOT NULL,           -- 当周周一日期
  generated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  overall_progress NUMERIC(5,2),      -- 加权进度百分比
  health_score INTEGER,
  progress_change NUMERIC(5,2),       -- 本周进度变化量

  new_risks_count INTEGER DEFAULT 0,
  new_obstacles_count INTEGER DEFAULT 0,
  max_risk_level TEXT,                -- 'critical'|'high'|'medium'|'low'
  UNIQUE(project_id, week_start)
);

### 具体动作

#### A. taskBusinessStatus.ts

1. 新增类型：
   ```typescript
   export type LaggingLevel = 'lagging_mild' | 'lagging_moderate' | 'lagging_severe'
   ```

2. 新增纯函数 `isLaggingTask(task)`（返回 `LaggingLevel | null`）：
   - 适用条件：`status='in_progress'` AND 有 `planned_start_date` AND 有 `planned_end_date` AND 计划工期 > 3 天 AND 未逾期（`planned_end_date >= today`）
   - 计算：
     ```
     elapsed   = daysBetween(planned_start_date, today)
     duration  = daysBetween(planned_start_date, planned_end_date)
     timeRatio = elapsed / duration
     biasRatio = (progress / 100) / timeRatio   // timeRatio=0 时跳过
     remaining = daysBetween(today, planned_end_date)
     ```
   - 判定（按优先级从高到低）：
     - `biasRatio < 0.5 AND remaining < 3` → `'lagging_severe'`
     - `biasRatio < 0.5` → `'lagging_moderate'`
     - `biasRatio < 0.7`（关键路径任务 `is_critical=true` 则阈值为 `0.8`）→ `'lagging_mild'`
     - 否则 → `null`

3. 在 `TASK_STATUS_THEME` 中**替换** `blocked` 条目为三档 lagging 条目：
   ```typescript
   lagging_mild: {
     code: 'lagging_mild',
     label: '进度偏慢',
     className: 'bg-amber-50 text-amber-600 border border-amber-200',
   },
   lagging_moderate: {
     code: 'lagging_moderate',
     label: '进度慢',
     className: 'bg-amber-100 text-amber-700 border border-amber-200',
   },
   lagging_severe: {
     code: 'lagging_severe',
     label: '进度严重滞后',
     className: 'bg-orange-100 text-orange-700 border border-orange-200',
   },
   ```
   （同时删除原 `blocked` 条目；`blocked` 作为 DB 来源的 status 值，通过下面优先级链中 `in_progress` 分支处理）

4. 调整 `getTaskBusinessStatus()` 优先级链（在 `completed` 之后、`in_progress` 之前插入 lagging 判定）：
   ```
   ① completed          → status='completed' 或 progress≥100
   ② lagging_severe     → isLaggingTask(task) === 'lagging_severe'
   ③ lagging_moderate   → isLaggingTask(task) === 'lagging_moderate'
   ④ lagging_mild       → isLaggingTask(task) === 'lagging_mild'
   ⑤ in_progress        → status='in_progress' 或 status='blocked'（保留 blocked 的 DB 值兜底）
   ⑥ pending_conditions → status='todo' 且有未满足开工条件
   ⑦ ready              → status='todo' 且开工条件全部满足
   ⑧ pending            → 兜底
   ```

5. 从 `TaskBusinessStatus` 联合类型中删除 `'blocked'`，新增三个 lagging 字符串字面量
