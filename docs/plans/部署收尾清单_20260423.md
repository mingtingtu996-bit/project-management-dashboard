## 部署收尾清单

这份清单用于把当前仓库从“全量测试通过”推进到“可以正式发布”。

### 0. 先确认当前发布口径

- 正式发布链只有一条：`CloudBase Hosting + CloudRun`
- GitHub Actions 工作流：[`deploy.yml`](../../.github/workflows/deploy.yml)
- 工作流契约测试已明确禁止旧的 `VERCEL_*` secrets 口径：
  - [`deployWorkflowContract.test.ts`](../../server/src/__tests__/deployWorkflowContract.test.ts)
- 结论：当前发布不需要 `VERCEL_TOKEN / VERCEL_ORG_ID / VERCEL_PROJECT_ID`

### 1. 发布前必须通过

```bash
cd client
pnpm run lint
pnpm run typecheck
pnpm run test:run
pnpm run build

cd ../server
npm run typecheck
npm test
npm run build
npm run migrate:plan
```

### 2. GitHub Actions secrets 清单

#### 前端构建

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`

#### 数据迁移

- `SUPABASE_HOST`
- `SUPABASE_PORT`
- `SUPABASE_DATABASE`
- `SUPABASE_USER`
- `SUPABASE_PASSWORD`

#### CloudBase 部署

- `CLOUDBASE_ENV_ID`
- `CLOUDBASE_SECRET_ID` + `CLOUDBASE_SECRET_KEY`
  或
- `TENCENT_SECRET_ID` + `TENCENT_SECRET_KEY`

#### 可选

- `SLACK_WEBHOOK`

### 3. 历史 key 泄漏处理

如果之前已经确认有历史泄漏：

1. 先在对应平台吊销旧 key
2. 在 GitHub secrets 中写入新 key
3. 确认仓库当前代码和文档里不再包含明文有效 key

说明：

- 当前仓库正式工作流里已经没有 `VERCEL_*` 依赖
- 历史报告中出现的旧平台描述属于过去事实，不代表当前发布链仍依赖它

### 4. 正式发布动作

1. 提交当前本地改动
2. 推送到 `main/master`
3. 等待 GitHub Actions：
   - `client-quality`
   - `server-quality`
   - `build-frontend`
   - `browser-checks`
   - `database-migration`
   - `deploy-cloudbase`
4. 检查 CloudBase Hosting / CloudRun 发布结果

### 5. 发布后核查

```bash
curl http://<your-api-host>/api/health
```

重点核查：

- 首页可打开
- 公司驾驶舱可打开
- 甘特图可打开
- 风险管理可打开
- 报表页可打开
- 关键接口返回正常

### 6. 当前收尾结论

- “CloudBase 方向基本能接上”这句仍成立
- “Vercel 还会卡在缺少 `VERCEL_* secrets`”这句对当前仓库已经不成立
- 现在真正需要准备的是：
  - 全量测试通过
  - GitHub Actions 的 `Supabase + CloudBase` secrets 配齐
  - 推送后让现有工作流跑完
