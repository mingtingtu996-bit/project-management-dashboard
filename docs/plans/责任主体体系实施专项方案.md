# 责任主体体系实施专项方案

编制日期：2026-04-15  
需求来源：`v2功能增强需求文档_20260414.md` §一 + §1.8

## 1. 方案目标

本专项用于把“任务责任人、责任单位、系统用户、预警送达、责任主体分析”收口到同一条业务主线，形成三层闭环：

1. 数据层能识别“这个责任人到底是谁”。
2. 预警与汇总能按“人/单位”精准触达和聚合。
3. 前端形成独立的“责任主体”子视图，承接监控、分析、交叉查询和 Dashboard 跳转。

本次范围覆盖：

- `tasks.assignee_user_id` 外键补齐。
- 验收域 `responsible_user_id` 补齐并与责任主体体系打通。
- Dashboard 责任单位卡片跳转收口。
- 责任主体分析从 `TaskSummary` 迁出，形成独立子视图。
- 责任主体监控能力新增：健康趋势、异常预警、关注标记、恢复确认。

明确不纳入本专项：

- 施工图纸补 `user_id` 外键。
- 前期证照补 `user_id` 外键。
- 风险/问题域统一补 `_user_id` 外键。

## 2. 当前现状与缺口

结合现有代码，当前底座和缺口如下：

### 2.1 已有底座

- `participant_units` 相关表、路由和测试已经存在，责任单位基础数据底座已具备。
- `TaskSummary` 内已经有责任主体分析壳子，说明分析口径和页面组件并非从零开始。
- `warningService`、`task_progress_snapshots`、项目成员体系、项目级权限体系都已存在，可复用。
- 验收域已经保留 `responsible_person` 字符串字段，具备兼容显示基础。

### 2.2 关键缺口

- 任务表当前没有 `assignee_user_id`，系统无法把任务责任人与 `users.id` 精准关联。
- `TaskSummary` 仍承载责任主体分析，未迁出为独立子视图。
- Dashboard 的责任单位摘要跳转尚未收口到责任主体子视图。
- 验收流程当前未完整落地 `responsible_user_id`，现有接口还存在主动丢弃该字段的旧逻辑。
- 责任主体级异常预警、关注标记、恢复确认尚无独立数据结构与服务链路。
- 交叉查询能力尚未从“排行榜分析”升级到“人/单位双向联动”。

## 3. 目标业务形态

## 3.1 导航结构

```text
任务管理
  ├── 任务列表（GanttView）
  ├── 任务总结（TaskSummary）
  └── 责任主体
```

落地原则：

- `TaskSummary` 只保留“结果总结 -> 月度兑现”主线。
- 责任主体相关监控、排行、明细、交叉查询全部迁入独立页面。
- Dashboard 的责任单位入口统一导流到该页面。

## 3.2 页面结构

页面采用单页纵向双区块结构：

1. 区块一：责任主体监控。
2. 区块二：责任主体分析。

页面级统一控件：

- 顶部维度切换：责任人 / 责任单位。
- “参建单位管理”按钮：保持弹窗/抽屉模式，不单独拆页面。
- 锚点导航：监控、分析。

## 3.3 数据主线

```text
users
  ↑
tasks.assignee_user_id
  ↓
participant_units

acceptance_plans.responsible_user_id
```

展示原则：

- 前端显示继续使用 `assignee`、`responsible_person` 这类字符串显示名。
- 系统关联、送达、聚合、追溯统一使用 `_user_id` 外键。

## 4. 实施设计

## 4.1 数据库与模型层

### 4.1.1 任务域

- 为 `tasks` 增加 `assignee_user_id UUID NULL REFERENCES users(id)`。
- 保留 `assignee` 作为显示名兼容字段。
- 后端创建/更新任务时：
  - 若从项目成员中选择责任人，则同时写入 `assignee` 和 `assignee_user_id`。
  - 若手工输入纯文本，则仅写入 `assignee`，`assignee_user_id` 为空。

### 4.1.2 验收域

- 为验收模型补齐 `responsible_user_id UUID NULL REFERENCES users(id)`。
- 保留 `responsible_person` 作为显示名。
- 清理验收接口中主动删除 `responsible_user_id` 的兼容逻辑。

### 4.1.3 责任主体监控扩展表

建议新增两张表，支撑“关注”和“异常汇总”：

1. `responsibility_watchlist`
   - `id`
   - `project_id`
   - `dimension`：`person | unit`
   - `subject_user_id`
   - `subject_unit_id`
   - `created_by`
   - `status`：`active | suggested_to_clear | cleared`
   - `created_at / updated_at`

2. `responsibility_alert_states`
   - `id`
   - `project_id`
   - `dimension`
   - `subject_user_id`
   - `subject_unit_id`
   - `alert_type`
   - `current_level`
   - `consecutive_unhealthy_periods`
   - `consecutive_healthy_periods`
   - `last_snapshot_week`
   - `last_message_id`

这样可以避免每次都从通知表反推责任主体健康状态。

## 4.2 后端服务改造

### 4.2.1 任务写入链路

涉及模块：

- 任务创建/编辑接口
- Excel 导入接口
- 批量更新接口
- 本地缓存同步写回接口

改造要求：

- 统一接受 `assignee_user_id`。
- 返回体统一补回 `assignee_user_id`，供前端缓存和详情抽屉使用。
- 历史数据兼容：若仅有 `assignee` 无 `assignee_user_id`，前端照常展示。

### 4.2.2 预警送达精准化

`warningService` 调整为：

- 任务类预警：
  - 优先送达 `assignee_user_id`。
  - 若为空，则退化到项目 `owner` 责任组。
- 管理动作类通知：
  - 延期审批。
  - 关账超期升级。
  - 独立修正审批。
  - 仍按 `owner` 责任组送达。

### 4.2.3 责任主体监控聚合

新增 `ResponsibilityInsightService`，每周聚合两类口径：

1. 责任主体健康快照
   - 按时完成率。
   - 延期任务数。
   - 活跃阻碍数。
   - 本周 vs 上周趋势。

2. 责任主体异常判定
   - 连续两个统计周期按时完成率 < 60%。
   - 活跃延期任务数 >= 3。
   - 本月重点项未兑现数 >= 2。

输出供以下场景复用：

- 责任主体页区块一。
- Dashboard 责任单位跳转入口。
- `warningService` 责任主体异常通知。

### 4.2.4 交叉查询

查询能力统一在服务层提供：

- 选中责任人时：
  - 返回该人的任务明细。
  - 返回其所属责任单位的整体表现。
- 选中责任单位时：
  - 返回单位汇总。
  - 返回单位下所有责任人排行与明细。

这部分不继续放在 `TaskSummary` 的局部组件内，而是抽成独立接口。

## 4.3 前端页面改造

### 4.3.1 路由与导航

新增页面建议：

- 路由：`/projects/:projectId/responsibility`
- 页面：`client/src/pages/ResponsibilityView.tsx`

导航改造：

- 在“任务管理”组加入“责任主体”入口。
- Dashboard `UnitProgressCard` 的“查看全部”改为跳转该路由，并带默认维度 `unit`。
- `TaskSummary` 中移除旧的责任主体分析区块。

### 4.3.2 页面组件拆分建议

建议拆成以下组件，避免再次演变成超大页：

- `ResponsibilityHeader`
- `ResponsibilityDimensionToggle`
- `ParticipantUnitManageDialog`
- `ResponsibilityMonitorPanel`
- `ResponsibilityWatchlistPanel`
- `ResponsibilityAnalysisPanel`
- `ResponsibilityDetailTable`

### 4.3.3 UI 行为

区块一监控面板：

- amber 语义色。
- 展示健康趋势、异常标签、关注按钮。
- 异常项支持“一键标记关注”。
- 已关注主体单独展示本周变化摘要。

区块二区分析面板：

- blue 语义色。
- 保留现有指标卡、排行、重点关注、明细下钻能力。
- 新增人/单位交叉查询入口。

## 4.4 数据迁移与兼容策略

### 4.4.1 历史任务兼容

- 历史任务若无 `assignee_user_id`，保持只显示名称，不阻塞页面。
- 提供一次性匹配脚本：
  - 同项目下用 `assignee` 与项目成员显示名、用户名做模糊/精确映射。
  - 只对唯一匹配的记录自动回填。
  - 冲突记录输出人工处理清单。

### 4.4.2 验收兼容

- 历史验收记录保留 `responsible_person`。
- 未来新增/编辑时才强制支持从成员中选择。

## 5. 分阶段执行

## 5.1 第一阶段：底座收口

- 增加 `tasks.assignee_user_id`。
- 补齐验收 `responsible_user_id`。
- 统一任务/验收接口与类型定义。
- 调整 `warningService` 送达逻辑。

交付结果：

- 系统能识别责任人对应的真实用户。
- 任务类通知具备精准送达基础。

## 5.2 第二阶段：页面迁移

- 新增责任主体独立子视图。
- Dashboard 跳转改造。
- `TaskSummary` 移除责任主体分析区块。
- 责任人/责任单位交叉查询落地。

交付结果：

- 责任主体视角从结果页中剥离，形成稳定入口。

## 5.3 第三阶段：监控闭环

- 责任主体健康趋势。
- 责任主体异常预警。
- 关注标记。
- 恢复确认与自动消除。

交付结果：

- 从“能看分析”升级到“能盯异常、能跟踪恢复”。

## 7. 验收标准

满足以下条件即视为本专项完成：

1. 任务创建/编辑支持从项目成员选择责任人，并落库 `assignee_user_id`。
2. 任务类预警在存在 `assignee_user_id` 时可精准送达具体用户。
3. 验收流程支持 `responsible_user_id`，且不再被接口链路丢弃。
4. “责任主体”成为任务管理下独立子视图，`TaskSummary` 不再承载该分析区。
5. Dashboard 责任单位卡片“查看全部”能跳转到责任主体页。
6. 页面支持责任人/责任单位切换与交叉查询。
7. 页面支持责任主体异常预警、关注、恢复确认。
8. 图纸与前期证照域未被错误纳入 `_user_id` 改造范围。

## 8. 风险与注意事项

- `assignee` 纯文本历史数据较脏，自动回填只能做“保守匹配”，不能强行全量迁移。
- 责任单位与项目成员不是一一对应关系，交叉查询时要允许“单位存在、人未关联”的半结构状态。
- 验收域有外部机构参与，不可把外部机构错误并入 `participant_units`。
- `TaskSummary` 拆分后，旧测试和快照需要同步更新，避免页面职责重叠。

## 9. 建议优先级

- P1：`assignee_user_id`、验收 `responsible_user_id`、通知精准送达。
- P2：责任主体独立子视图、Dashboard 跳转、交叉查询。
- P2：责任主体监控、关注与恢复闭环。
