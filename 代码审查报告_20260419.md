#### High（高优先级）- 3 个

**H1. AuthContext 未使用统一 API 客户端**

- **文件：** `client/src/context/AuthContext.tsx`
- **行号：** 62-68, 100-107, 129-135, 151-158, 183-191, 204-212
- **问题：** 所有认证相关 API 调用直接使用原生 `fetch`，绕过了 `apiClient.ts` 的统一错误处理
- **影响：** 
  - 认证错误不会触发统一的 toast 通知
  - 离线状态下可能发送无效请求
  - 错误处理逻辑不一致
- **修复建议：** 改用 `apiPost/apiGet` 替代直接 `fetch`

**H2. localStorage 直接访问未使用安全包装**

- **文件：** 多个文件（auditLog.ts, backup.ts, FeedbackModal.tsx, modalManager.ts）
- **问题：** 直接使用 `localStorage.getItem/setItem`，未使用 `safeStorageGet/Set` 包装
- **影响：** 存储空间不足时可能抛出未捕获的 QuotaExceededError
- **修复建议：** 统一使用 `safeStorageGet/Set/Remove`

**H3. JSON.parse 缺少错误处理**

- **文件：** 多个文件（FeedbackModal.tsx, RecentTasksCard.tsx, auditLog.ts, backup.ts）
- **问题：** 部分 `JSON.parse` 调用没有 try-catch 保护
- **影响：** 存储数据损坏时应用可能崩溃
- **修复建议：** 创建 `safeJsonParse` 包装函数

---

### 3.2 后端发现的问题

#### Critical（严重）- 3 个

**C1. 迁移文件编号重复**

- **问题：** 存在重复编号的迁移文件（037, 038, 086, 087, 091, 100）
- **影响：** 数据库版本管理混乱，可能导致迁移执行顺序错误
- **修复建议：** 清理重复编号，建立单一权威迁移序列

**C2. 部分异步路由未使用错误处理包装**

- **统计：** 15/60 路由未使用 `asyncHandler`
- **文件：** auth.ts, auth-logout.ts, client-errors.ts 等
- **影响：** 可能导致未捕获的 Promise rejection，服务器崩溃
- **修复建议：** 为所有异步路由添加 `asyncHandler` 包装

**C3. 多表写操作缺少事务保护**

- **问题：** 仅 1 个路由文件使用事务，其他依赖 Supabase RPC
- **影响：** 可能导致数据不一致
- **修复建议：** 审查多表写操作，确保原子性

---

#### High（高优先级）- 4 个

**H1. auth 路由响应格式不统一**

- **文件：** auth.ts, auth-register.ts, auth-profile.ts, auth-reset-password.ts
- **问题：** 使用旧格式 `{ success, message }` 而非标准 `ApiResponse`
- **影响：** API 一致性差
- **修复建议：** 迁移到标准 `ApiResponse` 格式

**H2. 输入验证覆盖不足**

- **统计：** 仅 21/60 路由使用验证中间件（35%）
- **影响：** 可能接受无效数据，导致数据库错误或安全问题
- **修复建议：** 为所有 POST/PUT/PATCH 路由添加 Zod 验证

**H3. SQL 模板字符串需审查**

- **统计：** 20 处使用模板字符串
- **影响：** 潜在的 SQL 注入风险
