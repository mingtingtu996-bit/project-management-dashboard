# 展示层回调方案 V2｜逐页面改动方案（执行版）

更新日期：2026-04-05  
文档定位：用于后续逐项拆任务、排期和验收的展示层执行清单，不涉及代码实现说明。

## 一、全局执行边界

### 1.1 本轮只做什么

- 只做展示层回调。
- 只调整页面结构、信息层级、入口组织、命名、共享样式语言、局部展示承接关系。
- 只收回成熟版中有价值的信息层、入口、图示能力，并统一进当前壳子和共享样式体系。

### 1.2 本轮绝不能动什么

- 共享摘要口径不改。
- 任务主链不改。
- 风险 / 问题链不改。
- 证照 / 验收 / 图纸业务契约不改。
- 任务时间线和任务总结沉淀逻辑不改。
- Dashboard 和 CompanyCockpit 继续只做读取聚合，不反向侵入录入链路。
- [ProjectDetail.tsx](C:/Users/jjj64/WorkBuddy/20260318232610/client/src/pages/ProjectDetail.tsx#L26) 的既有语法问题属于 v1 收尾，不属于本次展示层回调实现范围。

### 1.3 实施分批

- 第一批高频页：`CompanyCockpit`、`Dashboard`、`RiskManagement / RiskAndIssues`、`Notifications`、`Milestones`、`TaskSummary`
- 第二批支撑页：`WBSTemplates`、`GanttView（任务列表页 / 历史文件名）`、`PreMilestones`、`AcceptanceTimeline`、`Reports`
- 结构承接页与工具页：`ProjectLayout`、`Header`、`Sidebar`、`Monitoring`、`TeamMembers`、历史兼容入口

### 1.4 关键补充边界

- `TaskManagement` 和 `LicenseManagement` 当前都属于“父模块概念已成立，但统一父页面承接还未彻底成立”的状态。
- 本轮对这两个父模块的处理重点不是新增重页面，而是先把导航语义、父子关系、标题区和子页切换承接做成立。
- `Reports` 拆回各模块后，必须遵守统一规则：只按业务归属承接，不允许每个模块再长出一套新的“总页式报表”。
- 隐藏入口必须区分三种处理方式：隐藏但保留路由、隐藏且直接跳转、隐藏且改为辅助面板承接，不能模糊处理。
- `CompanyCockpit` 与 `Dashboard` 虽然都属于总览页，但严禁信息重复堆叠：前者只回答公司级跨项目问题，后者只回答单项目问题。

### 1.5 页面承接关系表

| 旧入口 | 新归属 | 是否隐藏 | 是否保留路由 | 是否直接跳转 | 用户默认进入哪里 |
| --- | --- | --- | --- | --- | --- |
| `ProjectList` | `CompanyCockpit` | 是 | 是 | 是 | `CompanyCockpit` |
| 全局 `Dashboard` | `CompanyCockpit / 项目 Dashboard` | 是 | 是 | 是 | 公司级默认进 `CompanyCockpit` |
| `Reports` | 拆回各模块承接 | 是 | 是 | 否 | 项目默认进 `Dashboard`，统计入口按模块进入 |
| `Monitoring` | 隐藏工具路由 | 是 | 是 | 否 | 无默认进入，仅隐藏入口访问 |
| `TeamMembers` | `Header / 项目辅助面板` | 是 | 是 | 否 | 项目内当前页的辅助入口 |
| `GanttView` | `任务管理 / 任务列表页（历史路由名）` | 否 | 是 | 否 | `任务管理 / 任务列表` |
| `TaskSummary` | `任务管理` 子页 | 否 | 是 | 否 | `任务管理 / 任务总结` |
| `PreMilestones` | `证照管理` 子页 | 否 | 是 | 否 | `证照管理 / 前期证照` |
| `AcceptanceTimeline` | `证照管理` 子页 | 否 | 是 | 否 | `证照管理 / 验收时间轴` |

### 1.6 统一入口命名规范

- 公司级入口只使用：`公司驾驶舱`、`提醒中心`
- 项目级主模块只使用：`Dashboard`、`里程碑`、`WBS模板`、`任务管理`、`风险与问题`、`证照管理`
- 任务相关子页只使用：`任务列表`、`任务总结`
- `WBS` 只作为 `任务列表` 页内部结构表达，不作为独立子页命名
- 证照相关子页只使用：`前期证照`、`施工图纸`、`验收时间轴`
- 辅助能力只使用：`团队成员`、`设置面板`、`提醒设置`
- 不再把 `项目列表`、全局 `Dashboard`、独立 `报表`、独立 `监控` 作为主导航命名
- 当前文件名可以保留，但页面文案、标题区、导航文案必须统一到上述命名

### 1.7 快捷入口密度规则

- 标题区最多 `1` 个主按钮 + `2` 个次按钮；超过部分必须收进更多操作
- 一级总览卡默认只允许“整卡点击”或 `1` 个局部动作，不能每张卡都堆满按钮
- 单个信息区最多展示 `3` 个快捷入口，其余入口下沉到次级区域或列表项内部
- 列表项右侧最多展示 `2` 个直接动作，避免把列表做成操作面板
- `Dashboard` 和 `CompanyCockpit` 的快捷入口优先承担“进入下一级页面”，不承担新增业务处理动作
- 所有快捷入口都必须优先保留高频入口，不能为了“简洁”删掉原有高频动作

## 二、全局层

### 2.1 顶层导航契约

- 当前问题：
  - 公司级入口与项目级入口仍有混挂。
  - 历史上属于工具页、辅助页、兼容页的入口仍干扰主导航。
  - 顶层和项目内的命名没有完全回到成熟版契约。
- 本轮目标：
  - 顶层只保留 `公司驾驶舱`、`提醒中心` 两个公司级主入口。
  - 项目内统一展开 `Dashboard / 里程碑 / WBS模板 / 任务管理 / 风险与问题 / 证照管理` 六大模块。
- 保留项：
  - 公司级进入项目的能力。
  - 提醒跳转到具体项目与具体模块的能力。
  - 历史路由的可访问性。
- 恢复项：
  - 公司级入口与项目级入口两层结构。
  - 项目内模块命名、顺序、归属的稳定表达。
- 合并项：
  - `项目列表` 表达并入 `公司驾驶舱`。
  - `全局 Dashboard` 表达拆解为 `公司驾驶舱 + 项目 Dashboard`。
- 隐藏但保留路由项：
  - `Monitoring`
  - `Settings`
  - `Reports`
  - `TeamMembers`
- 页面结构建议：
  - 顶层导航只展示公司级入口。
  - 进入项目后切换为项目级工作台导航。
  - 不在同一层级同时暴露公司入口、工具入口、项目模块入口。
- 信息层级建议：
  - 一级：公司级入口。
  - 二级：项目级六大模块。
  - 三级：模块子页或辅助能力。
- 入口调整建议：
  - `Sidebar` 只承接主入口。
  - `Header` 承接辅助能力和快捷能力。
  - 历史兼容入口通过跳转承接，不再常驻导航。
- 与共享样式体系的对齐要求：
  - 导航选中态、分组标题、二级入口样式统一。
  - 顶层与项目级导航使用同一套状态 badge、悬停节奏与留白基线。
- 实现优先级：
  - `P0`
- 验收点：
  - 顶层只剩公司级入口。
  - 进入项目后才出现项目级模块。
  - 隐藏入口路由可达但不再干扰主导航。

### 2.2 Header 全局能力

- 当前问题：
  - Header 与侧边栏分工不够明确。
  - 团队、提醒、设置类能力缺少清晰归属。
- 本轮目标：
  - Header 只承接全站辅助能力，不承担主业务导航。
- 保留项：
  - 搜索、提醒、项目上下文、用户操作、快捷入口。
- 恢复项：
  - 团队成员与局部设置作为辅助能力的定位。
  - 提醒与全站通知在 Header 中的存在感。
- 合并项：
  - 团队成员入口并入 Header 或项目辅助面板。
  - 局部设置入口与提醒中心设置面板协同。
- 隐藏但保留路由项：
  - 独立 `TeamMembers`
  - 独立 `Settings`
- 页面结构建议：
  - 左侧承接项目上下文或返回关系。
  - 右侧承接提醒、搜索、团队、用户操作。
- 信息层级建议：
  - 高优先级只显示高频辅助能力。
  - 低优先级收进面板或下拉层。
- 入口调整建议：
  - 不把团队成员放回侧边栏主入口。
  - 设置能力只通过提醒中心或 Header 面板触达。
- 与共享样式体系的对齐要求：
  - Header 高度、图标尺寸、按钮密度、角标样式与侧边栏一致。
- 实现优先级：
  - `P0`
- 验收点：
  - Header 与侧边栏分工清晰。
  - 辅助能力完整保留但不干扰主入口。

### 2.3 ProjectLayout / 项目工作台壳层

- 当前问题：
  - 项目级入口承接关系不够完整。
  - 模块和子页缺少统一壳层表达。
- 本轮目标：
  - 建立项目级统一工作台壳层，承接六大模块和子页关系。
- 保留项：
  - 项目上下文、项目 ID 路由、子页切换。
- 恢复项：
  - 项目内模块分组、返回路径、面包屑或上下文表达。
- 合并项：
  - 零散项目页统一挂靠到六大模块之下。
- 隐藏但保留路由项：
  - 历史 `reports`、`team`、工具类子路由。
- 页面结构建议：
  - 顶部项目上下文。
  - 左侧模块导航。
  - 主区域承接模块页与子页。
- 信息层级建议：
  - 壳层只提供上下文，不重复承接页面内容。
- 入口调整建议：
  - 默认进入项目先进入 `Dashboard`。
  - 任务相关子页全部归口到任务管理认知下。
- 与共享样式体系的对齐要求：
  - 项目标题区、模块切换样式、面包屑密度统一。
- 实现优先级：
  - `P0`
- 验收点：
  - 项目内导航层级稳定。
  - 页面归属清晰，不再出现游离子页。

### 2.4 全站共享样式体系

- 当前问题：
  - 页面之间存在卡片层级、标题区、空状态、按钮密度不一致的问题。
- 本轮目标：
  - 用同一套成熟样式语言收口全站展示层。
- 保留项：
  - 已稳定可复用的成熟组件结构。
- 恢复项：
  - 标题区双层结构。
  - 一级统计卡与二级信息卡层级。
  - badge / pill 的统一表达。
  - 空状态、列表节奏、留白结构的一致性。
- 合并项：
  - 重复卡片风格、重复标题样式、重复局部壳子。
- 隐藏但保留路由项：
  - 无。
- 页面结构建议：
  - 各页统一采用“标题区 + 一级总览 + 重点区 + 明细区”思路。
- 信息层级建议：
  - 一级总览承接结果。
  - 二级模块承接行动与明细。
  - 三级列表承接具体事项。
- 入口调整建议：
  - 主按钮统一放在标题区或一级总览附近。
  - 次级操作尽量下沉。
- 与共享样式体系的对齐要求：
  - 标题区、统计卡层级、状态 badge/pill、列表节奏、空状态、信息分组、按钮密度、留白结构全站统一。
- 实现优先级：
  - `P0`
- 验收点：
  - 全站不再出现多代风格混搭。
  - 高频页之间视觉语言一致。

## 三、公司级

### 3.1 CompanyCockpit

- 当前问题：
  - 当前版本更像简化后的聚合列表页，未完整承接成熟版公司驾驶舱的图示能力和管理视角。
  - 历史存在的项目排序、风险矩阵、交付倒计时、跨项目里程碑等有价值展示尚未完整回收。
  - 局部卡片风格与全站壳层语言尚未完全统一。
  - 与项目 `Dashboard` 的信息边界虽然方向正确，但仍需防止重复展示同一层级摘要。
- 本轮目标：
  - 把公司驾驶舱恢复为公司级第一主入口，完整承接“汇总、排序、趋势、关注、进入项目”这五类能力。
  - 保持只读聚合，不新增录入逻辑。
- 保留项：
  - 共享摘要聚合链路。
  - 项目搜索、筛选、进入项目能力。
  - 项目级跳转、刷新、关键入口联动。
- 恢复项：
  - 公司级总览指标区。
  - 健康度趋势或热力表达。
  - 项目排序 / 矩阵视图。
  - 优先跟进 Top5。
  - 跨项目里程碑双视图。
  - 项目卡片中的关键状态摘要。
- 合并项：
  - 历史 `ProjectList` 的浏览与筛选能力。
  - 历史公司级汇总报表能力。
  - 与独立全局 Dashboard 重叠的公司级总览能力。
- 隐藏但保留路由项：
  - 历史 `/projects` 入口。
  - 历史全局总览兼容入口。
- 页面结构建议：
  - 标题区：公司级标题、全局筛选、主按钮、刷新。
  - 一级总览：项目数、平均进度、平均健康度、风险项目数、近期交付数。
  - 重点区：优先跟进、风险矩阵、交付倒计时。
  - 趋势区：健康度趋势、阶段分布、排名。
  - 明细区：项目卡片、跨项目里程碑。
- 信息层级建议：
  - 第一层先回答“整体怎么样”。
  - 第二层回答“哪些项目最值得关注”。
  - 第三层回答“为什么要关注、下一步进哪个项目”。
- 去重边界：
  - 只保留公司级、跨项目、排序型、趋势型信息。
  - 不重复展开单项目详细执行块，不把项目 `Dashboard` 的详细信息搬上来。
  - 单项目深度信息只允许以“摘要 + 进入项目”方式出现。
- 入口调整建议：
  - 项目进入作为每张卡片和重点区的核心动作。
  - 图示区允许跳转到项目 Dashboard 或风险与问题。
  - 不再出现独立“项目列表”入口与公司驾驶舱并列。
- 与共享样式体系的对齐要求：
  - 标题区使用统一双层结构。
  - 一级统计卡与 Dashboard 使用同一层级体系。
  - badge / pill 统一使用项目状态、健康度、风险等级语言。
  - 列表与项目卡节奏统一，不用独立深色强主题破坏壳层。
  - 空状态保持与全站一致的说明 + 主动作结构。
- 实现优先级：
  - `P0`
- 验收点：
  - 页面已不再是“项目列表替代品”，而是完整公司驾驶舱。
  - 原有有价值图示能力要么恢复，要么明确合并到新区域中。
  - 从公司级看项目状态、风险、交付压力、下一步入口都清晰。

### 3.2 Notifications / ReminderCenter

- 当前问题：
  - 提醒中心虽然已具备能力，但仍需要进一步明确其作为公司级第二主入口的结构地位。
  - 系统设置与提醒、与我相关、跨模块跳转的承接关系还需统一。
- 本轮目标：
  - 把提醒中心固定为公司级第二入口，同时承接提醒聚合、与我相关、设置面板。
- 保留项：
  - 通知列表、分类、状态处理、项目跳转、模块跳转。
  - 已有提醒中心数据链和操作逻辑。
- 恢复项：
  - 统计概览区。
  - 与我相关视图。
  - 分类 Tab 和筛选区的优先级。
  - 设置面板与提醒中心的同屏承接关系。
- 合并项：
  - 历史 `Settings` 展示承接到提醒中心。
  - 零散预警查看入口集中到提醒中心的通知语义下。
- 隐藏但保留路由项：
  - `/settings`
- 页面结构建议：
  - 标题区：页面标题、筛选、批量操作、设置入口。
  - 一级总览：未读数、待处理数、与我相关数、跨项目提醒数。
  - 主体区：Tab + 列表。
  - 辅助区：设置面板或右侧配置区。
- 信息层级建议：
  - 第一层先回答“今天有什么需要处理”。
  - 第二层再按提醒类别和与我相关拆分。
  - 第三层通过列表进入具体页面。
- 入口调整建议：
  - 从 Header 和公司级导航进入。
  - 列表项必须保留跳转到 `Dashboard / 风险与问题 / 证照管理 / 任务管理` 的路径。
- 与共享样式体系的对齐要求：
  - 标题区、统计卡、Tab、列表项状态 badge、批量按钮密度与高频页统一。
  - 空状态用“无待处理 / 去公司驾驶舱 / 去项目”的双动作结构。
- 实现优先级：
  - `P0`
- 验收点：
  - 提醒中心的角色明确，设置不再独立游离。
  - 通知到业务页的跳转链路不断。
  - 与我相关、分类、状态处理都在同一套清晰结构内。

### 3.3 Settings 面板承接

- 当前问题：
  - `Settings` 独立存在的产品价值低于其占用的导航位置。
- 本轮目标：
  - 将设置能力固定为提醒中心附属面板，不再作为主导航页面。
- 保留项：
  - 必要设置能力与历史可访问路径。
- 恢复项：
  - 设置作为辅助能力的合理归位。
- 合并项：
  - 独立设置页表达并入 `Notifications / ReminderCenter`。
- 隐藏但保留路由项：

### 5.7 Reports（能力拆回，不保留孤立总页）

- 恢复项：

  - 独立报表总页不再是核心依赖。

## 七、执行验收总则

- 页面回调后，必须先看层级是否正确，再看样式是否统一，最后看入口是否完整。
- 所有高频入口必须保留，只允许合并承接，不允许因为布局调整直接消失。
- 所有统计、状态、图示能力必须先判断“是否有价值”，只有确认重复且低价值时才允许下沉或弱化。

- Dashboard 和 CompanyCockpit 的任何调整都必须保持“只读聚合”边界。
- 一旦涉及共享摘要、任务主链、风险问题链、证照 / 验收 / 图纸契约、任务时间线、任务总结沉淀逻辑，立即停止，归类为本轮不可触碰项。
