## 🔐 **权限体系详情**

### 三级权限体系 (`permissions.ts`)

| 角色 | 权限范围 | 适用用户 |
|------|----------|----------|
| **🔑 管理员** | 完整权限：查看、创建、编辑、删除、邀请成员、管理设置 | 项目创建者、团队负责人 |
| **✏️ 编辑者** | 核心权限：查看、创建、编辑（不能删除、不能管理成员） | 核心团队成员、项目经理 |
| **👁️ 访客** | 只读权限：只能查看项目信息 | 外部顾问、观察者、客户代表 |

### 权限矩阵

```typescript
// ROLE_PERMISSIONS 定义
admin: [view:project, edit:project, delete:project, invite:member, ...]
editor: [view:project, edit:project, create:task, edit:task, ...]
guest: [view:project, view:task, view:risk, view:milestone, view:team]
```

## 🔄 **实时协作流程**

### 1. 项目创建与邀请流程
```
1. 用户创建项目 → 生成主邀请码
2. 管理员生成邀请码 → 设置权限和有效期
3. 其他用户使用邀请码 → 加入项目成为成员
4. 系统自动分配权限 → 基于邀请码设置的权限级别
```

### 2. 实时数据同步流程
```
1. 用户A修改任务 → 触发Supabase数据库变更
2. Supabase实时订阅 → 向所有订阅者推送变更事件
3. 用户B接收通知 → 实时更新本地任务列表
4. OnlineMembers组件 → 显示用户A在线状态
```

### 3. 离线协作流程
```
1. 网络断开 → 使用离线缓存服务
2. 用户继续工作 → 操作进入同步队列
3. 网络恢复 → 自动同步队列中的操作
4. 冲突处理 → 时间戳优先的简单冲突解决
```

## ⚠️ **限制与边界条件**

### 1. **技术限制**
- **用户规模**: 适合中小型团队（2-50人），不适合大型企业
- **实时性**: 依赖Supabase服务，网络中断时降级为离线模式
- **存储容量**: 免费版Supabase有存储限制

### 2. **功能边界**
- **无协同编辑**: 不支持文档协同编辑（如Google Docs风格）
- **无实时聊天**: 需要集成第三方聊天工具或使用评论功能
- **简单冲突解决**: 基于时间戳的简单冲突解决，无复杂合并逻辑

### 3. **部署要求**
- **Supabase账号**: 需要配置Supabase项目
- **服务器部署**: 后端API需要部署到服务器
- **客户端配置**: 需要配置环境变量

## 🚀 **协作流程示例**

### 场景: 3人团队开发项目
```
1. 项目创建者 → 创建项目，成为管理员
2. 管理员 → 生成2个邀请码（开发权限）
3. 开发者A → 使用邀请码加入，成为编辑者
4. 开发者B → 使用邀请码加入，成为编辑者
5. 项目开始 → 三人同时在项目中工作

协作过程：
- 开发者A创建任务 → 实时同步给所有成员
- 开发者B修改任务进度 → 实时更新显示
- 项目创建者添加里程碑 → 所有成员实时可见
- 网络断开 → 开发者A继续工作，变更进入队列
- 网络恢复 → 队列变更自动同步
```
