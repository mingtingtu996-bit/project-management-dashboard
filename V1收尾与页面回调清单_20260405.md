# V1 收尾与页面回调清单

更新日期：2026-04-05

- 现在还不能进入 v1 收尾与真实环境落地。
- 当前状态只能算“全局壳子已回调，CompanyCockpit 与 Dashboard 完成第一轮表现层重写”，还没有完成“按成熟版基线回调 + 页面级核对”。
- `client/src/pages/ProjectDetail.tsx` 现有语法错误必须纳入收尾清单，否则前端全量 TypeScript 校验无法恢复。

## 已确认的偏差

### 1. 导航契约未回到原应用

- 当前全局侧边栏仍是：`项目列表 / 仪表盘 / 监控 / 设置`
- 当前项目内侧边栏仍是：`概览 / 甘特图 / 风险管理 / 里程碑 / 报表 / 团队`
- 这与回调基线不一致。

目标应调整为：

- 顶层：`公司驾驶舱 / 提醒中心`
- 进入项目后：`Dashboard / 里程碑 / WBS模板 / 任务管理 / 风险与问题 / 证照管理`

补充说明：

- 现有路由中其实还保留了 `acceptance`、`pre-milestones`、`task-summary`、`wbs-templates` 等页面能力，但没有按成熟版入口结构正确暴露。

### 2. CompanyCockpit 只完成了第一轮，不是最终回调结果

- 当前页保留了公司概览、优先跟进、项目卡片、里程碑汇总。
- 但 `client/src/pages/CompanyCockpit/components/` 下仍存在成熟版图示能力与分析组件，例如：
  - `ProjectRanking.tsx`
  - `RiskBubbleMatrix.tsx`
  - `MilestoneAchievementChart.tsx`
  - `DeliveryCountdown.tsx`
- 这说明公司驾驶舱原先的一部分图示和统计展示，并没有完成“确认无价值后合并”，而是暂时没有重新接回页面。
- “本月信号”当前使用深色强调区，和应用整体浅色信息层级还没有完全统一，需要二次回调。

### 3. Dashboard 保留了主数据口径，但没有完整回到 UI 文件结构

- 当前 Dashboard 已保留的核心信息：
  - 项目健康度
  - 问题与风险
  - 里程碑
  - 证照事项
  - 验收计划
  - 施工图纸
  - 项目概况
  - 快速入口
- 但与 `UI设计方案_V4_最终版.html` 对比，仍有未完全回到位的内容：
  - “项目健康度”卡片的四维拆解与展开区没有按原结构恢复
  - “问题与风险”卡片的 2x2 统计布局没有按原结构恢复
  - Top N 紧急问题列表没有按原结构恢复
  - “里程碑追踪”卡片的原展示密度和动作入口没有完全回到成熟版

结论：

- 当前 Dashboard 不是“功能已完整回调”，而是“核心信息大体还在，但展示结构仍需第二轮对齐”。

### 4. 其他页面仍在回调范围内，不能默认不管

- 不能只回调 `CompanyCockpit.tsx` 和 `Dashboard.tsx` 就结束。
- 结合当前代码与 UI 文件，以下页面仍需纳入回调范围：
  - `Notifications.tsx`
  - `RiskManagement.tsx`
  - `Milestones.tsx`
  - `GanttView.tsx`
  - `WBSTemplates.tsx`
  - `PreMilestones.tsx`
  - `AcceptanceTimeline.tsx`
  - `TaskSummary.tsx`
- `Reports.tsx` 当前仍是独立页面，需要重新判断：
  - 如果原应用是“报表能力下沉到各子模块”，则应继续弱化独立报表页，并把必要统计归回对应模块
  - 只有真正独立、非重复的报表能力才保留

### 5. ProjectDetail.tsx 语法错误列为收尾阻断项

- 文件：`client/src/pages/ProjectDetail.tsx`
- 现状：`npx tsc -p client/tsconfig.json --noEmit` 被该文件阻断
- 当前确认的错误类型包括：
  - `TS1005`
  - `TS1127`
  - `TS17008`
  - `TS1382`
  - `TS1381`
- 该项不一定是回调页面的第一优先级，但必须进入 v1 收尾阻断清单。

## 继续回调的执行顺序

### 第一步：先收导航契约

- 回调 `Sidebar.tsx`
- 必要时同步调整 `App.tsx` 与项目内入口映射
- 目标不是新增页面，而是把现有页面能力按成熟版名称和层级重新挂回去

验收标准：

- 顶层只保留成熟版需要的主入口
- 进入项目后再显示项目模块入口
- 名称统一，不再出现“概览/监控/设置/报表/团队”这类偏离基线的自由命名

### 第二步：CompanyCockpit 二次回调

- 逐项核对原有图示、排行、矩阵、节奏类展示
- 只删除确认重复且低价值的模块
- 对保留能力做风格统一，去掉与全局壳子冲突的突兀表现

验收标准：

- 原先高频查看的信息不能无故消失
- 图示能力要么恢复，要么明确合并到哪个区域展示
- 公司驾驶舱的视觉语言与全局壳子一致

### 第三步：Dashboard 二次回调

- 以 `UI设计方案_V4_最终版.html` 的 Dashboard 为主参照
- 对齐模块顺序、卡片层级、信息密度、动作入口
- 对“已合并展示”的信息做逐项核对，不能只保留概念，不保留使用体验

验收标准：

- UI 文件中应展示的关键信息都能在页面找到
- 如果是合并展示，需要能明确指出被合并到哪里
- 不得因为美化页面而削弱原来的关键操作入口

### 第四步：其余项目页面继续回调

- 先做项目内高频页：
  - `RiskManagement.tsx`
  - `Milestones.tsx`
  - `GanttView.tsx`
  - `WBSTemplates.tsx`
- 再做专项与辅助页：
  - `PreMilestones.tsx`
  - `AcceptanceTimeline.tsx`
  - `TaskSummary.tsx`
  - `Notifications.tsx`
- `Reports.tsx` 放在这一轮中重新定位，决定保留、弱化还是拆回子模块

### 第五步：回调后的页面级核对

- 看起来是否回到成熟版风格
- 常用入口是否都还在
- 页面加载、刷新、跳转、主按钮是否正常
- 确认没有把原应用高频功能误删

### 第六步：再进入 v1 收尾与真实环境落地

前提是以下条件全部满足：

- 页面回调完成
- 页面级人工核对完成
- `ProjectDetail.tsx` 语法错误处理完成
- 前端全量 TypeScript 校验恢复
- 之后再进入真实环境迁移、人工验收、发布包整理、v2 backlog 拆分
