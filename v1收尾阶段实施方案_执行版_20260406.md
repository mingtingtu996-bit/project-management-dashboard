## 0. 使用规则

### 0.1 执行目标

本文件不是方向说明，而是给大模型直接执行的顺序版。每一步都必须：

1. 只修改本步骤列出的目标文件  
2. 优先修复当前步骤，不顺手扩写其他步骤  
3. 每步结束后都执行本文列出的验证命令  
4. 如果命中“停止”条件，立即停止，不继续下一步

### 0.2 通用报错策略

- **忽略**：仅开发期 warning，且不影响当前步骤功能与测试结果  
  例：React Router future flag warning、`act(...)` warning
- **修复**：当前步骤直接引入的类型错误、运行时错误、测试失败
- **停止**：出现以下任一情况时停止该步骤并记录
  - 目标文件不存在且不是预期删除步骤
  - 发现有用户新改动与当前步骤直接冲突
  - 需要改动本文未列出的高风险文件才能继续
  - 验证命令显示当前步骤相关逻辑已严重偏离文档假设

### 0.4 进度跟踪

- 进度文件：`EXECUTION_PROGRESS.json`，位于项目根目录
- 每步**开始前**，读取进度文件，跳过 status 为 `completed` 的步骤
- 每步**完成后**，将该步骤标记为 `completed` 并记录验证结果摘要
- 如果步骤命中"停止"条件，标记为 `blocked` 并记录原因
- 新会话启动时，第一件事是读取进度文件，从第一个 `pending` 步骤继续

进度文件格式：

```json
{
  "last_updated": "2026-04-06T10:00:00",
  "current_phase": 0,
  "steps": {
    "1.1": { "status": "completed", "note": "快照提交成功" },
    "1.2": { "status": "completed", "note": "6个文件已标废" },
    "1.3": { "status": "blocked", "note": "UserMenu 仍被 Header 引用" },
    "2.1": { "status": "pending" }
  }
}
```

### 0.5 会话恢复规则

每次会话启动时，执行以下操作：

1. 读取 `EXECUTION_PROGRESS.json`
2. 读取本文件中对应步骤的内容
3. 如果上一步状态为 `in_progress`，先运行该步的验证命令，确认是否已实际完成
   - 验证通过 → 标记为 `completed`
   - 验证未通过 → 从该步继续
4. 从第一个 `pending` 步骤开始执行
5. 每完成一步，立即更新进度文件后再继续下一步
